## 1. مقدمة عن دليل التشغيل

### 1.1 لمن هذا الدليل؟

| الجمهور الأساسي | كيف يستخدم الدليل |
|---|---|
| 👑 **المالك (OWNER)** | مراجعة الإجراءات الشهرية، لوحة المالك، التقارير المالية، قرار الإلغاء الكامل، الجرد الشهري. |
| 🛡️ **مدير النظام (ADMIN)** | إدارة المستخدمين، الإعدادات، ترحيل فواتير الشراء، سجل التدقيق، حل المشكلات التقنية. |
| ✅ **المدير التشغيلي (MANAGER)** | الوردية اليومية، الاعتمادات (جرد/ورديات)، مراجعة المرتجعات، التسعير، الإلغاء الكامل. |

> 💡 هذا الدليل **ليس بديلاً** عن أدلة المستخدمين السبعة المفصّلة، بل يُكملها: يخبرك **بالتسلسل الزمني** لما يجب فعله خلال اليوم/الأسبوع/الشهر، ويُحيلك للأدلة الفرعية عند الحاجة لتفاصيل كل زر.

### 1.2 العلاقة مع أدوار المستخدمين السبعة

النظام مبني على سبعة أدوار متمايزة، كلٌّ بنطاق صلاحياته. هذا الدليل يُنسِّق عملها معاً:

| الدور | الشارة | الوظيفة في المتجر | القسم الأكثر استعمالاً |
|---|:---:|---|---|
| 👑 OWNER | كهرماني | المالك — رؤية شاملة، قرارات استراتيجية | لوحة المالك، المحاسبة، التقارير |
| 🛡️ ADMIN | وردي | مدير النظام — إعدادات ومستخدمون وتدقيق | الإعدادات، المستخدمون، سجل التدقيق |
| ✅ MANAGER | أزرق | المدير التشغيلي — ورديات واعتمادات يومية | لوحة المدير، المخزون، التسعير |
| 🧮 ACCOUNTANT | بنفسجي | المحاسب — قيود ومصروفات وتقارير مالية | المحاسبة، المورّدون |
| 🏭 WAREHOUSE | كهرماني | أمين المخزن — استلام وتحويل وجرد | المخزون، المشتريات، التركيبات |
| 🛒 SALES | سماوي | موظف المبيعات — بيع وتبديل وفتح وردية | POS، الفواتير، العملاء |
| 🛒 CASHIER | أخضر زمردي | الكاشير — بيع فقط + وردية خاصة به | POS (وضع Express)، الورديات |

> ⚠️ **قاعدة ذهبية:** كل دور مقيَّد بنطاقه. الكاشير لا يرى إلا POS والورديات. المحاسب لا يدخل POS. أمين المخزن لا يرى المالية. فقط OWNER يرى كل شيء بما في ذلك **لوحة المالك الحصرية**.

### 1.3 رموز الإشارة المستخدمة في الدليل

| الرمز | المعنى |
|---|---|
| ✅ | خطوة مكتملة / إجراء ناجح / عنصر ضروري |
| ⚠️ | تنبيه — انتباه لملاحظة مهمة |
| 🚨 | خطر — إجراء حرج قد يؤثر على المالية أو المخزون |
| 📅 | تكرار زمني (يومي/أسبوعي/شهري) |
| 💡 | نصيحة عملية لتسهيل العمل |
| 🔁 | إجراء متكرر عدة مرات |
| 📄 | مستند أو تقرير يُطبع أو يُصدَّر |
| 🔒 | إجراء يتطلب صلاحية محددة |

## 3. الإجراءات اليومية (Daily Operations)

### 3.1 قبل الافتتاح — 30 دقيقة قبل فتح المتجر 🌅

#### قائمة التحقق قبل الافتتاح (Pre-opening Checklist)

| ✅ | المهمة | المسؤول | ملاحظات |
|:---:|---|---|---|
| ✅ | فحص الكهرباء العامة للمتجر | مدير الوردية | تأكد من عمل التكييف والإضاءة |
| ✅ | فحص اتصال الإنترنت | مدير الوردية | جرب فتح أي موقع — النظام يعمل على السحابة (Supabase) |
| ✅ | تشغيل جهاز الكاشير + تسجيل الدخول | مدير الوردية | استخدم حسابك الخاص |
| ✅ | فحص طابعة الإيصالات (80mm) | مدير الوردية | اطبع إيصال تجريبي |
| ✅ | فحص قارئ الباركود | مدير الوردية | امسح باركود أي منتج للتأكد |
| ✅ | فحص درج النقدية (Cash Drawer) | مدير الوردية | جرّب الفتح من زر Express |
| ✅ | فحص طابعة A4 (للفواتير التفصيلية) | مدير الوردية | إن وُجدت |
| ✅ | تسجيل دخول مدير الوردية على لوحة المدير | المدير | صفحة **لوحة المدير** تتحدث كل 60 ثانية |
| ✅ | مراجعة تنبيهات المخزون المنخفض | مدير الوردية | من بطاقة «نقص المخزون» في لوحة المدير |
| ✅ | مراجعة أوامر الشراء المعلّقة | مدير الوردية + أمين المخزن | PENDING / PENDING_APPROVAL / APPROVED |
| ✅ | مراجعة الورديات المفتوحة من الليلة السابقة | مدير الوردية | إن وُجدت وردية OPEN من اليوم السابق = خطأ يجب إغلاقه فوراً |
| ✅ | تزويد الدرج بالنقدية الافتتاحية | مدير الوردية | لتُعطى للكاشير كرصيد افتتاحي |

> ⚠️ **تنبيه:** إن كان اتصال الإنترنت غير متاح، **لا يمكن** استخدام النظام (لا يوجد وضع دون اتصال). راجع §7.1 للتعامل مع انقطاع الإنترنت.

#### 3.1.1 مراجعة تنبيهات لوحة المدير

| البطاقة | ماذا تعرض | الإجراء المطلوب |
|---|---|---|
| **مبيعات اليوم** | مبيعات اليوم + شارة مقارنة بأمس | لا إجراء — متابعة فقط |
| **الورديات المفتوحة** | عدد الورديات OPEN + أسماء أول كاشيرين | التأكد من عدم وجود وردية متروكة |
| **نقص المخزون** | عدد المنتجات `quantity ≤ reorderLevel` | النقر للانتقال للمخزون + تجهيز أمر شراء |
| **مستحقات موردين** | مجموع الفواتير POSTED غير المدفوعة | التخطيط للسداد هذا الأسبوع |
| **لافتة نسبة الحذف/المرتجع** | VOID + REFUND ÷ المبيعات خلال 7 أيام | 🚨 > 10% أحمر = تحقيق فوري §7.5 |

### 3.2 افتتاح الوَردية (Shift Opening) 🔓

#### 3.2.1 خطوات فتح الوردية

| الخطوة | الإجراء | المسؤول |
|:---:|---|---|
| 1 | الدخول لشاشة **الورديات** (`shifts`) من القائمة الجانبية | الكاشير |
| 2 | الضغط على زر **«فتح وردية جديدة»** (Play) | الكاشير |
| 3 | إدخال **الرصيد الافتتاحي** (النقدية الموجودة في الدرج) | الكاشير |
| 4 | تأكيد الفتح — يُولَّد رقم `SHF-XXXX` تلقائياً | النظام |
| 5 | التحقق من ظهور الوردية في لوحة المدير خلال 60 ثانية | المدير |
| 6 | اعتماد المدير المبدئي (مرئي فقط — لا يلزم ضغط زر) | المدير |

> 💡 **الرصيد الافتتاحي يُضاف للنقدية المتوقعة عند الإغلاق:**
> ```
> expectedCash = openingBalance + cashSales − cashRefunds
> ```
> لذا يجب عدّ النقدية فعلياً قبل الإدخال، لا تخمينها.

#### 3.2.2 من يفتح ومن يعتمد؟

| الإجراء | المسؤول | القيد على الخادم |
|---|---|---|
| فتح الوردية | أي مستخدم مسجَّل دخوله (الكاشير عادةً) | لا يوجد قيد على الدور في `POST /api/shifts` |
| الوردية المفتوحة للمستخدم | لا يمكن فتح أخرى حتى تُغلق | `409 user-shift-already-open` |
| ورديات متعددة لكاشيرين مختلفين | ✅ مسموحة في نفس الوقت | كل كاشير له وردية مستقلة |

#### 3.2.3 ماذا لو نسي الكاشير الرصيد الافتتاحي؟

🚨 **المشكلة:** لو فُتحت الوردية برصيد افتتاحي = 0 (لم يُدخله الكاشير)، ستكون النقدية المتوقعة عند الإغلاق = مبيعات النقدية فقط، دون احتساب البدل الذي في الدرج.

| الحل | الخطوات |
|---|---|
| **الحل المفضّل (قبل أي بيع):** | 1. المدير يطلب من الكاشير إغلاق الوردية فوراً (countedCash = 0).<br>2. إعادة فتح وردية جديدة برصيد افتتاحي صحيح.<br>3. لا أثر محاسبي لأن الوردية المغلقة بلا مبيعات. |
| **الحل البديل (بعد بدء البيع):** | 1. إغلاق الوردية الحالية مع تدوين العجز.<br>2. فتح وردية جديدة برصيد صحيح.<br>3. توثيق العجز في حقل «ملاحظة».<br>4. إبلاغ المدير لمراجعة سجل التدقيق. |

### 3.3 أثناء اليوم (Mid-day Operations) 🛒

#### 3.3.1 دورة البيع المختصرة

| الخطوة | الإجراء | زر/شاشة |
|:---:|---|---|
| 1 | اختيار المنتج بالبحث أو مسح الباركود | حقل البحث أعلى POS |
| 2 | تحديد الكمية (+ / −) | أسفل كل سطر في السلة |
| 3 | ربط العميل (هاتف → بحث تلقائي) | حقل «هاتف العميل» |
| 4 | اختيار طريقة الدفع (نقدي / K-Net / Visa) | قائمة طريقة الدفع |
| 5 | (اختياري) تطبيق الخصم — ADMIN/MANAGER/OWNER فقط | حقل الخصم أسفل السلة |
| 6 | الضغط على «إتمام البيع بقيمة X» | أسفل السلة |
| 7 | تأكيد البيع (زر أخضر «اعتماد») | نافذة التأكيد |
| 8 | طباعة الإيصال الحراري (تلقائي إن مُفعَّل) | نافذة الإيصال |
| 9 | «بيع جديد» للعميل التالي | زر في نافذة الإيصال |

> 📄 **تفاصيل كل زر في POS:** راجع دليل `docs/03-pos.md` و دليل `docs/user-guides/cashier-guide.md`.

#### 3.3.2 تعليق (Park) واستئناف الفاتورة

| السيناريو | الإجراء |
|---|---|
| عميل يريد التريّث، وآخر ينتظر | 1. زر **«تعليق»** (Pause) في رأس السلة.<br>2. يُحفظ برقم `HOLD-XXX`.<br>3. السلة تُفرَّغ لخدمة العميل التالي.<br>4. عند العودة: زر **«المعلَّقات»** → اختيار الـ HOLD-XXX → زر استئناف. |
| سلة حالية + تريد استعادة معلَّق | النظام يحذّرك: «السلة الحالية ستُستبدَل» → اضغط «تأكيد الاستبدال». |
| حذف فاتورة معلَّقة | من قائمة المعلَّقات → زر سلة المهملات بجانب HOLD-XXX. |

> ⚠️ **مهم:** التعليق **لا يخصم من المخزون**. المخزون يُخصم فقط عند إتمام البيع فعلياً.

#### 3.3.3 المرتجع الجزئي (Partial Refund)

| القاعدة | التفصيل |
|---|---|
| **القاعدة الزمنية** | خلال **14 يوماً** من تاريخ الفاتورة. الثابت `MAX_REFUND_DAYS = 14`. |
| **التجاوز بعد 14 يوماً** | مربع **«موافقة المسؤول»** داخل نافذة المرتجع. |
| **الصلاحية الفعلية** | **ADMIN فقط** على الخادم. الزر يظهر لـ OWNER/ADMIN/MANAGER لكن الخادم يرفض غير ADMIN بـ 403. |
| **النتيجة** | إشعار دائن `CN-INV-XXXXXX` + إعادة المخزون + قيد محاسبي عكسي + تحديث `refundStatus`. |
| **الحد الأقصى للإرجاع** | `quantity − returnedQty` لكل سطر — لا يمكن تجاوز القابل للإرجاع. |

#### 3.3.4 التبديل (Exchange)

| القاعدة | التفصيل |
|---|---|
| **الفاتورة الأصلية** | إلزامية (`original-invoice-required`). |
| **نطاق المنتجات** | فقط منتجات كانت في الفاتورة الأصلية (`product-not-in-invoice`). |
| **المدة** | 14 يوماً (`invoice-too-old`). |
| **القيد المحاسبي** | قيد واحد بفرق الصافي (مدين/دائن حسب الإشارة). |
| **الصلاحية** | SALES / ADMIN (لوحة التبديل `exchanges`). |
| **الإيصال** | إيصال تبديل 80mm عبر `printExchangeReceipt`. |

#### 3.3.5 إلغاء الفاتورة (Full Cancel) 🚨

| الجانب | التفصيل |
|---|---|
| **الصلاحية** | OWNER + ADMIN + MANAGER |
| **السبب** | إلزامي (≥ 3 أحرف) |
| **خاضع لقاعدة 14 يوماً؟** | ❌ لا — الإلغاء إجراء إداري |
| **يُسمح إذا الفاتورة** | COMPLETED فقط (لا يمكن إلغاء ملغاة أو مُرتجعة بالكامل أو مُرتجعة جزئياً) |
| **الأثر** | `status = CANCELLED` + إعادة كل المخزون + قيد عكسي + أرشيف (لا حذف فعلي) |
| **سجل التدقيق** | `CANCEL_TXN` مع السبب |

> 📄 **الفرق بين الإلغاء والمرتجع والحذف:** راجع §6.3 و§6.4.

#### 3.3.6 متى يتدخل المدير؟

| الموقف | الزر/الإجراء | الصلاحية | الأثر |
|---|---|---|---|
| خصم > 10% من المجموع الفرعي | حقل الخصم يُرفض للكاشير | MANAGER/ADMIN/OWNER | `discount-requires-manager` للكاشير، يجب أن يُدخل المدير الخصم |
| فتح درج النقدية بدون بيع (NO_SALE) | زر «فتح الدرج» في الوضع Express | MANAGER+ (مسجَّل `DRAWER_OPEN` في التدقيق) | يُفتح الدرج للصيانة/الصرف |
| اعتماد وردية بعد الإغلاق | زر «اعتماد» | المدير | ينشئ قيود الترحيل |
| تجاوز 14 يوماً للمرتجع | مربع «موافقة المسؤول» | ADMIN | يُفعَّل زر الاعتماد |
| إنشاء قيد محاسبي يدوي | نافذة القيد اليدوي | المحاسب/ADMIN | قيد متوازن إلزامي |

#### 3.3.7 فتح درج النقدية بدون بيع (NO_SALE) 🔒

| الإجراء | التفصيل |
|---|---|
| **من يقوم به** | MANAGER أو أعلى |
| **الزر** | «فتح الدرج» في الوضع Express |
| **التسجيل** | يُكتَب `DRAWER_OPEN` في سجل التدقيق تلقائياً |
| **سبب شائع** | صرف فكة لعميل، صيانة، عدّ رقابي |
| **المراجعة** | على المدير مراجعة عدد مرات `DRAWER_OPEN` أسبوعياً — تكراره بلا مبرر يدل على شبهة |

### 3.4 إجراءات المخزون أثناء اليوم 📦

#### 3.4.1 استلام بضاعة (فاتورة مشتريات)

| الخطوة | الإجراء | المسؤول |
|:---:|---|---|
| 1 | فتح شاشة **المشتريات** ← تبويب **الفواتير** | أمين المخزن |
| 2 | زر **«فاتورة جديدة»** (أو «استلام وإنشاء فاتورة» من أمر شراء APPROVED) | أمين المخزن |
| 3 | اختيار **المورد** + **المخزن** + **التاريخ** | أمين المخزن |
| 4 | إضافة سطور الأصناف (المنتج + الكمية + التكلفة) | أمين المخزن |
| 5 | إدخال رسوم التكلفة الكاملة (للأجانب: جمارك/شحن/مصاريف) | أمين المخزن |
| 6 | الضغط على **«حفظ كمسودة»** أو **«حفظ وترحيل»** | أمين المخزن |
| 7 | عند الترحيل: ⚠️ نافذة تحذيرية تُنبّه أن المخزون والتكلفة ستُحدَّث | أمين المخزن |
| 8 | تأكيد الترحيل → `status = POSTED` + قيد محاسبي + تحديث المخزون | النظام |

> 📄 **الفرق بين DRAFT و POSTED:** DRAFT لا أثر له. POSTED يُحدِّث المخزون والتكلفة وذمم المورد والمحاسبة. **لا يمكن تعديل أو حذف فاتورة مرحَّلة**.

#### 3.4.2 تحويل بين المخازن

| الخطوة | الإجراء | ملاحظة |
|:---:|---|---|
| 1 | فتح شاشة **المخزون** ← تبويب **التحويلات** | — |
| 2 | اختيار «من مخزن» (المصدر) و«إلى مخزن» (الوجهة) — يجب أن يكونا مختلفين | `same-warehouse` يُرفض |
| 3 | إضافة الأصناف + الكميات (لا تتجاوز المتاح في المصدر) | `exceeds-available` يُرفض |
| 4 | الضغط على **«إنشاء تحويل (إرسال)»** | `TR-XXXXX` + خصم من المصدر (حالة OUT) |
| 5 | عند وصول البضاعة للوجهة: زر **«استلام»** | حالة RECEIVED + إضافة للوجهة |

> ⚠️ **بين مرحلتي OUT و RECEIVED البضاعة «بالطريق»** — خُصمت من المصدر ولم تُضف للوجهة بعد. أمين المخزن الوجهة يجب أن يستلم في النظام فور وصولها فعلياً.

#### 3.4.3 إنتاج تركيبة (Composition Production) 🧪

| الخطوة | الإجراء |
|:---:|---|
| 1 | فتح شاشة **التركيبات** (`compositions`) |
| 2 | اختيار التركيبة المطلوبة + زر **«إنتاج»** |
| 3 | إدخال **عدد وحدات الإنتاج** المطلوبة مباشرة |
| 4 | مراجعة جدول المكونات المُستهلَكة + شارات «مخزون غير كافٍ» الحمراء |
| 5 | الضغط على **«إنتاج»** — عملية ذرية (إما تنجح كلها أو تُلغى كلها) |
| 6 | النتيجة: خصم المكونات + إضافة الناتج للمخزون + سجل تدقيق `COMPOSITION_PRODUCED` |

> 📄 **تفاصيل الإنتاج والفرق عن الباقة:** راجع §6.2 ودليل `docs/10-bundles-compositions.md`.

### 3.5 إغلاق الوَردية (Shift Closing) 🔒

#### 3.5.1 قائمة إغلاق الوردية

| ✅ | المهمة | المسؤول |
|:---:|---|---|
| ✅ | الكاشير يتأكد من عدم وجود فواتير معلَّقة يمكن إتمامها | الكاشير |
| ✅ | الكاشير يطبع **ملخص الوردية** (إن لزم) | الكاشير |
| ✅ | المدير يحضر لعدّ النقدية + كشوف جهاز K-Net و Visa | المدير |
| ✅ | الكاشير يفتح شاشة **الورديات** ← يضغط زر **«إغلاق الوردية»** | الكاشير |
| ✅ | إدخال المعدود في 3 حقول: **نقدية / K-Net / Visa** | الكاشير + المدير معاً |
| ✅ | مراجعة الفروقات المعروضة (✓ أخضر / −X أحمر / +X أحمر) | المدير |
| ✅ | كتابة ملاحظة (اختياري) عند وجود فرق | الكاشير |
| ✅ | الضغط على زر **«إغلاق الوردية والمطابقة»** (أحمر) | الكاشير |
| ✅ | المدير يراجع الفروقات + يعتمد أو يرفض مع سبب | المدير |
| ✅ | النظام يُنشئ **قيد محاسبي** تلقائياً للعجز/الفائض | النظام |

#### 3.5.2 حساب الفروقات (Variance)

| الفرق | المعادلة | التفسير |
|---|---|---|
| `cashVariance` | `countedCash − (openingBalance + expectedCash)` | النقدية = المعدود − (الافتتاحي + المبيعات النقدية) |
| `knetVariance` | `countedKnet − expectedKnet` | K-Net = المعدود − مبيعات CARD |
| `visaVariance` | `countedVisa − expectedVisa` | Visa = المعدود − مبيعات TRANSFER |

#### 3.5.3 تفسير الفروقات

| الحالة | المعنى | الإجراء |
|---|---|---|
| ✓ فرق النقدية = 0 | مطابقة تامة | ✅ اعتماد فوري |
| فرق نقدي بسيط (±0.5%) | خطأ تقريب/فكة | ✅ اعتماد + ملاحظة |
| فرق نقدي كبير (عجز) | نقدية مفقودة / فاتورة لم تُسجَّل | 🚨 تحقيق قبل الاعتماد |
| فرق نقدي كبير (فائض) | فاتورة مزدوجة / خلل في العدّ | 🚨 إعادة العدّ + تحقيق |
| فرق K-Net/Visa ≠ 0 | 🚨 غالباً خطأ تصنيف وسيلة الدفع | راجع آخر الفواتير — هل سُجِّلت TRANSFER بدل CARD؟ |

> 💡 **نصيحة عملية:** فروقات K-Net و Visa شبه مستحيلة لأن أرصدتها تأتي من البنك تلقائياً. إن ظهرت، ابحث عن خطأ في التصنيف قبل الاعتماد.

#### 3.5.4 القيد المحاسبي للفرق

| الحالة | مدين | دائن |
|---|---|---|
| **عجز** (cashVariance < 0) | 5070 (مصروف عجز) | 1010 (النقدية) |
| **فائض** (cashVariance > 0) | 1010 (النقدية) | 4060 (إيراد فائض) |

> ⚠️ **تنبيه:** الحسابات 5070 و 4060 قد لا تكون موجودة في seed الافتراضي. راجع §6 لتفاصيل إنشائها.

#### 3.5.5 اعتماد المدير أو الرفض

| الإجراء | الخطوات |
|---|---|
| **اعتماد** | 1. المدير يراجع الفروقات.<br>2. الضغط على زر «اعتماد».<br>3. الحالة تنتقل من `CLOSED` (أو `BLIND_CLOSED`) إلى `APPROVED`.<br>4. يُنشأ قيد ترحيل المبيعات: مدين 1010/1020 — دائن 4010. |
| **رفض مع سبب** | 1. لا يوجد زر «رفض» صريح في الواجهة الحالية.<br>2. الحل العملي: إعادة فتح الوردية (إن لزم) أو توثيق الرفض في سجل التدقيق اليدوي + إبلاغ المالك. |

#### قائمة إغلاق اليوم

| ✅ | المهمة | المسؤول | الأداة |
|:---:|---|---|---|
| ✅ | التأكد من إغلاق كل الورديات (لا توجد OPEN) | المدير | شاشة الورديات |
| ✅ | مراجعة تقرير المبيعات اليومي | المدير | شاشة التقارير العامة، فلتر «اليوم» |

| ✅ | التحقق من عدد عمليات الإلغاء | المدير | سجل التدقيق ← فلتر CANCEL_TXN |
| ✅ | التحقق من نسبة الحذف (VOID_RATE) | المدير | لوحة المدير — البطاقة الملوّنة |

|---|---|---|---|

| نسبة الحذف الفردية للكاشير (void-rate) | < 3% | 3–5% | > 5% (مُعلَّم كمشبوه) |

| إلغاء فاتورة كاملة (CANCEL_TXN) يومياً | 0 | 1–2 | > 2 (كل واحد يحتاج سبب) |

## 4. الإجراءات الأسبوعية (Weekly Operations)

### 4.1 يوم السبت (أو بداية الأسبوع) 📅

#### قائمة بداية الأسبوع

| ✅ | المهمة | الشاشة | المسؤول |
|:---:|---|---|---|
| ✅ | مراجعة تقرير المبيعات الأسبوعي | التقارير العامة (فلتر آخر 7 أيام) | المدير + المالك |
| ✅ | مراجعة الأصناف الراكدة | التحليلات ← تبويب «الراكد» | المدير |
| ✅ | مراجعة نقاط الولاء — أعلى العملاء | تقرير الولاء (إن وُجدت شاشته) أو API | المدير / SALES |
| ✅ | مراجعة طلبات إعادة التموين | أوامر شراء PENDING_APPROVAL | أمين المخزن + المدير |
| ✅ | مراجعة مصفوفة الأداء (الأقسام والمنتجات) | شاشة التقارير ← تبويب «مصفوفة الأداء» | المدير |

#### 4.1.1 مراجعة الأصناف الراكدة

| الشارة | المعنى | الإجراء المقترح |
|---|---|---|
| 🔴 حمراء «لا مباع أبداً» | `quantitySold = 0` في الفترة | تخفيض سعر / باقة / تخليص |
| 🟡 كهرمانية «بطيء» | نسبة الدوران < 10% | مراجعة التسعير أو العرض |
| 🟢 خضراء | نسبة دوران طبيعية | لا إجراء |

> 💡 **قيمة الركود** = `currentStock × costPrice` = رأس مال مجمد. اهتم بهذا الرقم — كل دينار راكد = دينار غير مُستثمَر.

#### 4.1.2 مراجعة طلبات إعادة التموين

| المصدر | الإجراء | الأداة |
|---|---|---|
| **الأصناف تحت حد الطلب** | توليد أمر شراء تلقائي | زر **«جلب الأصناف المطلوبة من المورد»** في شاشة المشتريات |
| **أمر PENDING_APPROVAL** | مراجعة + اعتماد أو رفض | لوحة `PoApprovalPanel` (ADMIN فقط) |
| **أمر APPROVED بانتظار الاستلام** | متابعة المورد | تبويب «أوامر الشراء» |
| **أمر PENDING يدوي** | استلام أو إلغاء | قائمة MoreVertical للأمر |

### 4.2 منتصف الأسبوع 📅

#### 4.2.1 الجرد السريع (Spot Check) للأصناف الحساسة

| الخطوة | الإجراء |
|:---:|---|
| 1 | فتح شاشة **الجرد العشوائي** (`spotcheck`) |
| 2 | البحث عن صنف حساس بالاسم/الباركود (عطر غالي / ساعة / إكسسوار) |
| 3 | ⚠️ النظام **لا يعرض الرصيد الدفتري** (جرد أعمى) |
| 4 | عدّ فعلي للصنف على الرف |
| 5 | إدخال الكمية الفعلية |
| 6 | النتيجة فورية: 🟢 مطابق / 🔴 عجز / 🔵 فائض |

> 💡 **الفرق بين Spot Check و Stock Take:** Spot Check أعمى وسريع ولا يُعدّل المخزون ولا يُنشئ قيداً. Stock Take شامل ويُعدّل المخزون ويُنشئ قيداً محاسبياً.

#### 4.2.2 مراجعة مستحقات الموردين + السداد

| الخطوة | الإجراء |
|:---:|---|
| 1 | فتح شاشة **الموردون** (`suppliers`) — مراجعة الرصيد المستحق لكل مورد (شارة حمراء = مدينون) |
| 2 | فتح شاشة **سداد الموردين** (تبويب `payments` داخل المشتريات) |
| 3 | اختيار مورد + مبلغ + طريقة دفع (CASH/BANK/CHECK) + تاريخ + مرجع |
| 4 | ⚠️ النظام يرفض السداد الزائد عن الرصيد المستحق (`exceeds-balance`) ما لم يُفعَّل تجاوز إداري |
| 5 | الحفظ → يُنشأ قيد: مدين 2010 (ذمم دائنة) / دائن 1010 (نقدية) أو 1020 (بنك) |
| 6 | طباعة سند الصرف أو تصديره |

#### 4.2.3 مراجعة المرتجعات وأسبابها

| المؤشر | المصدر | الإجراء |
|---|---|---|
| عدد المرتجعات الأسبوعي | سجل التدقيق ← `SALE_REFUNDED` | قارن مع الأسبوع السابق |
| قيمة المرتجعات / المبيعات | التقارير العامة | حد طبيعي < 5% |
| الأصناف الأكثر إرجاعاً | تقرير حركة المخزون ← فلتر REFUND | راجع الجودة/التعبئة |
| الأسباب المتكررة | حقل الملاحظات في المرتجع | علاج السبب الجذري |

### 4.3 نهاية الأسبوع 📅

#### 4.3.1 تقرير الأداء العام (لوحة المالك)

| البطاقة | المصدر | القرار المحتمل |
|---|---|---|
| **مبيعات اليوم مقابل أمس** | لوحة المالك | هل هناك تراجع يستحق تحقيقاً؟ |
| **مصروفات اليوم (تقديري 75%)** | لوحة المالك | تقدير مبسَّط — راجع قائمة الدخل للمصروفات الفعلية |
| **صافي الربح التقديري** | لوحة المالك | لو سالب → مراجعة فورية |
| **الهامش %** | لوحة المالك | < 20% = مراجعة تسعير |

> ⚠️ **تنبيه:** النسبة 75% تقدير ثابت في الكود وليست محسوبة فعلياً من المصروفات. لرؤية المصروفات الحقيقية راجع قائمة الدخل في المحاسبة.

#### 4.3.2 مراجعة مصفوفة أداء الأقسام

| الحقل | التفسير | الحد الحرج 🚨 |
|---|---|---|
| الكمية الصافية | `quantitySold − returnedQty` | سالبة = مرتجعات أكثر من المبيعات |
| مجمل الربح | الإيراد − التكلفة | سالب = بيع تحت التكلفة |
| هامش % | الربح ÷ الإيراد × 100 | < 10% أحمر / 10–30% كهرماني / ≥ 30% أخضر |
| معدل الدوران | التكلفة ÷ متوسط المخزون | < 1 = المخزون لم يدرّ كاملاً |
| أيام في المخزون | 365 ÷ معدل الدوران | ≥ أيام الفترة = راكد |

#### 4.3.3 اعتماد أوامر شراء الأسبوع القادم

| الإجراء | المسؤول | الأداة |
|---|---|---|
| مراجعة مسودات الأوامر التلقائية (PENDING_APPROVAL) | ADMIN | لوحة `PoApprovalPanel` |
| تعديل الكميات/التكلفة لكل سطر إن لزم | ADMIN | زر «تعديل وقبول» |
| اعتماد كأوامر APPROVED | ADMIN | زر «اعتماد وقبول» |
| رفض مع سبب إلزامي | ADMIN | زر «رفض» |
| متابعة استلام الأوامر المعتمدة | أمين المخزن | زر «تأكيد الاستلام» عند وصول الشحنة |

## 5. الإجراءات الشهرية (Monthly Operations)

### 5.1 بداية الشهر 📅

#### قائمة بداية الشهر

| ✅ | المهمة | الشاشة | المسؤول |
|:---:|---|---|---|
| ✅ | مراجعة قائمة الدخل للشهر الماضي | المحاسبة ← قائمة الدخل | المحاسب + المالك |
| ✅ | مراجعة الميزانية العمومية | المحاسبة ← الميزانية العمومية | المحاسب |
| ✅ | مراجعة التدفقات النقدية | المحاسبة ← التدفقات النقدية | المحاسب |
| ✅ | مراجعة ميزان المراجعة | المحاسبة ← ميزان المراجعة | المحاسب |
| ✅ | مراجعة مصاريف الشهر السابق (رواتب + إيجار + تسويق) | المحاسبة ← المصروفات | المحاسب |
| ✅ | إدخال مصاريف الشهر الجديد إن لم تُسجَّل | المحاسبة ← المصروفات | المحاسب |

#### 5.1.1 قائمة الدخل — القراءة السريعة

| البند | المعنى | علامة الاستفسار |
|---|---|---|
| إجمالي الإيرادات | POS + Shopify | منخفض عن المتوقع؟ |
| (−) تكلفة البضاعة المباعة (COGS) | كمية × costPrice | مرتفعة بشكل غير طبيعي؟ |
| (=) مجمل الربح | الإيراد − التكلفة | سالب = خسارة على البضاعة |
| (−) الرواتب | مصروفات SALARY | ثابتة شهرياً |
| (−) المصروفات الإدارية | الإيجار/المرافق/الاشتراكات/التسويق | راجع كل فئة |
| (=) صافي الربح | مجمل الربح − المصروفات | 🔴 سالب = مراجعة شاملة |

#### 5.1.2 الميزانية العمومية — شارة التوازن

| الشارة | المعنى | الإجراء |
|---|---|---|
| ✅ **الميزانية متوازنة** (أخضر) | الأصول = الخصوم + حقوق الملكية | لا إجراء |
| ❌ **غير متوازنة** (أحمر) | هناك خلل في القيود | 🚨 تحقيق محاسبي فوري |

### 5.2 الجرد الشهري الكامل (Stock Take) 📅

#### 5.2.1 قائمة الجرد الشهري

| ✅ | المهمة | المسؤول | ملاحظة |
|:---:|---|---|---|
| ✅ | التخطيط: توقف البيع في القسم المُجرَّد أو المتجر كله | المدير | يُفضَّل بعد الإغلاق أو يوم عطلة |
| ✅ | طباعة كشف الجرد الأعمى (بدون رصيد دفتري) | أمين المخزن | زر «طباعة كشف جرد» |
| ✅ | توزيع العدّادين على الرفوف | المدير | عادة 2–4 موظفين |
| ✅ | العدّ الفعلي وكتابته على الورق | العدّادون | لا يستخدمون النظام أثناء العدّ |
| ✅ | البدء بجرد جديد في النظام | أمين المخزن | زر «جرد جديد» |
| ✅ | اختيار المخزن + القسم | أمين المخزن | النظام يحمِّل كل الأصناف تلقائياً |
| ✅ | إدخال الكمية الفعلية لكل صنف | أمين المخزن | حقل `actualQty` |
| ✅ | حفظ التقدم (يُحدِّث نفس المسودة) | أمين المخزن | زر «حفظ كمسودة» |
| ✅ | مراجعة ملخص العجز/الفائض الجانبي | أمين المخزن + المدير | قيم بالأحمر/الأخضر |
| ✅ | الاعتماد النهائي → يُنشئ قيد محاسبي للفروقات | المدير | زر «اعتماد نهائي» |
| ✅ | طباعة كشف الجرد الأعمى للأرشيف | أمين المخزن | بدون رصيد دفتري |

#### 5.2.2 الجرد الأعمى — قاعدة أساسية

> 🚨 **الجرد الأعمى:** لا يُعرض الرصيد الدفتري (`systemQty`) أثناء العدّ. العدّاد يُدخل ما يرى على الرف فقط. الفرق يُحسب ويُعرض **بعد** الإدخال.

#### 5.2.3 الأثر على المخزون والمحاسبة عند الاعتماد

| الحالة | الأثر على المخزون | القيد المحاسبي |
|---|---|---|
| **فائض** (الفعلي > الدفتري) | `incrementStockItem` (+variance) | مدين 1010 (المخزون) / دائن 4060 (إيرادات تسوية) |
| **عجز** (الفعلي < الدفتري) | `decrementStockItem` (−\|variance\|) — يُرفض إن لم يكن مخزون كافٍ | مدين 5070 (عجز مخزون) / دائن 1010 (المخزون) |
| **فرق = 0** | لا تغيير | لا قيد |

> 💡 **الحسابات 5070 و 4060** ليست في seed الافتراضي. يجب إنشاؤها يدوياً في دليل الحسابات قبل أول جرد (ADMIN فقط).

### 5.3 نهاية الشهر (Month-End Close) 📅

#### قائمة نهاية الشهر

| ✅ | المهمة | المسؤول | الشاشة |
|:---:|---|---|---|
| ✅ | التأكد من ترحيل كل فواتير المشتريات | أمين المخزن + المحاسب | المشتريات ← الفواتير |
| ✅ | إنشاء ملاحق التخليص الجمركي للفواتير المستوردة | المحاسب / أمين المخزن | الفاتورة ← «ملحق جمركي» |
| ✅ | مراجعة كشوف حسابات الموردين ومطابقتها | المحاسب | الموردون ← كشف حساب |
| ✅ | مراجعة تقارير VAT (إن كانت الدولة تفرضها) | المحاسب | المحاسبة ← تقرير ض.ق.م |
| ✅ | تقرير مؤشر كفاءة المنتج (100 نقطة) | المدير / المالك | التقارير |
| ✅ | أرشفة الفواتير الملغاة | المدير | الفواتير ← فلتر CANCELLED |
| ✅ | مراجعة قائمة الدخل النهائية للشهر | المحاسب + المالك | المحاسبة ← قائمة الدخل |

#### 5.3.1 ترحيل فواتير المشتريات

| الحالة | المعنى | الإجراء |
|---|---|---|
| `DRAFT` | لم تُرحَّل بعد — لا أثر على المخزون | ترحيل إذا استُلمت البضاعة فعلياً |
| `POSTED` | مُرحَّلة — أثرت على المخزون والمحاسبة | لا يمكن تعديل/حذف |
| `CANCELLED` | مُلغاة | للأرشيف فقط |

> 🚨 **مهم:** كل فاتورة DRAFT بنهاية الشهر تعني مخزوناً مُستلَماً فعلياً لكنه **غير مُسجَّل** في النظام. هذا يُربك الجرد والتقارير. رحِّل كل الفواتير قبل الجرد الشهري.

#### 5.3.2 ملاحق التخليص الجمركي (Customs Annex)

> 📄 التفاصيل الكاملة في §6.1. ملخص سريع:

| الخطوة | الإجراء |
|:---:|---|
| 1 | فتح شاشة المشتريات ← الفواتير |
| 2 | اختيار فاتورة POSTED لمورد **أجنبي** |
| 3 | زر «ملحق جمركي» ← نافذة الإنشاء |
| 4 | إدخال: نسبة جمارك% / نسبة ضريبة% / نسبة شحن% / مصاريف أخرى |
| 5 | حفظ كـ DRAFT |
| 6 | الترحيل (POST) → قيد محاسبي + رسملة التكاليف على قيمة المخزون |

#### 5.3.3 مراجعة كشوف حسابات الموردين

| الخطوة | الإجراء |
|:---:|---|
| 1 | فتح شاشة الموردين ← MoreVertical ← «كشف حساب» |
| 2 | ضبط الفلتر (من/إلى) لبداية ونهاية الشهر |
| 3 | مراجعة **الرصيد الافتتاحي** و**الرصيد الختامي** |
| 4 | مطابقة كل سطر مع المستند الأصلي (فاتورة/سداد/مرتجع) |
| 5 | تصدير CSV/Excel للأرشيف أو طباعة |

#### 5.3.4 تقرير VAT (ضريبة القيمة المضافة)

| البطاقة | المصدر | التفسير |
|---|---|---|
| **ضريبة المخرجات** | `sale.taxAmount` (مجموع) | الضريبة المحصَّلة من المبيعات |
| **ضريبة المدخلات** | فواتير المشتريات POSTED + أوامر استلام بلا فاتورة | الضريبة المدفوعة على المشتريات |
| **صافي الضريبة** | output − input | موجب = مستحقة السداد للدولة، سالب = قابلة للاسترداد |
| إجمالي المبيعات/المشتريات | بدون الضريبة | للإقرار |

> ⚠️ **ملاحظة لدولة الكويت:** الضريبة الافتراضية 0% (لا VAT حالياً). لمدى آخر راجع اللوائح الكويتية. هذا التقرير مهم للفروع في السعودية/الإمارات/مصر.

## 6. الإجراءات الحرجة (Critical Procedures)

### 6.1 ملحق التخليص الجمركي (Customs Annex) 🚨

#### 6.1.1 متى يُنشأ؟

| الشرط | التفصيل |
|---|---|
| **فاتورة مشتريات POSTED** | إلزامي — لا يمكن إنشاء ملحق لفاتورة DRAFT |
| **مورد أجنبي** | منطقياً — المورد المحلي لا يدفع جمارك |
| **بعد وصول البضاعة فعلياً** | لأن الملحق يُرسمل التكاليف على قيمة المخزون |

#### 6.1.2 الحقول المطلوبة

| الحقل | النوع | مثال |
|---|---|---|
| `customsRate` | نسبة % | 5 (أي 5% من قيمة الفاتورة) |
| `taxRate` | نسبة % | 15 (ضريبة استيراد) |
| `shippingRate` | نسبة % | 2 (شحن بحري) |
| `otherCharges` | مبلغ ثابت | 50 د.ك (مصاريف clearance) |
| `billOfLading` | نص (اختياري) | B/L No. — رقم بوليصة الشحن |
| `arrivalDate` | تاريخ (اختياري) | تاريخ وصول الحاوية |
| `note` | نص (اختياري) | ملاحظة |

#### 6.1.3 الحساب التلقائي

```
customsAmount   = subtotal × customsRate / 100
taxAmount       = subtotal × taxRate / 100
shippingAmount  = subtotal × shippingRate / 100
totalAnnexCost  = customsAmount + taxAmount + shippingAmount + otherCharges
```

رقم الملحق: `ANX-XXXXXXXX` (آخر 8 أرقام من Date.now).

#### 6.1.4 تأثيره على قيمة المخزون (رسملة التكاليف)

> 🚨 **رسملة التكاليف (Cost Capitalization):** عند ترحيل الملحق، تُضاف قيمة `totalAnnexCost` إلى تكلفة البضاعة المستوردة، فترتفع قيمة المخزون في الميزانية، ويُعاد حساب **التكلفة المتوسطة المرجَّحة** للمنتجات.

#### 6.1.5 ملاحق متعددة لفاتورة واحدة

> ✅ النظام **يسمح بعدة ملاحق لفاتورة واحدة**. مثال: شحنة وصلت على دفعتين، كل دفعة لها بوليصة شحن ومصاريف جمركية مستقلة. كل ملحق يُنشأ ويُرحَّل على حدة.

#### 6.1.6 الترحيل (POST)

| الخطوة | الإجراء |
|:---:|---|
| 1 | فتح الملحق (DRAFT) |
| 2 | مراجعة الحسابات تلقائياً |
| 3 | زر «ترحيل» → `status = POSTED` |
| 4 | يُنشأ **قيد محاسبي** تلقائياً: مدين 1010 (المخزون) / دائن 2010 (ذمم دائنة للجمارك/الشاحن) |
| 5 | سجل تدقيق: `CUSTOMS_ANNEX_POSTED` |

### 6.2 إنتاج تركيبة (Composition Production) 🧪

#### 6.2.1 شروط الإنتاج

| الشرط | التفصيل |
|---|---|
| **تركيبة فعّالة** | `isActive = true` |
| **مخزون كافٍ لكل مكون** | `stock ≥ ingredient.quantity × productionQty` |
| **مخزن افتراضي محدد** | النظام يحل المخزن الافتراضي تلقائياً |
| **الصلاحية** | OWNER/ADMIN/WAREHOUSE |

#### 6.2.2 خطوات الإنتاج

| الخطوة | الإجراء | ملاحظة |
|:---:|---|---|
| 1 | فتح شاشة التركيبات (`compositions`) | — |
| 2 | اختيار التركيبة المطلوبة | — |
| 3 | زر **«إنتاج»** أسفل البطاقة | يفتح `ProduceDialog` |
| 4 | إدخال **الكمية المطلوبة مباشرة** (عدد وحدات الإنتاج) | مبسَّط من مفهوم «عدد الدفعات» |
| 5 | مراجعة جدول المكونات المُستهلَكة | شارة حمراء لكل مكون ناقص |
| 6 | مراجعة التكلفة الإجمالية | `costPerUnit × quantity` |
| 7 | الضغط على **«إنتاج»** | عملية ذرية |
| 8 | النجاح: خصم المكونات + إضافة الناتج + سجل تدقيق `COMPOSITION_PRODUCED` | — |

#### 6.2.3 ماذا يحدث تلقائياً؟

| الأثر | التفصيل |
|---|---|
| **خصم المكونات** | كل مكون يُخصم من `StockItem` بـ `ingredient.quantity × productionQty` |
| **إضافة الناتج** | يُضاف `productionQty` إلى `StockItem` للمنتج الناتج |
| **مزامنة Product.quantity** | يُعاد حسابه من مجموع `StockItem` |
| **سجل التدقيق** | `COMPOSITION_PRODUCED` |
| **شارة POS** | المنتج الناتج يحمل `isManufactured = true` ويظهر وسم **«تركيبة»** بجانب اسمه في POS |

> ⚠️ **الفحص المسبق:** النظام يفحص كل المكونات قبل أي تعديل. لو أي مكون ناقص، تُرفض العملية بالكامل بقائمة كل النواقص (`insufficient-stock`) — لا تكرار.

### 6.3 إلغاء فاتورة (Invoice Cancellation) 🚨

#### 6.3.1 من يقدر؟

| الدور | يقدر؟ | ملاحظة |
|---|:---:|---|
| 👑 OWNER | ✅ | — |
| 🛡️ ADMIN | ✅ | — |
| ✅ MANAGER | ✅ | — |
| 🧮 ACCOUNTANT | ❌ | — |
| 🏭 WAREHOUSE | ❌ | — |
| 🛒 SALES | ❌ | — |
| 🛒 CASHIER | ❌ | — |

#### 6.3.2 سبب إلزامي

| القاعدة | التفصيل |
|---|---|
| الحد الأدنى للطول | 3 أحرف (`reason-required` يُرفض لو أقل) |
| الحقل | نص حر في `CancelDialog` |
| التخزين | `cancellationReason` على سجل الفاتورة |
| سجل التدقيق | `CANCEL_TXN` مع السبب |

#### 6.3.3 ماذا يعكس النظام تلقائياً؟

| الجانب | الأثر |
|---|---|
| **المخزون** | إعادة كل الكمية (`quantity − returnedQty`) إلى المخزن الحالي للمستخدم |
| **القيد المحاسبي** | قيد عكسي: دائن حساب الدفع (رد المبلغ) / مدين 4010 (عكس الإيراد) + مدين 2010 (عكس الضريبة) |
| **نقاط الولاء** | تُخصم نقاط الفاتورة من العميل (إن كان مرتبطاً) |
| **حالة الفاتورة** | `status = CANCELLED` + `refundStatus = FULL` + `refundTotal = total` |
| **الفاتورة في القائمة** | تبقى ظاهرة بشارة «ملغاة» — لا تُحذف |

#### 6.3.4 الفرق بين الإلغاء والمرتجع

| الجانب | الإلغاء (Cancel) | المرتجع (Refund) |
|---|---|---|
| **النطاق** | الفاتورة كاملة | صنف أو أكثر (جزئي أو كلي) |
| **قاعدة 14 يوماً** | ❌ لا تنطبق (إجراء إداري) | ✅ تنطبق (مع تجاوز إداري) |
| **السبب** | إلزامي (≥ 3 أحرف) | اختياري |
| **الصلاحية** | OWNER/ADMIN/MANAGER | ADMIN فقط فعلياً |
| **رقم المرجع** | لا (الفاتورة نفسها) | `CN-INV-XXXXXX` |
| **الاستخدام النموذجي** | فاتورة مكررة / خطأ إداري / غش موظف | عميل أعاد بضاعة |

#### 6.3.5 متى يُمنع الإلغاء؟

| الحالة | الرمز | الرسالة |
|---|---|---|
| فاتورة ملغاة أصلاً | `already-cancelled` | «الفاتورة ملغاة أصلاً» |
| فاتورة مُرتجعة بالكامل | `already-refunded` | «الفاتورة مُرتجعة بالكامل» |
| فاتورة مُرتجعة جزئياً | `has-partial-refund` | «لا يمكن إلغاء فاتورة بها مرتجع جزئي» |

## 7. الإجراءات الطارئة (Emergency Procedures)

### 7.1 انقطاع الإنترنت 🚨

#### 7.1.1 ماذا يعمل النظام؟

| السؤال | الجواب |
|---|---|
| هل يدعم النظام العمل دون اتصال (Offline)? | ❌ لا — قاعدة البيانات على Supabase (السحابة) |
| هل تُخزَّن المبيعات محلياً ثم تُزامَم? | ❌ لا — كل عملية تحتاج اتصالاً لحظياً |
| ماذا يحدث للفاتورة قيد الإتمام? | تفشل العملية برمز `checkout-failed` |

#### 7.1.2 ماذا يفعل الكاشير؟

| الخطوة | الإجراء |
|:---:|---|
| 1 | إبلاغ مدير الوردية فوراً |
| 2 | التوقف عن إدخال فواتير جديدة في النظام |
| 3 | استخدام **نموذج بيع ورقي** (يُحتفظ به في المتجر) — تدوين: رقم فاتورة يدوي، اسم العميل، الأصناف، الكمية، السعر، طريقة الدفع |
| 4 | خصم المخزون يدوياً على ورقة (لكي لا يُباع صنف نافد) |
| 5 | عند عودة الإنترنت: إدخال كل فواتير الورق في النظام كبيع عادي |
| 6 | مطابقة الكميات الورقية مع النظام |

#### 7.1.3 متى يوقف البيع؟

| مدة الانقطاع | الإجراء |
|---|---|
| < 5 دقائق | انتظر — قد يعود سريعاً |
| 5–15 دقيقة | ابدأ التسجيل اليدوي على الورق |
| > 15 دقيقة | 🚨 أوقف البيع مؤقتاً وانتظر عودة الإنترنت، خاصة للمنتجات الثمينة |

> 💡 **نصيحة:** احتفظ بـ «طوارئ كاشير» في المتجر: دفتر فواتير ورقي، آلة حاسبة، قائمة أسعار مطبوعة محدَّثة شهرياً.

### 7.2 انقطاع الكهرباء ⚡

#### 7.2.1 الإجراءات الفورية

| الخطوة | الإجراء |

| 1 | عدم محاولة إتمام أي فاتورة قيد المعالجة |
| 2 | إغلاق آمن للأجهزة عند عودة الكهرباء (لا تطفئ من الزر أثناء الكتابة) |
| 3 | فحص درج النقدية — قد يكون مفتوحاً |
| 4 | عند عودة الكهرباء: إعادة تشغيل الأجهزة بالتسلسل (مودم → كاشير → طابعة → قارئ) |

#### 7.2.2 استعادة البيانات

| السؤال | الجواب |
|---|---|
| هل تفقد أي بيانات? | ❌ لا — قاعدة البيانات على Supabase (السحابة) |
| ماذا عن الفاتورة قيد الإتمام? | إن اكتملت قبل الانقطاع = محفوظة. إن لم تكتمل = لم تُحفظ أصلاً |
| ماذا عن الوردية? | محفوظة بحالتها قبل الانقطاع |
| ماذا عن المخزون? | محفوظ — آخر عملية ناجحة هي ما يُحسب |

> ✅ **الخلاصة:** لا فقدان للبيانات. أسوأ سيناريو = إعادة إدخال فاتورة واحدة قيد المعالجة.

### 7.4 نزاع مع عميل (مرتجع/تبديل رفضه النظام) 😤

| الحالة | الإجراء |
|---|---|
| النظام يرفض المرتجع (> 14 يوماً) وبدون تجاوز إداري | 1. صعود للمدير.<br>2. المدير يفعِّل مربع «موافقة المسؤول».<br>3. إجراء المرتجع. |
| النظام يرفض التبديل (`product-not-in-invoice`) | 1. صعود للمدير.<br>2. التحقق من الفاتورة الأصلية — هل المنتج فعلاً فيها?<br>3. إن لم يكن: النظام محق. الحل: مرتجع نقدي خارج النظام + توثيق، أو رفض الطلب. |
| النظام يرفض الإلغاء (`already-refunded`) | لا يمكن — الإلغاء غير ممكن لفاتورة بها مرتجع. الحل: مرتجع آخر للكمية المتبقية. |
| النزاع مستمر | 1. 🚨 صعود للمالك.<br>2. توثيق في سجل التدقيق اليدوي (نص حر).<br>3. في الحالات القصوى: إلغاء الفاتورة + فتح فاتورة جديدة بالشروط الصحيحة. |

> ⚠️ **مهم:** كل تجاوز إداري يُسجَّل في سجل التدقيق. لا تخف من استخدامه عند الحاجة الفعلية، لكن وثّق السبب.

### 7.5 شك بسلوك كاشير (نسبة حذف مرتفعة) 🚨

#### 7.5.1 مؤشرات الشك

| المؤشر | المصدر | الحد الحرج |
|---|---|---|
| نسبة الحذف الفردية > 3% | `useVoidRate` API | يُعلَّم في سجل التدقيق كمشبوه |
| فتح درج بدون بيع متكرر (DRAWER_OPEN) | سجل التدقيق | > 5 مرات/يوم |
| نسبة الحذف/المرتجع الإجمالية > 10% | لوحة المدير | شارة حمراء |
| إلغاء فاتورة بأسباب متكررة | سجل التدقيق | نفس السبب لفواتير متعددة |
| عجز نقد متكرر في ورديات نفس الكاشير | شاشة الورديات | > 2 وردية/أسبوع |

#### 7.5.2 إجراءات التحقيق

| الخطوة | الإجراء | الأداة |
|:---:|---|---|
| 1 | مراجعة سجل التدقيق — فلتر `VOID_ITEM` | شاشة التدقيق |
| 2 | مراجعة فتح الدرج بدون بيع — فلتر `DRAWER_OPEN` | شاشة التدقيق |
| 3 | تقرير تحليل الحذف لكل كاشير | API `useVoidRate` |
| 4 | مراجعة فواتير الكاشير المشبوه | شاشة الفواتير — فلتر باسم البائع |
| 5 | مراجعة ورديات الكاشير — الفروقات | شاشة الورديات |
| 6 | مراجعة المرتجعات التي أجراها (إن كان ADMIN) | شاشة الفواتير — فلتر REFUND |

#### 7.5.3 الإجراء التأديبي

| المستوى | الإجراء |
|---|---|
| **مستوى 1 — تنبيه** | لقاء مع المدير + توثيق |
| **مستوى 2 — مراقبة** | تفعيل الإغلاق الأعمى (Blind Close) لورديات الكاشير — راجع `docs/12-shifts.md` §5 |
| **مستوى 3 — إجراء رسمي** | توقيف مؤقت + مراجعة المحاسب |
| **مستوى 4 — فصل** | بقرار المالك + إلغاء صلاحياته في النظام (ADMIN يغيِّر الدور أو يوقف الحساب) |

> 🚨 **مهم:** كل الإجراءات تُوثَّق في سجل التدقيق. لا تخف من مواجهة الكاشير بالبيانات — النظام يسجِّل كل شيء بدقة ولا يكذب.

## 🎯 الخلاصة التنفيذية

| الدورة | الأهمية الأكبر |
|---|---|
| **يومياً** | افتتاح وردية برصيد صحيح + إغلاق بمطابقة + مراجعة تنبيهات التدقيق |
| **أسبوعياً** | مراجعة الراكد + الولاء + مستحقات الموردين + مصفوفة الأداء |
| **شهرياً** | الجرد الكامل + قائمة الدخل + ترحيل كل الفواتير + ملاحق الجمارك |
| **عند الطوارئ** | تسجيل ورقي فوري + إبلاغ المدير + توثيق في سجل التدقيق |

> 📌 **القاعدة الذهبية:** النظام لا يَنسى شيئاً. كل عملية تُسجَّل. مهمتك كمدير/مالك أن **تراجع** لا أن **تُدخِل** — راجع سجل التدقيق، راجع الفروقات، راجع التنبيهات. النظام يحميك إذا حاسبته يومياً.
