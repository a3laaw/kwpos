# 09 — محرك التسعير والترويجات وسجل تغيير الأسعار

> دليل تدريبي مبسّط لوحدة التسعير في نظام KWPOS. يغطي محرك التسعير (Retail / Wholesale / Corporate)، منطق حساب السعر الفعلي على الخادم، الترويجات (نوع الخصم ونطاقه وتاريخه)، سجل تغيير الأسعار (PriceChange)، منع التلاعب بالأسعار من جانب العميل، شاشة محرك التسعير وكل زر فيها. مكتوب لغير المختصين تقنياً مع الإشارة للملفات المصدرية.

---

## 1. نظرة عامة على محرك التسعير

يملك كل منتج في النظام **ثلاثة أسعار بيع** مخزّنة في جدول `Product` (ملف `prisma/schema.prisma`):

| الحقل | يستخدم لـ |
|---|---|
| `salePrice` | العملاء العاديين (RETAIL) |
| `wholesalePrice` | عملاء الجملة (WHOLESALE) |
| `corporatePrice` | عملاء الشركات (CORPORATE) |

قد يُضاف خصم مؤقت عبر **الترويجات** (Promotions)، فيُصبح السعر الفعلي = السعر الأساسي للفئة بعد تطبيق الخصم.

الملف المسؤول عن المنطق: `src/lib/pricing.ts`.

---

## 2. الأسعار الثلاثة وكيفية اختيارها

دالة `tierBasePrice(product, tier)` تختار السعر الأساسي:

```
إذا tier = WHOLESALE  و wholesalePrice > 0  ← اختار wholesalePrice
إذا tier = CORPORATE  و corporatePrice > 0  ← اختار corporatePrice
خلاف ذلك                                  ← اختار salePrice
```

> الحماية: إذا كان سعر الجملة أو الشركات = 0 (لم يُضبط)، يرجع النظام تلقائياً لسعر التجزئة. هذا يمنع بيع المنتج بسعر صفر.

| فئة العميل | الحقل المُستخدَم | مثال (salePrice=10, wholesalePrice=7, corporatePrice=8) |
|---|---|---|
| **RETAIL** | `salePrice` | 10 |
| **WHOLESALE** | `wholesalePrice` | 7 |
| **CORPORATE** | `corporatePrice` | 8 |
| **WHOLESALE (سعر جملة = 0)** | يرجع لـ `salePrice` | 10 |

## 3. كيف يعمل التسعير على الخادم (Server-side Pricing)

الملف الأساسي: `src/lib/sale/totals.ts` → دالة `computeSaleTotals`.

### الخطوات

1. تُستقبل سلة البيع من العميل (POS) وتحتوي فقط على `productId` و `quantity`. **سعر الوحدة المُرسَل من العميل يُهمَل تماماً**.
2. يُحمَّل المنتج من قاعدة البيانات بكل حقول الأسعار + الفئة.
3. تُستدعى `computeEffectivePrice(product, customerTier, promotions)` لكل سطر في السلة.
4. يُحسَب `lineSubtotal = quantity × serverUnitPrice`.
5. يُحسَب الضريبة لكل سطر على حدة (أو على مستوى السلة إذا لم يُضبط معدل المنتج).
6. المجموع = `subtotal − discount + tax + deliveryFee`.

### الكود المختصر

```ts
// SECURITY: unitPrice يُحسَب على الخادم فقط
const priced = computeEffectivePrice(
  {
    id: it.productId,
    categoryId: product?.categoryId,
    salePrice: toNum(product?.salePrice),
    wholesalePrice: toNum(product?.wholesalePrice),
    corporatePrice: toNum(product?.corporatePrice),
  },
  customerTier,
  promotions // تُجلب من الخادم
)
const serverUnitPrice = priced.effectivePrice
```

### منع التلاعب بالأسعار من جانب العميل

| الإجراء | التفاصيل |
|---|---|
| **تجاهل `unitPrice` المُرسَل** | العميل لا يملك سلطة على السعر — يُرسِل `productId + quantity` فقط |
| **الترويجات تُجلب من الخادم** | قائمة الترويجات تُحمَّل داخل معالج الـ API، لا يُعتمد على ما يرسله العميل |
| **الحساب داخل معاملة (Transaction)** | لا يمكن تنصّل الكاشير من السعر المعتمد |
| **سجل تدقيق (PriceChange)** | كل تغيير في الأسعار يُسجَّل بشكل دائم وغير قابل للتعديل (انظر القسم 7) |

> هذا التصميم يحمي من سيناريوهات التلاعب مثل تعديل سعر الوحدة في أدوات المطور قبل إرسال الفاتورة.

## 4. الترويجات (Promotions)

### الحقول (من جدول `Promotion` في `prisma/schema.prisma`)

| الحقل | النوع | الوصف |
|---|---|---|
| `scope` | String | نطاق التطبيق: `PRODUCT` / `CATEGORY` / `ALL` / `ALL_EXCEPT_CATEGORIES` |
| `productId` | String? | مطلوب فقط عندما `scope = PRODUCT` |
| `categoryIdsJson` | String? | مصفوفة JSON لمعرّفات الفئات (لـ `CATEGORY` و `ALL_EXCEPT_CATEGORIES`) |
| `discountType` | String | `PERCENT` (نسبة مئوية) أو `AMOUNT` (مبلغ ثابت) |
| `discountValue` | Float | النسبة (0-100) أو المبلغ بالعملة |
| `startAt` | DateTime | تاريخ بداية الترويج |
| `endAt` | DateTime | تاريخ نهاية الترويج |
| `isActive` | Boolean | مفتاح التفعيل (يُغلَق تلقائياً عند انتهاء الفترة) |
| `note` | String? | ملاحظة اختيارية |
| `createdById` | String? | المُنشِئ |

### أنواع الخصم (discountType)

| النوع | المعنى | الصيغة |
|---|---|---|
| **PERCENT** | نسبة مئوية | `effectivePrice = base × (1 − value/100)` |
| **AMOUNT** | مبلغ ثابت بالعملة | `effectivePrice = max(0, base − value)` |

> الحد الأقصى لـ PERCENT = 100% (لا يمكن أن يتجاوز). يُقبَل 0-100 فقط.

### نطاقات التطبيق (scope)

| النطاق | شرط التطبيق على المنتج |
|---|---|
| `PRODUCT` | يجب أن يطابق `promo.productId` معرّف المنتج |
| `CATEGORY` | يجب أن تكون فئة المنتج ضمن `promo.categoryIds` |
| `ALL` | يُطبَّق على كل المنتجات دائماً |
| `ALL_EXCEPT_CATEGORIES` | يُطبَّق على كل المنتجات **ما عدا** فئات `promo.categoryIds` (المنتج بلا فئة يُطبَّق عليه) |

### منطق اختيار الترويج الفعّال

في `computeEffectivePrice`:

1. تُفلتر الترويجات بـ `isActive = true`.
2. تُفلتر بمنطق `promotionAppliesTo` (نطاق التطبيق).
3. تُفلتر بالتاريخ (الوقت الحالي ضمن `[startAt, endAt]`).
4. تُرتَّب تصاعدياً حسب `startAt` ويُختار **الأول** (الأقدم بداية).

> إذا تعدّدت الترويجات الفعّالة على منتج واحد، يفوز الأقدم تاريخ بدء — لا تُجمَع الترويجات.

## 5. كيف تُجلب الترويجات وتُطبَّق على الخادم

عند إنشاء فاتورة بيع:

| الخطوة | المكان |
|---|---|
| 1. يحمّل الخادم كل الترويجات `isActive = true` | داخل `POST /api/sales` |
| 2. تُمرَّر إلى `computeSaleTotals` | `src/lib/sale/totals.ts` |
| 3. لكل سطر، تُطبَّق `computeEffectivePrice` | `src/lib/pricing.ts` |
| 4. يُحسَب `effectivePrice` = السعر الأساسي للفئة بعد الترويج | يُستخدم كـ `unitPrice` للسطر |

> قائمة الترويجات تُحمَّل **على الخادم فقط**. العميل (POS) يعرض معاينة الأسعار فقط لأغراض العرض، لكن السعر الحقيقي الذي يُحفَظ في الفاتورة هو ما حسبه الخادم.

---

## 6. التفعيل/الإلغاء التلقائي للترويج

في `GET /api/pricing/route.ts`، عند كل استدعاء:

- **إلغاء تلقائي**: أي ترويج `isActive = true` وانتهى `endAt` < الآن ← يُغلَق (`isActive = false`).
- **تفعيل تلقائي**: أي ترويج `isActive = false` وبدأ `startAt` ≤ الآن ولم ينتهِ ← يُفتَح (`isActive = true`).

> لا حاجة لمهمة مجدولة (cron) — يكفي فتح شاشة التسعير ليتزامن الوضع.

## 7. سجل تغيير الأسعار (PriceChange)

الملف: `src/app/api/pricing/audit/route.ts` + جدول `PriceChange` في `prisma/schema.prisma`.

### الحقول

| الحقل | الوصف |
|---|---|
| `productId` | المنتج المُعدَّل |
| `priceType` | `RETAIL` / `WHOLESALE` / `CORPORATE` |
| `oldPrice` | السعر القديم |
| `newPrice` | السعر الجديد |
| `changedById` | المستخدم الذي غيّر |
| `changedAt` | تاريخ التغيير |
| `note` | ملاحظة اختيارية |

### الخاصية الأساسية

> **السجل غير قابل للتعديل أو الحذف.** لا يوجد API من نوع `PATCH` أو `DELETE` على `/api/pricing/audit`. كل تغيير سعر يُكتَب داخل معاملة (Transaction) مع تحديث المنتج، فلا يمكن فصلهما.

### ما يعرضه السجل

| العمود | الوصف |
|---|---|
| التاريخ | `changedAt` بصيغة datetime |
| المنتج | اسم المنتج + الباركود |
| النوع | شارة RETAIL/WHOLESALE/CORPORATE |
| من | `oldPrice` |
| إلى | `newPrice` |
| التغيير | الفرق (`newPrice − oldPrice`) — أخضر للموجب، أحمر للسالب |
| بواسطة | `changedBy.name` |
| الملاحظة | `note` (شرطة إن لم توجد) |

> يُرجِع الـ API آخر 500 سجل مرتبة تنازلياً، ويُرشِّح العميل (client) البحث نصياً.

---

## 8. منع التلاعم بأسعار العميل (Client-side Tampering)

| التهديد | الحماية في النظام |
|---|---|
| تعديل `unitPrice` في طلب الفاتورة | يُهمَل تماماً — السعر يُحسَب من `Product` على الخادم |
| إرسال قائمة ترويجات مزيّفة | لا تُقبَل — تُجلب من قاعدة البيانات على الخادم |
| تخطي فحص التكلفة | فحص على الخادم: يُرجِع 409 `below-cost` قبل التطبيق |
| حذف سجل تغيير سعر | لا يوجد `DELETE` على `/api/pricing/audit` — السجل دائم |
| تخفيف سعر أدنى من التكلفة | يحتاج `confirm: true` صريح من المدير (تحت صفة ADMIN) |

## 10. تبويب الأسعار (PriceManagementTab)

### تخطيط الجدول

| العمود | الحقل | ملاحظات |
|---|---|---|
| **المنتج** | `name` | نص مقطوع (truncate) |
| **الباركود** | `barcode` | خط Mono، LTR |
| **الفئة** | `categoryName` | شارة رمادية |
| **سعر التكلفة** | `costPrice` | يظهر فقط لمن يملك `canSeeCost` (ADMIN/OWNER/MANAGER) |
| **ضريبة %** | `taxRate` | حقل قابل للتعديل (يُحدِّث المنتج مباشرة) |
| **RETAIL** | `salePrice` | حقل قابل للتعديل (ADMIN فقط) |
| **WHOLESALE** | `wholesalePrice` | حقل قابل للتعديل (ADMIN فقط) |
| **CORPORATE** | `corporatePrice` | حقل قابل للتعديل (ADMIN فقط) |
| **الترويج الفعّال** | `activePromotion` | شارة خضراء بنوع الخصم وقيمته |

### الأزرار والإجراءات

| العنصر | الوظيفة | الظهور |
|---|---|---|
| **حقل البحث** | بحث في الاسم/الباركود | دائماً |
| **شارة «للعرض فقط — يمكن للأدمن التعديل»** | قفل للقراءة | لغير ADMIN |
| **زر «إلغاء N تغييرات»** | مسح التغييرات المؤجلة | عند وجود تغييرات مرحلية (staged) |
| **خانة سعر (خلية)** | تعديل قيمة الخلية | ADMIN فقط |
| **شريط سفيلي «اعتماد وتطبيق»** | تطبيق كل التغييرات المؤجلة | ADMIN فقط |
| **عداد التغييرات المؤجلة** | شارة برقم N | في الزر |
| **حقل ضريبة %** | تعديل معدل الضريبة للمنتج | دائماً (يستدعي `PUT /api/products/[id]`) |

### منطق التطبيق (Cost Guard)

1. الكاشير/المدير يُعدِّل الخلايا — تُخزَّن محلياً (staged).
2. عند الضغط على «اعتماد وتطبيق»:
   - **فحص العميل**: يُسرد المنتجات التي سعرها الجديد أدنى من `costPrice`.
   - إن وُجدت: تُفتَح نافذة تحذير (below-cost modal) بقائمة المنتجات المخالفة.
   - إن لم توجد: تُفتَح نافذة تأكيد بسيطة.
3. عند التأكيد، يُرسَل `POST /api/pricing` بـ `confirm: true` (في حالة below-cost) أو `false`.
4. يكتب الـ API سجل `PriceChange` لكل تغيير، ثم يُحدِّث المنتج في معاملة واحدة.

### نافذة التأكيد (PricingConfirmDialog)

| الزر | الوظيفة |
|---|---|
| **«رجوع» (إلغاء)** | يغلق دون تطبيق |
| **«اعتماد وتطبيق»** | يُنفِّذ التغييرات |
| تلميح `Ctrl+Enter` | اختصار لوحة المفاتيح للتطبيق |

> لا يمكن إغلاق النافذة بالضغط خارجها (backdrop) أو بمفتاح Escape أثناء التحميل — حماية من الإغلاق العرضي.

## 11. تبويب الترويجات (PromotionsTab)

### تخطيط التبويب

عمودان: **نموذج الإنشاء** (يسار، 2/5) + **قائمة الترويجات** (يمين، 3/5).

### نموذج إنشاء ترويج

| الحقل | النوع | التحقق |
|---|---|---|
| **نطاق التطبيق** (`scope`) | Select | PRODUCT / CATEGORY / ALL / ALL_EXCEPT_CATEGORIES |
| **المنتج** | Combobox | مطلوب إذا scope = PRODUCT |
| **الفئات المُضمَّنة** | Checkboxes | مطلوب فئة واحدة على الأقل إذا scope = CATEGORY |
| **الفئات المُستثناة** | Checkboxes | مطلوب فئة واحدة على الأقل إذا scope = ALL_EXCEPT_CATEGORIES |
| **نوع الخصم** | Select | PERCENT / AMOUNT |
| **القيمة** | Input رقمي | `> 0`، و`≤ 100` لـ PERCENT |
| **من تاريخ** (`startAt`) | datetime-local | مطلوب، افتراضي = الآن |
| **إلى تاريخ** (`endAt`) | datetime-local | مطلوب، افتراضي = بعد 7 أيام، يجب أن يكون بعد `startAt` |
| **ملاحظة** | Input نصي | اختياري |
| **زر «إنشاء ترويج»** | Submit | يُرسِل `POST /api/promotions` |

### قائمة الترويجات الحالية

كل بطاقة ترويج تعرض:

| العنصر | الوصف |
|---|---|
| **العنوان** | اسم المنتج (لـ PRODUCT) أو الفئات (CATEGORY) أو «الكل» (ALL) أو «الكل ما عدا» (ALL_EXCEPT_CATEGORIES) |
| **شارة النطاق** | اختصار (منتج/فئة/الكل/الكل ما عدا) |
| **شارة الحالة** | «فعّال الآن» (أخضر) / «مجدول» (أصفر) / «موقوف» (رمادي) |
| **الخصم** | `−X%` أو `−X د.ك` |
| **الفترة** | `startAt ← endAt` |
| **الملاحظة** | نص إن وُجد |
| **المُنشِئ** | اسم المستخدم |
| **زر «إيقاف»** | يضع `isActive = false` (ADMIN فقط) |

> لا يوجد زر «تعديل» للترويج — الإيقاف هو الإجراء الوحيد. التعديل يتطلب إنشاء ترويج جديد. الحذف الناعم (soft-delete) يُبقي السجل للمراجعة التاريخية.

### سلوك لغير ADMIN

- لا يظهر نموذج الإنشاء.
- تظهر قائمة الترويجات للقراءة فقط بدون زر «إيقاف».

## 13. APIs المرجعية

| المسار | الطريقة | الدور | الوظيفة |
|---|---|---|---|
| `/api/pricing` | GET | أي مستخدم مسجّل | قائمة المنتجات بأسعارها + الترويج الفعّال |
| `/api/pricing` | POST | OWNER/ADMIN/MANAGER (`canManagePricing`) | تطبيق تغييرات أسعار مجمّعة + سجل تدقيق |
| `/api/pricing/effective` | GET | أي مستخدم مسجّل | `?productId=&tier=` سعر فعلي لمنتج واحد |
| `/api/pricing/audit` | GET | أي مستخدم مسجّل | آخر 500 سجل تغيير سعر (للقراءة فقط) |
| `/api/promotions` | GET | أي مستخدم مسجّل | قائمة كل الترويجات |
| `/api/promotions` | POST | OWNER/ADMIN/MANAGER (`canManagePricing`) | إنشاء ترويج جديد |
| `/api/promotions?id=xxx` | DELETE | OWNER/ADMIN/MANAGER | إيقاف ترويج (سوفت-حذف فقط) |

## 15. خلاصة تدريبية

1. **ثلاثة أسعار** لكل منتج (Retail/Wholesale/Corporate) + ترويج اختياري = السعر الفعلي.
2. **السعر يُحسَب على الخادم** دائماً — لا يُعتمد على ما يرسله العميل، فلا تلاعب.
3. **الترويج** له نطاق (منتج/فئة/الكل/الكل ما عدا فئات) + نوع (نسبة/مبلغ) + فترة زمنية + مفتاح تفعيل.
4. **التفعيل/الإلغاء تلقائي** عند فتح شاشة التسعير، دون حاجة لمهمة مجدولة.
5. **سجل تغييرات الأسعار دائم** — يُكتَب في معاملة واحدة مع تحديث المنتج، ولا يمكن تعديله أو حذفه.
6. **التحذير من الأسعار أدنى من التكلفة** يظهر في نافذة تأكيد قبل التطبيق، ويتطلب `confirm: true` صريح.
7. **تبويبات الشاشة**: الأسعار / الترويجات / الباقات / سجل التغييرات — مع حفظ حالة التبويب.
8. **الإيقاف (soft-deactivate)** هو الإجراء الوحيد على الترويج القائم — لا حذف ولا تعديل، حفاظاً على السجل التاريخي.
