# 10 — الباقات (Bundles) والتركيبات (Compositions)

> دليل تدريبي مبسّط لوحدتي الباقات والتركيبات في نظام KWPOS. يشرح الفرق الجوهري بينهما، نموذج المصنع في التركيبات، تأثير كل منهما على المخزون، كل زر في الشاشتين، والشارات الظاهرة في نقطة البيع. مكتوب لغير المختصين تقنياً مع الإشارة للملفات المصدرية.

---

## 1. الباقة (Bundle) — ما هي؟

**الباقة = تجميع عدة منتجات معاً وبيعها بسعر موحّد.** لا يُنتج النظام منتجاً جديداً، بل يبيع المنتجات الموجودة مجتمعة بسعر مخفّض.

### مثال

باقة «العطور الثلاثة» تحتوي على:
- عطر ورد × 1
- عطر زعفران × 1
- عطر مسك × 1

سعر الباقة = 25 د.ك (بدلاً من 30 د.ك لو اشتريتها منفردة).

### خصائص الباقة

| الخاصية | القيمة |
|---|---|
| منتج جديد؟ | لا — الباقة تستخدم المنتجات الموجودة |
| تأثير المخزون عند التعريف؟ | لا شيء (المخزون يتأثر فقط عند البيع) |
| تأثير المخزون عند البيع؟ | يُخصَم من كل منتج بحسب كميته في الباقة |
| السعر | ثابت على مستوى الباقة (`salePrice`) |
| التفعيل الموسمي | اختياري (`startDate` / `endDate`) |
| الصلاحية | الإدارة: ADMIN/WAREHOUSE، الحذف: ADMIN فقط |

الملف: `src/components/bundles/bundles-view.tsx` و `src/app/api/bundles/route.ts`.

| **السعر** | `salePrice` بخط عريض ملوّن |
| **إجمالي أسعار التجزئة** | `itemsRetailTotal` مشطوب (للمقارنة) |
| **عدد المنتجات** | شارة برقم `items.length` |

| العنصر | الوظيفة |
|---|---|
| **حقل بحث المنتج** | يبحث بالاسم في `useProducts` |

| **زر سلة المهملات** | يحذف السطر |
| **عرض التكلفة لكل سطر** | `unitCost × quantity` + `lineSale` |

### ملخص التكلفة الحيّ

| المؤشر | الصيغة | اللون |
|---|---|---|
| **إجمالي التكلفة** | `Σ(costPrice × qty)` | رمادي |
| **إجمالي أسعار التجزئة** | `Σ(salePrice × qty)` | رمادي |
| **الربح** | `salePrice − totalCost` | أخضر إن موجب، أحمر إن سالب |
| **نسبة الخصم %** | `(retailTotal − salePrice) / retailTotal × 100` | أخضر/أحمر |

### أزرار الإجراءات

| الزر | الوظيفة |
|---|---|
| **«إلغاء»** | يغلق النموذج دون حفظ |

## 4. تأثير الباقة على المخزون

### عند التعريف / التعديل

**لا تأثير** — لا تُخصَم كميات عند إنشاء الباقة. المخزون يبقى كما هو.

### عند بيع الباقة

> ملاحظة: منطق بيع الباقة يقع في وحدة نقطة البيع (`src/components/sales/`)، ويُخصَم من كل منتج في الباقة بحسب كميته. الآلية الكاملة موصوفة في `03-pos.md` و `04-sales-invoices-refunds-cancel.md`.

| الحدث | التأثير |
|---|---|
| بيع 1 باقة فيها منتج A × 2 و B × 1 | يُخصَم 2 من A و 1 من B |
| إرجاع الباقة | يُعاد 2 إلى A و 1 إلى B (تبعاً لمنطق المرتجع) |

### التفعيل/الإيقاف التلقائي

في `GET /api/bundles/route.ts`:

- **إيقاف تلقائي**: باقة `isActive = true` و`endDate < الآن` ← `isActive = false`.
- **تفعيل تلقائي**: باقة `isActive = false` و`startDate ≤ الآن` و`endDate ≥ الآن` ← `isActive = true`.

> لا حاجة لمهمة مجدولة — يكفي فتح شاشة الباقات لتتزامن الحالة.

## 5. APIs الباقات

| المسار | الطريقة | الدور | الوظيفة |
|---|---|---|---|
| `/api/bundles` | GET | أي مستخدم مسجّل | قائمة الباقات + حقول التكلفة تُحجَب لـ SALES/CASHIER |
| `/api/bundles` | POST | OWNER/ADMIN/WAREHOUSE | إنشاء باقة + منتجاتها (معاملة واحدة) |
| `/api/bundles/[id]` | GET | أي مستخدم مسجّل | تفاصيل باقة واحدة |
| `/api/bundles/[id]` | PUT | OWNER/ADMIN/WAREHOUSE | تعديل (يستبدل المنتجات إن أُرسلت) |
| `/api/bundles/[id]` | DELETE | OWNER/ADMIN فقط | حذف باقة (cascade حذف BundleItem) |

> كل إنشاء/تعديل/حذف يُسجَّل في سجل التدقيق (`logAuditEvent`) بإجراء `BUNDLE_CREATED` / `BUNDLE_UPDATED` / `BUNDLE_DELETED`.

### التحقق من الـ API

| الحالة | الكود | الرسالة |
|---|---|---|
| اسم مكرر | 409 | `name-exists` |
| لا يوجد منتجات | 400 | `items-required` |
| سعر غير موجب | 400 | `sale-price-required` |
| منتج غير موجود | 400 | `invalid-product` |
| تاريخ غير صالح | 400 | `invalid-start-date` / `invalid-end-date` |

## 6. التركيبة (Composition) — ما هي؟

**التركيبة = وصفة تصنيع: مكونات خام تُستهلَك من المخزون لإنتاج منتج جديد.** النظام يتبنّى نموذج «المصنع» (factory model) — تُنتج التركيبة منتجاً جديداً يُضاف للمخزون.

### مثال

تركيبة «العود الملكي»:
- زيت عود × 5 جرام
- بخور × 3 جرام
- مسك × 2 جرام

الناتج: منتج جديد اسمه «العود الملكي» بحجم 10 جرام.

### حالات الاستخدام

| المجال | مثال |
|---|---|
| **محل عطور** | دهن عود + بخور + مسك → خليط مخصّص |
| **مطاعم** | 0.5 كجم أرز + 0.5 دجاج + توابل → وجبة |
| **تصنيع** | قائمة مكونات (BOM) لمنتج مُجمَّع |

الملف: `src/components/compositions/compositions-view.tsx` و `src/app/api/compositions/route.ts`.

## 8. نموذج إضافة/تعديل تركيبة

الملف: `src/components/compositions/composition-form-dialog.tsx`.

### الحقول الرئيسية

| الحقل | النوع | الإلزام | الملاحظات |
|---|---|---|---|
| **صورة التركيبة** | ImageUpload | لا | رفع صورة |
| **اسم التركيبة** | Input | نعم | يجب أن يكون فريداً |
| **الوصف** | Textarea | لا | سطران |
| **المنتج الناتج** | Toggle + Combobox | نعم | زرّان: «إنشاء منتج جديد» أو «استخدام منتج موجود» |
| **مبلغ الربح** | Input رقمي | — | مبلغ ثابت بالعملة (لا نسبة)، يُضاف لـ `costPrice` |
| **النسبة التقريبية** | عرض حسابي فقط | — | `profitAmount / costPerUnit × 100` |
| **سعر البيع التلقائي** | عرض حسابي فقط | — | `costPerUnit + profitAmount` |
| **وحدة الإنتاج** | Select | — | من قائمة الوحدات في الإعدادات، افتراضي «قطعة» |
| **ملاحظات** | Textarea | لا | تعليمات التحضير |
| **فعّالة** | Checkbox | — | افتراضي = مفعّل |
| **قائمة المكونات** | جدول ديناميكي | نعم (مكون واحد على الأقل) | كل سطر: منتج + كمية + وحدة + ملاحظة |

### نموذج المصنع — إنشاء منتج جديد تلقائياً

عند اختيار «إنشاء منتج جديد» (الافتراضي) عند الإنشاء:

| الحقل في المنتج الجديد | القيمة |
|---|---|
| `name` | اسم التركيبة (نفس الاسم) |
| `unit` | `yieldUnit` (وحدة الإنتاج) |
| `costPrice` | `Σ(ingredient.costPrice × quantity) / yieldQty` (التكلفة للوحدة) |
| `salePrice` | `costPrice + profitAmount` (مبلغ ربح ثابت بالعملة) |
| `isManufactured` | `true` (شارة «تركيبة» في POS) |
| `quantity` | `0` (يبدأ بصفر، يُضاف عند الإنتاج) |

> إذا اختار المستخدم «استخدام منتج موجود»، يُربَط المنتج ويُوضَع `isManufactured = true` دون إنشاء جديد.

### لماذا «مبلغ ربح ثابت» وليس نسبة؟

القرار تصميمي في النظام: الربح **مبلغ ثابت بالعملة** يُضاف فوق التكلفة، وليس نسبة مئوية. هذا يبسّط التسعير ويضمن أن تغيّر تكاليف المكونات لا يخفض الربح بالخطأ.

| المدخل | الناتج |
|---|---|
| `profitAmount = 2 د.ك` و `costPerUnit = 5 د.ك` | `salePrice = 7 د.ك` (نسبة ≈ 40%) |
| `profitAmount = 2 د.ك` و `costPerUnit = 8 د.ك` | `salePrice = 10 د.ك` (نسبة ≈ 25%) |

> النسبة المئوية تُعرَض للمستخدم كمعونة بصرية فقط (`≈ X% من التكلفة`)، لكنها **لا تُستخدم** في حساب السعر.

### إدارة المكونات

| العنصر | الوظيفة |
|---|---|
| **حقل بحث المنتج** | يبحث في `useProducts` |
| **Combobox اختيار مكون** | يعرض المنتج + سعر تكلفته، يستثني المضافين + المنتج الناتج |
| **زر «+ إضافة مكوّن»** | يضيف سطراً جديداً بالكمية الافتراضية 1 |
| **حقل الكمية** | رقمي قابل للتعديل |
| **حقل الوحدة** | Select من قائمة الوحدات (افتراضي «جرام») |
| **حقل الملاحظة** | نص حر (مثلاً: «يُضاف ببطء») |
| **زر سلة المهملات** | يحذف السطر |

### ملخص التكلفة الحيّ

| المؤشر | الصيغة |
|---|---|
| **التكلفة لكل دفعة** | `Σ(ingredient.costPrice × quantity)` |
| **التكلفة للوحدة** | `costPerBatch / yieldQty` |

### أزرار الإجراءات

| الزر | الوظيفة |
|---|---|
| **«إلغاء»** | يغلق دون حفظ |
| **«حفظ»** / **«إضافة»** | يُرسِل `POST` أو `PUT` |

## 9. إنتاج دفعة (Produce Batch)

الملف: `src/components/compositions/produce-dialog.tsx` و `src/app/api/compositions/[id]/produce/route.ts`.

### نافذة الإنتاج

| الحقل | الوصف |
|---|---|
| **الكمية** | عدد **وحدات الإنتاج** المطلوب إنتاجها مباشرة (مبسّطة من مفهوم «عدد الدفعات» السابق) |
| **ملخص الإنتاج** | `producedQty + yieldUnit` + اسم المنتج الناتج |
| **جدول المكونات المُستهلَكة** | كل مكون: الكمية المطلوبة × الكمية = الإجمالي، مع الكمية المتاحة ومؤشر النقص |
| **شارة «مخزون غير كافٍ»** | حمراء لكل مكون ناقص |
| **التكلفة للوحدة** | `costPerUnit` |
| **التكلفة الإجمالية** | `costPerUnit × quantity` |
| **زر «إنتاج»** | يؤكد العملية |

### منطق الإنتاج على الخادم

```ts
const quantity = Math.max(1, Math.floor(body?.batches ?? body?.quantity))
```

> المستخدم يُدخِل **عدد وحدات الإنتاج المطلوبة مباشرة** (وليس عدد مرات تكرار الوصفة). مثال: إدخال 10 = إنتاج 10 وحدات من المنتج الناتج.

### الخطوات الذرية (داخل `db.$transaction`)

1. تحميل التركيبة + مكوناتها + المنتج الناتج.
2. حلّ المخزون الافتراضي (`getDefaultWarehouseId`).
3. **فحص مسبق لكل المكونات**: إن كان أي مكون ناقصاً، يُجمَع في قائمة وتُرجَع برسالة `400 insufficient-stock` قبل أي تعديل.
4. **خصم كل مكون** بمقدار `ingredient.quantity × quantity` من `StockItem`.
5. **إضافة كمية المنتج الناتج** بمقدار `quantity` إلى `StockItem`.
6. **مزامنة `Product.quantity`** من `StockItem` (مجموع كل المخازن).
7. **سجل تدقيق** بإجراء `COMPOSITION_PRODUCED`.
8. إرجاع `{ ok: true, produced, unit }`.

> إما أن تنجح كل الخطوات أو تُلغى كلها — لا حالة وسيطة.

### نافذة نقص المخزون

عند الإرجاع `insufficient-stock`، تُعرَض قائمة بكل مكون ناقص:

| الحقل | الوصف |
|---|---|
| اسم المكون | اسم المنتج |
| المتاح | الكمية الحالية في المخزون |
| المطلوب | `ingredient.quantity × quantity` |
| الوحدة | جرام/مل/قطعة... |

> الفحص المسبق يضمن عرض **كل** النواقص دفعة واحدة، فلا يضطر المستخدم لإصلاح مكون واحد في كل مرة.

## 10. وسم «تركيبة» في نقطة البيع

عند إنشاء تركيبة، يُوضَع `isManufactured = true` على المنتج الناتج. في نقطة البيع السريعة (`src/components/sales/express-pos-view.tsx`)، يظهر وسم صغير بجانب اسم المنتج:

```
[تركيبة]
```

> النص الكامل: `manufacturedProduct` = «مُصنّع»، النص المختصر: `manufacturedProductShort` = «تركيبة» (انظر `src/lib/i18n/inventory.ts`).

هذا التنبيه البصري يساعد الكاشير على تمييز المنتجات المُنتَجة داخلياً عن المشتراة جاهزة.

## 11. الفرق الجوهري بين الباقة والتركيبة

| المعيار | الباقة (Bundle) | التركيبة (Composition) |
|---|---|---|
| **المفهوم** | بيع منتجات معاً بسعر مخفّض | تصنيع منتج جديد من مكونات |
| **المنتج الناتج** | لا — تستخدم المنتجات الموجودة | نعم — تُنشئ منتجاً جديداً (factory model) |
| **التأثير على المخزون عند التعريف** | لا شيء | لا شيء |
| **التأثير على المخزون عند البيع** | يُخصَم من كل منتج بحسب كميته | يُخصَم المنتج الناتج فقط (المكونات خُصِمَت عند الإنتاج) |
| **التأثير على المخزون عند الإنتاج** | لا ينطبق | يُخصَم من كل مكون + يُضاف للمنتج الناتج |
| **السعر** | ثابت على الباقة (`salePrice`) | يُحسَب آلياً = التكلفة + مبلغ ربح ثابت |
| **الربح** | `salePrice − totalCost` (متفاوت) | مبلغ ثابت بالعملة (`profitAmount`) |
| **وحدة الإنتاج** | لا تنطبق | `yieldUnit` (جرام/مل/قطعة...) |
| **التاريخ الموسمي** | اختياري (`startDate`/`endDate`) | لا (تفعيل/إيقاف يدوي فقط) |
| **الفئة** | حقل نصي حر | لا |
| **علم التصنيع** | لا | `isManufactured = true` على المنتج الناتج |
| **وسم POS** | لا | «تركيبة» بجانب اسم المنتج |
| **المسار في API** | `/api/bundles` | `/api/compositions` (+ `/produce`) |
| **إجراءات الإنتاج** | لا | `POST /api/compositions/[id]/produce` |
| **الفئات** | `Bundle` + `BundleItem` | `Composition` + `CompositionIngredient` |
| **جدول الـ API لإعادة العناصر** | يحلَّل ويُعاد إنشاؤه عند PUT | نفس الشيء |
| **الوحدة لكل سطر** | لا (كمية فقط) | كمية + وحدة + ملاحظة |

## 15. APIs التركيبات

| المسار | الطريقة | الدور | الوظيفة |
|---|---|---|---|
| `/api/compositions` | GET | أي مستخدم مسجّل | قائمة التركيبات + مكوناتها + المنتج الناتج |
| `/api/compositions` | POST | OWNER/ADMIN/WAREHOUSE | إنشاء تركيبة (يُنشئ المنتج الناتج آلياً إن لزم) |
| `/api/compositions/[id]` | GET | أي مستخدم مسجّل | تفاصيل تركيبة واحدة |
| `/api/compositions/[id]` | PUT | OWNER/ADMIN/WAREHOUSE | تعديل (يستبدل المكونات إن أُرسلت) |
| `/api/compositions/[id]` | DELETE | OWNER/ADMIN فقط | حذف التركيبة (cascade حذف المكونات) |
| `/api/compositions/[id]/produce` | POST | OWNER/ADMIN/WAREHOUSE | إنتاج دفعة (خصم مكونات + إضافة ناتج) |

> كل إنشاء/تعديل/حذف/إنتاج يُسجَّل في سجل التدقيق: `COMPOSITION_CREATED` / `COMPOSITION_UPDATED` / `COMPOSITION_DELETED` / `COMPOSITION_PRODUCED`.

### رموز الأخطاء الشائعة

| الكود | السبب |
|---|---|
| `name-exists` (409) | اسم التركيبة مكرر |
| `ingredients-required` (400) | لا توجد مكونات |
| `output-product-required` (400) | لم يُحدَّد منتج ناتج |
| `yield-required` (400) | `yieldQty ≤ 0` |
| `invalid-product` (400) | منتج مكون غير موجود |
| `invalid-output-product` (400) | المنتج الناتج غير موجود |
| `insufficient-stock` (400) | مخزون مكون ناقص + قائمة المكونات |
| `no-warehouse` (500) | لا يوجد مخزون افتراضي مضبوط |

## 17. خلاصة تدريبية

1. **الباقة** = بيع منتجات معاً بسعر مخفّض. لا يُنتج منتجاً جديداً. المخزون يتأثر فقط عند البيع.
2. **التركيبة** = تصنيع منتج جديد من مكونات خام. تُنتج منتجاً جديداً بـ `isManufactured = true`، والمخزون يتأثر عند الإنتاج (خصم مكونات + إضافة ناتج).
3. **التركيبة تتبنّى نموذج المصنع**: يُنشأ المنتج الناتج آلياً داخل معاملة واحدة مع التركيبة. لا حاجة لإنشاء المنتج يدوياً قبل.
4. **الربح في التركيبة مبلغ ثابت بالعملة** (لا نسبة مئوية)، يُضاف فوق التكلفة. النسبة تُعرَض للمستخدم كمعونة بصرية فقط.
5. **الإنتاج عملية ذرية**: إما تنجح كلها (خصم مكونات + إضافة ناتج + سجل تدقيق) أو تُلغى كلها.
6. **الفحص المسبق** في الإنتاج يعرض كل المكونات الناقصة دفعة واحدة، فلا تكرار.
7. **الوسم في POS**: المنتجات المُنتَجة تحمل وسم «تركيبة» بصرياً يميّزها عن المشتراة جاهزة.
8. **الباقات موسمية اختيارياً** (تواريخ بداية/نهاية)، التركيبات لا تحتاج تواريخ موسمية.
9. **الصلاحيات متطابقة**: إنشاء/تعديل = ADMIN/WAREHOUSE، حذف = ADMIN فقط.
10. **كل العمليات مسجَّلة** في سجل التدقيق (`logAuditEvent`) لمراجعة لاحقة.
