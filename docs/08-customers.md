# 08 — العملاء ونقاط الولاء وكشوف الحساب

> دليل تدريبي مبسّط لوحدة العملاء في نظام KWPOS. يغطي شاشة قائمة العملاء، نموذج الإضافة/التعديل/الحذف، أنواع العملاء وتأثيرها على السعر، نظام النقاط والفئات (BRONZE/SILVER/GOLD)، كشف حساب العميل، تقرير الولاء، ربط العملاء بفواتير البيع، ومفهوم «العميل النقدي» (Walk-in). مكتوب لغير المختصين تقنياً مع الإشارة للملفات المصدرية.

---

## 1. شاشة قائمة العملاء

الملف المسؤول: `src/components/customers/customers-view.tsx`.

تعرض العملاء في **جدول مبسّط** بدلاً من بطاقات. كل صف يعرض: اسم العميل (مع دائرة تحوي أول حرفين من الاسم)، رقم الهاتف (LTR)، العنوان، تاريخ الإضافة، وقائمة إجراءات منسدلة (MoreVertical).

| العمود | الحقل | ملاحظات |
|---|---|---|
| **اسم العميل** | `name` | دائرة ملوّنة بأول حرفين + الاسم |
| **الهاتف** | `phone` | خط Mono، اتجاه LTR، شرطة «—» إذا فارغ |
| **العنوان** | `address` | يظهر على الشاشات المتوسطة فأكبر، مع أيقونة خريطة |
| **تاريخ الإضافة** | `createdAt` | بصيغة التاريخ المحلي، يظهر على الشاشات الكبيرة |
| **الإجراءات** | قائمة MoreVertical | تعديل / حذف (حسب الصلاحية) |

### شريط البحث

حقل بحث واحد (`Search`) يبحث في **الاسم + الهاتف + العنوان** معاً (تحليل حساسية الحالة عبر الـ API). البحث مؤجَّل (Debounced) لتقليل الطلبات.

### رأس الشاشة وأزرارها

| العنصر | الظهور |
|---|---|
| زر **استيراد Excel** | OWNER / ADMIN / MANAGER فقط |
| زر **تصدير Excel** | OWNER / ADMIN / MANAGER فقط |
| زر **«إضافة عميل»** | OWNER / ADMIN / MANAGER / SALES (البيع يحتاج لتسجيل العملاء النقديين) |

> ملاحظة: دور SALES يمكنه إضافة العملاء واحداً واحداً لكن **لا يمكنه** الاستيراد/التصدير المجمّع لقاعدة بيانات العملاء.

## 2. إضافة / تعديل / حذف عميل

الملف: `src/components/customers/customer-form-dialog.tsx`.

### حقول النموذج

| الحقل | الإلزام | الملاحظات |
|---|---|---|
| **الاسم** | نعم | لا يقبل الفراغ |
| **الهاتف** | لا | اتجاه LTR، نص placeholder: `+965 5xxx xxxx` |
| **العنوان** | لا | نص حر |

### أزرار النموذج

- **«إلغاء»** — يغلق النموذج دون حفظ.
- **«حفظ»** (في التعديل) أو **«إضافة»** (في الإنشاء) — يُرسل البيانات للـ API.

### سلوك الحفظ

- **إنشاء**: `POST /api/customers` — يُرجع 201 عند النجاح.
- **تعديل**: `PUT /api/customers/[id]` — يُحدّث الحقول المُرسَلة فقط.
- **حذف**: زر «حذف» في قائمة MoreVertical يفتح نافذة تأكيد، ثم `DELETE /api/customers/[id]`.

### ملفات API

| المسار | الدور المطلوب | الوظيفة |
|---|---|---|
| `GET /api/customers` | أي مستخدم مسجّل | قائمة العملاء مع دعم البحث `?q=` |
| `POST /api/customers` | OWNER/ADMIN/MANAGER/SALES | إنشاء عميل |
| `PUT /api/customers/[id]` | OWNER/ADMIN/MANAGER/SALES | تعديل بيانات العميل |
| `DELETE /api/customers/[id]` | OWNER/ADMIN/MANAGER فقط | حذف عميل (إجراء تدميري) |

> ملاحظة هامة: النموذج الظاهر للمستخدم (`customer-form-dialog.tsx`) لا يعرض حالياً حقل **نوع العميل** (type). ومع ذلك، يقبل الـ API في `POST /api/customers/route.ts` الحقل `type` بقيم `RETAIL | WHOLESALE | CORPORATE` ويفترض `RETAIL` إذا لم يُذكر. الـ `PUT` الحالي لا يُعدّل الحقل `type` إطلاقاً — لذلك يبقى نوع العميل `RETAIL` افتراضياً عند الإنشاء من هذه الشاشة. لتغيير النوع يجب تعديله مباشرة في قاعدة البيانات أو عبر API مخصّص (لم يظهر في الكود).

## 3. أنواع العملاء وتأثيرها على السعر

يحمل كل عميل حقلاً اسمه `type` في الجدول `Customer` (ملف `prisma/schema.prisma`)، وله ثلاث قيم:

| النوع | المعنى | الحقل الذي يُستخدم كسعر بيع |
|---|---|---|
| **RETAIL** (افتراضي) | عميل تجزئة عادي | `Product.salePrice` |
| **WHOLESALE** | عميل جملة | `Product.wholesalePrice` (إن > 0، وإلا يرجع لـ `salePrice`) |
| **CORPORATE** | عميل شركات/مؤسسات | `Product.corporatePrice` (إن > 0، وإلا يرجع لـ `salePrice`) |

### أين يُطبَّق ذلك؟

دالة `tierBasePrice` في `src/lib/pricing.ts` تختار السعر الأساسي حسب نوع العميل. ثم تستخدمها دالة `computeEffectivePrice` لحساب السعر الفعلي بعد تطبيق أي ترويج نشط.

دالة `computeSaleTotals` في `src/lib/sale/totals.ts` تستقبل `customerTier` وتُمرّره لـ `computeEffectivePrice` لكل سطر في السلة.

### مثال عملي

لو منتج سعره: `salePrice=10`، `wholesalePrice=7`، `corporatePrice=8`:

| نوع العميل | السعر الأساسي قبل الترويج |
|---|---|
| RETAIL | 10 |
| WHOLESALE | 7 |
| CORPORATE | 8 |

> إذا كان `wholesalePrice=0` (لم يُضبط)، يرجع النظام تلقائياً لـ `salePrice=10` حتى لعملاء الجملة — حماية من بيع بسعر صفر.

## 4. نظام نقاط الولاء (Loyalty Points)

### كيف تُكتسب النقاط؟

عند إتمام أي فاتورة بيع مرتبطة بعميل (وليست بيعاً نقدياً بدون عميل)، تُحتسب النقاط تلقائياً في ملف `src/lib/sale/side-effects.ts`:

```
pointsEarned = Math.floor(afterDiscount)
```

أي **نقطة واحدة عن كل وحدة عملة** من صافي قيمة الفاتورة بعد الخصم (قبل الضريبة والتوصيل). مثال: فاتورة بـ 125.7 ديناراً بعد الخصم = 125 نقطة.

### أين تُخزَّن؟

حقلا `loyaltyPoints` (Int، افتراضي 0) و `loyaltyTier` (String?، افتراضي null) في جدول `Customer` بـ `prisma/schema.prisma`.

### استعلام النقاط والاسترداد

الملف: `src/app/api/customers/[id]/points/route.ts`.

| الطريقة | الدور | الوظيفة |
|---|---|---|
| `GET` | أي مستخدم مسجّل | استعلام رصيد النقاط + الفئة الحالية |
| `POST` | OWNER/ADMIN/SALES | `action: "add"` لإضافة أو `action: "redeem"` لخصم نقاط |

- الإضافة لا تتطلب رصيداً كافياً.
- الاسترداد يتحقق من الرصيد أولاً، ويرفض (400 `insufficient-points`) إذا كان المطلوب أكبر من المتاح.

## 5. فئات الولاء (Loyalty Tiers)

تُحدَّد الفئة تلقائياً حسب إجمالي النقاط. عتبات الفئات في `src/lib/sale/side-effects.ts`:

| الفئة | الحد الأدنى للنقاط | الترتيب |
|---|---|---|
| **BRONZE** (برونزية) | 1,000 نقطة | الأدنى |
| **SILVER** (فضية) | 5,000 نقطة | الوسط |
| **GOLD** (ذهبية) | 10,000 نقطة | الأعلى |
| — (بدون فئة) | أقل من 1,000 | null |

### متى تُحدَّث الفئة؟

- **بعد كل فاتورة بيع**: دالة `runPostSaleSideEffects` تُعيد حساب الفئة وتحدّثها في `Customer.loyaltyTier`.
- **عند الإضافة/الاسترداد اليدوي** عبر `POST /api/customers/[id]/points`: نفس العتبات تُطبَّق (لاحظ أن العتبات مكررة في الملفين).

> ملاحظة: لا توجد شاشة واجهة منفصلة لعرض رصيد نقاط عميل أو فئته داخل شاشة العملاء (`customers-view.tsx`). الحقول تُرجَع من الـ API لكنها **لم تظهر معروضة** في الشاشة الحالية. للاطلاع على رصيد عميل، يمكن استخدام `GET /api/customers/[id]/points` مباشرة.

## 6. كشف حساب العميل

الملف: `src/app/api/customers/[id]/statement/route.ts`.
شاشة العرض: `src/components/accounting/customer-statement-tab.tsx` (داخل وحدة المحاسبة، تبويب «كشف حساب العميل»).

### المعاملات (Query Params)

| المعامل | الوصف |
|---|---|
| `from` | تاريخ بداية الفترة (اختياري) |
| `to` | تاريخ نهاية الفترة (اختياري، يُضبط على نهاية اليوم تلقائياً) |

### ما يعرضه الكشف

| الحقل | المعنى |
|---|---|
| `customer` | بيانات العميل (الاسم، الهاتف، العنوان) |
| `openingBalance` | رصيد الافتتاح = (مبيعات − مرتجعات) قبل تاريخ `from` |
| `transactions[]` | قائمة الحركات مرتبة تصاعدياً بالتاريخ |
| `closingBalance` | الرصيد الختامي بعد كل الحركات |

### بنية كل حركة (Transaction)

| الحقل | الوصف |
|---|---|
| `date` | تاريخ الفاتورة |
| `type` | `SALE` (بيع مدين) أو `REFUND` (مرتجع دائن) |
| `referenceNo` | رقم الفاتورة (أو `رقم-R` للمرتجع) |
| `debit` | مبلغ البيع (موجب) |
| `credit` | مبلغ المرتجع |
| `balance` | الرصيد الجاري بعد الحركة |

> المنطق: البيع = مدين (debit)، المرتجع = دائن (credit). الرصيد يُحتسب تراكمياً: `balance = balance + debit − credit`.

## 7. تقرير الولاء (Loyalty Report)

الملف: `src/app/api/customers/loyalty-report/route.ts`.

يعرض كبار العملاء حسب رصيد النقاط + إحصائية عامة.

### المعاملات

| المعامل | الافتراضي | الوصف |
|---|---|---|
| `limit` | 20 | عدد العملاء المُعادة (الحد الأقصى 100) |
| `tier` | بدون فلتر | BRONZE / SILVER / GOLD (إن لم يُذكر، يُعرض كل من له نقاط > 0) |

### مخرجات التقرير

```json
{
  "items": [ /* قائمة العملاء بـ: id, name, phone, loyaltyPoints, loyaltyTier, totalSales */ ],
  "summary": {
    "totalCustomers": 150,
    "totalPointsInCirculation": 48250,
    "tierBreakdown": [
      { "tier": "GOLD",   "count": 3,   "points": 32000 },
      { "tier": "SILVER", "count": 8,   "points": 41000 },
      { "tier": "BRONZE", "count": 35,  "points": 28000 }
    ]
  }
}
```

### الصلاحيات

`OWNER / ADMIN / SALES` فقط (دور المحاسب والكيشان وال摩زن غير مصرّح).

> ملاحظة: الـ API موجود ويُرجِع بيانات صحيحة، لكن **لم يظهر في الكود** أي شاشة واجهة منفصلة لعرض هذا التقرير. لتشغيله يدوياً، اطلب `GET /api/customers/loyalty-report?limit=20` من مدير API أو أضِف تبويباً مستقبلاً في شاشة المحاسبة أو التقارير.

## 8. ربط العملاء بفواتير البيع

تُربط الفاتورة بالعميل عبر حقل `customerId` في جدول `Sale`. الربط يحدث في ملف `src/lib/sale/customer-resolver.ts` أثناء معالجة فاتورة البيع.

### ثلاثة سيناريوهات للربط

| السيناريو | الشرط | النتيجة |
|---|---|---|
| **عميل مسجّل مسبقاً** | الطلب يحوي هاتفاً موجوداً في `Customer.phone` | يُربط بـ `customerId` الموجود، ويُحدَّث العنوان إن كان فارغاً |
| **عميل جديد** | الطلب يحوي هاتفاً غير موجود | يُنشأ عميل جديد، يُربط بالفاتورة |
| **عميل نقدي (Walk-in)** | الطلب بلا هاتف | `customerId = undefined`، وفئة السعر = `RETAIL` افتراضياً |

### مفهوم «العميل النقدي» (Walk-in)

عميل لم يُسجّل في قاعدة البيانات، يُدخل الكاشير اسمه ورقم هاتفه فقط في نموذج الفاتورة دون إنشاء سجل `Customer`. يُستخدم هذا النمط للبيعات السريعة اليومية.

| الحقل | السلوك في البيع النقدي |
|---|---|
| `customerName` | نص حر يُحفظ في الفاتورة مباشرة (غير مرتبط بـ Customer) |
| `phone` | إن أُدخل وكان غير مسجّل، يُنشأ عميل جديد تلقائياً |
| `customerId` | `undefined` (إن لم يُدخل هاتف) |

> عند إنشاء عميل جديد تلقائياً من الفاتورة، يُستخدم الاسم المُدخل أو «عميل نقدي» إذا كان فارغاً (انظر `customer-resolver.ts`).

### أثر نوع العميل على الفاتورة

عند ربط العميل المسجّل، تُقرأ فئته (`type`) وتُمرَّر كـ `customerTier` لدالة `computeSaleTotals` لتختار سعر البيع المناسب (انظر القسم 3).

## 11. خلاصة تدريبية

1. **الشاشة الأساسية** للعملاء بسيطة: قائمة جدولية + بحث + إضافة/تعديل/حذف.
2. **نوع العميل** (RETAIL/WHOLESALE/CORPORATE) يتحكم في سعر البيع، لكن **لا يمكن تغييره من النموذج الظاهر حالياً** — يُفترض RETAIL لكل عميل جديد.
3. **النقاط تُكتسب تلقائياً** بمعدل نقطة لكل وحدة عملة من صافي الفاتورة، وتُحدَّث الفئة (BRONZE/SILVER/GOLD) تبعاً لها.
4. **كشف الحساب** يُعرض في وحدة المحاسبة، تبويب «كشف حساب العميل»، مع رصيد افتتاحي وختامي وحركات تراكمية.
5. **تقرير الولاء** متاح عبر API لكن بدون شاشة عرض مستقلة في الكود الحالي.
6. **العميل النقدي** (Walk-in) = بيع بدون سجل Customer، يُحفظ الاسم والهاتف في الفاتورة فقط.
