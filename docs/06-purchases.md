# 06 — المشتريات: الموردون، أوامر الشراء، الفواتير، المرتجعات

> دليل تدريبي مبسّط لوحدة المشتريات في نظام KWPOS. يغطي شاشة الموردين وأنواعهم (محلي/أجنبي)، أمر الشراء من الإنشاء حتى الاستلام مع الموافقة الإدارية، فاتورة المشتريات وتأثير ترحيلها (POST) على المخزون والتكلفة والقيود، ثم مرتجع المشتريات وأثره على المخزون وذمم الموردين. مكتوب لغير المختصين تقنياً مع الإشارة للملفات المصدرية.

الملف المسؤول العام: `src/components/purchases/purchases-view.tsx` (يستضيف 3 تبويبات: أوامر الشراء، الفواتير، سداد الموردين — وشاشة الموردين منفصلة في `suppliers-view.tsx`).

## 10.1 الموردون (Suppliers)

الملفات: `src/components/purchases/suppliers-view.tsx` و`supplier-form-dialog.tsx`، و`src/app/api/suppliers/route.ts` و`[id]/route.ts` و`[id]/statement/route.ts`.

### شاشة الموردين

تعرض الموردين على شكل بطاقات شبكية (3 أعمدة). كل بطاقة تحتوي:

| العنصر | الشرح |
|---|---|
| الاسم + شارة النوع | شارة «أجنبي» أو «محلي» |
| جهة الاتصال | نص حر (اختياري) |
| الهاتف / البريد / العنوان | مع أيقونات |
| عدد المنتجات | `productsCount` |
| عدد الطلبات | `ordersCount` |
| **الرصيد المستحق** | `balance` (موجب = متجر مدين للمورد، أحمر؛ صفر/سالب = أخضر) |
| زر «سداد مورد» | يفتح نافذة دفع جديدة |

### نموذج إضافة/تعديل مورد

| الحقل | النوع | الشرح |
|---|---|---|
| **اسم المورد** | نص (إلزامي) | |
| **نوع المورد** | Select: LOCAL / FOREIGN (إلزامي) | |
| جهة الاتصال | نص | اسم الشخص المسؤول |
| الهاتف | نص LTR | |
| البريد | email | |
| العنوان | نص | |

### نوع المورد وأثره على الفاتورة

| النوع | تأثيره في فاتورة المشتريات |
|---|---|
| **محلي (LOCAL)** | تُخفى حقول الجمارك والشحن والمصاريف الأخرى — يظهر فقط: الضريبة + الخصم |
| **أجنبي (FOREIGN)** | تظهر حقول التكلفة الكاملة: جمارك + شحن + مصاريف أخرى + الضريبة + الخصم |

عند تغيير نوع المورد من أجنبي إلى محلي في نموذج الفاتورة، يُصفِّر النظام حقول الرسوم تلقائياً (لأنها مخفية).

### الإجراءات

| الإجراء | المسار | الصلاحية |
|---|---|---|
| إضافة | `POST /api/suppliers` | OWNER/ADMIN/WAREHOUSE |
| تعديل | `PUT /api/suppliers/[id]` | OWNER/ADMIN/WAREHOUSE |
| حذف | `DELETE /api/suppliers/[id]` | `canDelete` (OWNER/ADMIN/MANAGER) |
| سداد | يفتح `SupplierPaymentDialog` | OWNER/ADMIN/WAREHOUSE |
| كشف حساب | يفتح `SupplierStatementDialog` | كل من يرى المورد |

> **حذف المورد يُرفض** إن كان مرتبطاً بمنتجات أو أوامر شراء (`cannot-delete-linked`، 409).

### كشف حساب المورد (Supplier Statement)

الملف: `src/app/api/suppliers/[id]/statement/route.ts`.

يعرض كشف حساب موحد يضم:
- **رصيد افتتاحي** = فواتير مدفوعة − سداد − مرتجعات (قبل تاريخ `from`).
- **معاملات الفترة** (ممرتبة زمنياً):
  - فاتورة مشتريات POSTED → مدين (`debit`) بقيمة الفاتورة.
  - سداد مورد → دائن (`credit`) بقيمة السداد.
  - مرتجع مشتريات → دائن (`credit`) بقيمة المرتجع.
- **رصيد جاري** لكل سطر = الرصيد السابق + المدين − الدائن.
- **رصيد ختامي** + ملخصات (إجمالي الفواتير، إجمالي السداد، إجمالي المرتجعات).

> الرصيد الموجب يعني المتجر مدين للمورد (طبيعي للمشتريات الآجلة).

يدعم الكشف: فلترة تاريخ (من/إلى)، تصدير (CSV/Excel)، طباعة.

## 10.2 أوامر الشراء (Purchase Orders)

الملفات: `src/components/purchases/purchases-view.tsx` (تبويب `orders`)، `purchase-order-dialog.tsx`، `po-approval-panel.tsx`، و`src/app/api/purchase-orders/`.

### إنشاء أمر شراء

انقر زر «أمر شراء جديد» → تفتح نافذة `PurchaseOrderDialog`:

| الحقل | النوع | الشرح |
|---|---|---|
| **المورد** | Combobox (إلزامي) | |
| **ملاحظة** | نص | |
| **سطور الأصناف** | جدول متعدد | لكل سطر: منتج (Combobox مع فلتر فئة)، كمية، تكلفة الوحدة، الإجمالي، سعر بيع مقترح (اختياري) |
| **التكلفة الكاملة** | قسم قابل للطي | جمارك + شحن + مصاريف أخرى (Landed Cost) |
| **الإجمالي الكلي** | محسوب | مجموع السطور + رسوم التكلفة الكاملة |

> عند اختيار صنف، يُملأ حقل `unitCost` بسعر تكلفة الصنف الحالي، وحقل `suggestedSalePrice` بسعر البيع الحالي (ليعدّله المدير إن أراد).

### الحقول المرسلة للخادوم

```json
{
  "supplierId": "...",
  "note": "...",
  "items": [{ "productId": "...", "quantity": 10, "unitCost": 5.0, "suggestedSalePrice": 12.0 }],
  "customsAmount": 0, "shippingAmount": 0, "otherCharges": 0
}
```

`POST /api/purchase-orders` ينشئ الأمر بحالة **`PENDING`** (بانتظار الاستلام).

### المسودة التلقائية (Auto-Draft)

زر «جلب الأصناف المطلوبة من المورد» (Sparkles) يفتح نافذة لاختيار مورد، ثم يستدعي `POST /api/purchase-orders/auto-draft`:

- يفحص كل المنتجات التي `defaultSupplierId === supplierId` و`quantity <= reorderLevel`.
- لكل صنف منخفض:
  - الكمية = `optimalOrderQty` (إن > 0).
  - وإلا: `reorderLevel × 2 − quantity` (مع ضمان ≥ 1).
- التكلفة = `costPrice` الحالية للصنف.
- يُنشأ أمر شراء بحالة **`PENDING_APPROVAL`** (بانتظار موافقة الإدارة).
- إن لم توجد أصناف منخفضة، يُعاد `{ message: "no-items-needed" }`.

### الموافقة الإدارية (للأوامر التلقائية)

للأدوار ADMIN فقط، تظهر لوحة `PoApprovalPanel` تعرض الأوامر بحالة `PENDING_APPROVAL`. لكل أمر:

| الزر | الوظيفة |
|---|---|
| **فتح** | فتح نافذة المحرر لمراجعة السطور |
| **رفض** | فتح نافذة سبب الرفض (نص إلزامي) |
| **تعديل وقبول** | تعديل الكميات/التكلفة لكل سطر ثم اعتماد |
| **اعتماد وقبول** | اعتماد كما هو |

يُحدَّث الأمر إلى `APPROVED` عبر `PATCH /api/purchase-orders/[id]` مع `status: "APPROVED"`. الرفض يتطلب `rejectionReason`.

### حالات أمر الشراء

| الحالة | المعنى | الشارة |
|---|---|---|
| `PENDING_APPROVAL` | مسودة تلقائية بانتظار الإدارة | كهرمانية |
| `APPROVED` | تم اعتماده (يدوي أو تلقائي) | خضراء |
| `PENDING` | أمر يدوي بانتظار الاستلام | كهرمانية |
| `RECEIVED` | تم استلام البضاعة | خضراء |
| `CANCELLED` | أمر ملغي | حمراء باهتة |
| `REJECTED` | رفضته الإدارة (مع سبب) | حمراء باهتة |

### الإجراءات المتاحة حسب الحالة (قائمة MoreVertical)

| الحالة | الإجراءات المتاحة |
|---|---|
| `PENDING_APPROVAL` | (تُدار من لوحة الموافقة) — فتح/تعديل وقبول/اعتماد/رفض |
| `PENDING` أو `APPROVED` | **تأكيد الاستلام**، **استلام وإنشاء فاتورة**، **إلغاء الأمر** (PENDING فقط)، حذف |
| `RECEIVED` | **مرتجع مشتريات جديد**، عرض التفاصيل |
| `CANCELLED` / `REJECTED` | عرض التفاصيل فقط |

### الاستلام (Receive)

عن طريق زر «تأكيد الاستلام» في القائمة المنسدلة، يُستدعى `POST /api/purchase-orders/[id]/receive`. تأكيد الاستلام يقوم بـ:

1. التحقق أن الحالة ليست `RECEIVED` (`already-received`).
2. قراءة المنتجات قبل الاستلام (لقطة للأسعار والكميات).
3. حساب **التكلفة المتوسطة المرجَّحة** عبر `allocateLandedCost` (عند وجود رسوم جمارك/شحن).
4. داخل معاملة:
   - `incrementStockItem` لكل صنف في المخزن (PO.warehouseId أو المخزن الافتراضي).
   - تحديث `Product.costPrice` بالتكلفة الجديدة.
   - تطبيق `suggestedSalePrice` عبر محرك التسعير (يكتب سجل `PriceChange` + يحدّث `Product.salePrice`) إن اختلف عن السعر الحالي.
   - حساب `receivedTaxAmount` (مجموع `subtotal × taxRate/100`).
   - تحديث الأمر: `status=RECEIVED` + قيم الجمارك/الشحن/المصاريف + `landedCostApplied`.
   - **قيد محاسبي:** مدين 1010 (المخزون) / دائن 2010 (ذمم دائنة) بقيمة `po.total`.
   - سجل تدقيق `PO_RECEIVED`.

> **تنبيه واجهة:** إن كان للأمر رسوم تكلفة كاملة (customs/shipping/other)، تظهر نافذة تأكيد تحذيرية تُنبّه أن أسعار التكلفة ستُحدَّث بالمتوسط المرجَّح.

### الاستلام وإنشاء فاتورة دفعة واحدة

زر «استلام وإنشاء فاتورة» يفتح `PurchaseInvoiceDialog` محمَّلاً مسبقاً بـ:
- `purchaseOrderId`، `supplierId`
- سطور الأصناف من الأمر (نفس الكميات والتكلفة + `purchaseOrderItemId`)

يُكمل المستخدم باقي بيانات الفاتورة (مخزن، تاريخ، رسوم، خصم، ضريبة، طريقة دفع) ثم يحفظ كمسودة أو يرحّل.

### حذف أمر شراء

- `DELETE /api/purchase-orders/[id]` — متاح لـ `canDelete` (OWNER/ADMIN/MANAGER).
- **لا يمكن حذف أمر مستلم** (`cannot-delete-received`، 409).

## 10.3 فواتير المشتريات (Purchase Invoices)

الملفات: `src/components/purchases/purchase-invoices-view.tsx` و`purchase-invoice-dialog.tsx`، و`src/app/api/purchase-invoices/route.ts` و`[id]/route.ts` و`[id]/post/route.ts`.

### شاشة الفواتير

جدول يعرض: رقم الفاتورة (`PINV-NNNN`)، المورد، التاريخ، عدد السطور، الإجمالي، الحالة (DRAFT/POSTED/CANCELLED)، إجراءات.

### نموذج إنشاء فاتورة

| الحقل | النوع | الشرح |
|---|---|---|
| **المورد** | Combobox (إلزامي) | يحدد نوع الفاتورة (محلي/أجنبي) |
| **المخزن** | Select | يحدد المخزن المستلم (يُخزَّن في الفاتورة) |
| **أمر الشراء** | Combobox اختياري | يُربط بأمر APPROVED أو PENDING + زر «استيراد من أمر الشراء» |
| **التاريخ** | date | افتراضياً اليوم |
| **الأصناف** | جدول متعدد | منتج + كمية + تكلفة + إجمالي |
| **رسوم التكلفة الكاملة** | قسم قابل للطي (للأجانب فقط) | شحن + جمارك + مصاريف أخرى |
| **طريقة الدفع** | Select: CASH / BANK / CREDIT | تحدد الحساب الدائن في القيد |
| **نسبة الضريبة** | number | تُحسب من المجموع الفرعي |
| **الخصم** | number | يُطرح من الإجمالي |
| **ملاحظة** | textarea | |

### الأزرار في النموذج

| الزر | الوظيفة |
|---|---|
| **إلغاء** | إغلاق النافذة |
| **حفظ كمسودة** | `post: false` → حالة `DRAFT` (لا أثر على المخزون/المحاسبة) |
| **حفظ وترحيل** | `post: true` → حالة `POSTED` (يُحدّث المخزون + القيد) — يتطلب تأكيد نافذة تحذيرية |

### حسابات الإجمالي

```
subtotal    = Σ (quantity × unitCost)
taxAmount   = subtotal × taxRate / 100
extraTotal  = shipping + customs + otherCharges
total       = subtotal + taxAmount + extraTotal − discount
```

### ماذا يحدث عند الترحيل (POST)؟

ملف `src/app/api/purchase-invoices/[id]/post/route.ts` (والمسار المماثل في `POST /api/purchase-invoices` مع `post: true`):

1. التحقق أن الفاتورة `DRAFT` (لا يمكن ترحيل `POSTED` أو `CANCELLED`).
2. داخل معاملة واحدة:
   - لكل صنف: التحقق أنه ليس تحت جرد مفعّل (`stock-frozen:{productId}`) — إن وُجد سجل `StockTakeItem` بـ `stockTake.status = DRAFT` يُمنع الترحيل.
   - `upsert StockItem` بـ `increment quantity` في المخزن المحدد.
   - تحديث `Product.costPrice` بـ `unitCost` من سطر الفاتورة.
   - إن وُجد `purchaseOrderId`: تحديث الأمر إلى `RECEIVED`.
   - تحديث الفاتورة إلى `status=POSTED`.
   - سجل تدقيق `PURCHASE_INVOICE_POSTED`.
3. بعد المعاملة (post-commit):
   - إعادة حساب `Product.quantity` لكل الأصناف.
   - **قيد محاسبي** عبر `createPurchaseInvoiceJournalEntry`:
     - مدين 1010 (المخزون) بقيمة (subtotal + رسوم التكلفة الكاملة − الخصم).
     - مدين 2110 (ضريبة مدخلات) بقيمة `taxAmount`.
     - دائن 2010 (ذمم دائنة) أو 1010/1020 حسب `paymentMethod`:
       - `CASH` → دائن 1010 (الصندوق).
       - `BANK` → دائن 1020 (البنك).
       - `CREDIT` → دائن 2010 (ذمم دائنة للمورد).

> فشل القيد المحاسبي بعد الترحيل لا يُلغي الترحيل (يُسجَّل خطأ في السجلات لكن الفاتورة تبقى POSTED).

### الإجراءات في شاشة الفواتير

| الإجراء | المسار | الصلاحية | الحالة المسموحة |
|---|---|---|---|
| **عرض التفاصيل** | — | الجميع | كل الحالات |
| **ترحيل** | `POST /api/purchase-invoices/[id]/post` | ADMIN/WAREHOUSE | `DRAFT` فقط |
| **حذف** | `DELETE /api/purchase-invoices/[id]` | **ADMIN فقط** | `DRAFT` فقط (`cannot-delete-posted`، 409) |
| **تعديل** | `PUT /api/purchase-invoices/[id]` | OWNER/ADMIN | `DRAFT` فقط (`cannot-edit-posted`، 409) |

> لا يمكن حذف أو تعديل فاتورة مرحَّلة. الأثر المحاسبي والمخزوني للفاتورة المرحَّلة دائم. لتصحيح خطأ في فاتورة مرحَّلة: عمل مرتجع مشتريات ثم إنشاء فاتورة صحيحة.

### تأثير الترحيل على المورد

- يُضاف إجمالي الفاتورة (`total`) لرصيد المورد المستحق (لأن `paymentMethod = CREDIT` يعني آجلاً).
- إن كان الدفع نقداً أو بنكياً، تُضاف القيمة أيضاً للرصيد ثم تُصفَّر فوراً عبر قيد الدفع، فيبقى الرصيد صفراً (إلا إن كانت البضاعة فعلية والمبلغ لم يُدفع فعلياً، فيظهر الرصيد مديناً للمورد حتى يتم سداد فعلي).

## 10.4 مرتجع المشتريات (Purchase Returns)

الملفات: `src/components/purchases/purchase-return-dialog.tsx` و`src/app/api/purchase-returns/route.ts`.

### متى يُستخدم؟

عند إرجاع بضاعة لمورد بعد استلامها. أمثلة:
- بضاعة تالفة أو منتهية الصلاحية.
- خطأ في الطلب (استلام كمية أكبر من المطلوب).
- استبدال صنف بآخر.

### الشرط الأساسي

- يجب أن يكون أمر الشراء الأصلي بحالة **`RECEIVED`** (`po-not-received`، 400).
- يفتح النظام نافذة المرتجع من القائمة المنسدلة لأمر مستلم فقط.

### خطوات المرتجع

| الخطوة | الإجراء |
|---|---|
| 1 | فتح أمر الشراء المستلم ← اختيار «مرتجع مشتريات جديد». |
| 2 | تعرض النافذة كل سطور الأمر بكمياتها الأصلية + المُرتجَعة سابقاً + القابل للإرجاع. |
| 3 | إدخال كمية الإرجاع لكل سطر (لا يمكن تجاوز القابل للإرجاع: `exceeds-returnable:{name}:{qty}`). |
| 4 | (اختياري) بحث بالباركود/الاسم لتحديد سطر بسرعة. |
| 5 | الضغط على «اعتماد المرتجع». |
| 6 | تظهر شاشة نجاح فيها رقم المرتجع (`PR-NNNNN`) والإجمالي. |

### ماذا يحدث على الخادوم؟

ملف `src/app/api/purchase-returns/route.ts`:

1. التحقق من أمر الشراء وحالته `RECEIVED`.
2. التحقق أن كل كمية إرجاع ≤ `quantity − returnedQty` لسطر الأمر.
3. حل مخزن الإرجاع: `po.warehouseId` أو المخزن الافتراضي.
4. داخل معاملة:
   - لكل سطر إرجاع:
     - `decrementStockItem` (يرفض إن لم يكن مخزون كافٍ: `stock-insufficient:...`).
     - إعادة حساب `Product.quantity`.
     - `increment returnedQty` على سطر أمر الشراء.
   - إنشاء سجل `PurchaseReturn` برقم `PR-NNNNN` + `status=APPROVED` + سطور المرتجع.
   - **قيد محاسبي عكسي** عبر `safeCreateJournalEntry` (غير فادح — فشله لا يلغي المرتجع):
     - مدين 2010 (ذمم دائنة — يقل ما نستحقه للمورد).
     - دائن 1010 (المخزون — يقل الأصل).
   - سجل تدقيق `PURCHASE_RETURN_CREATED`.

### الأثر على المخزون وذمم الموردين

| الجانب | الأثر |
|---|---|
| **المخزون** | يُخصم من `StockItem` في المخزن المُحدَّد، ويُعاد حساب `Product.quantity`. |
| **سطر أمر الشراء** | `returnedQty` يُزاد (يمنع إرجاع مكرر). |
| **رصيد المورد** | يُقلَّص بقيمة المرتجع (يظهر كـ «credit» في كشف الحساب). |
| **القيد المحاسبي** | عكسي: مدين 2010 / دائن 1010. |
| **رقم المرتجع** | `PR-NNNNN` يُولَّد تلقائياً. |

> **ملاحظة:** لا يمكن حذف مرتجع بعد إنشائه. لتصحيحه: عمل مرتجع عكسي غير ممكن مباشرة؛ الحل العملي إنشاء فاتورة شراء جديدة بالكميات الصحيحة.

## خلاصة

- الموردون نوعان: محلي (بدون جمارك/شحن) وأجنبي (تكلفة كاملة)، ويؤثر النوع على حقول فاتورة المشتريات.
- أمر الشراء يمر بـ: `PENDING_APPROVAL` (للمسودات التلقائية) → `APPROVED` → `RECEIVED`. الحذف متاح فقط قبل الاستلام.
- الاستلام يُحدّث المخزون + التكلفة (متوسط مرجَّح للتكلفة الكاملة) + سعر البيع المقترح + قيد محاسبي (1010/2010).
- فاتورة المشتريات لها حالتان أساسيتان: `DRAFT` (لا أثر) و`POSTED` (تُحدّث المخزون والتكلفة والمورد والمحاسبة). لا يمكن تعديل/حذف فاتورة مرحَّلة.
- ترحيل الفاتورة يُنشئ قيداً محاسبياً متوازناً: مدين 1010 + 2110 / دائن 2010 أو 1010/1020 حسب طريقة الدفع.
- مرتجع المشتريات متاح فقط لأمر مستلم، يُخصم المخزون ويرفع `returnedQty` على سطر الأمر، ويُنشئ قيداً عكسياً (مدين 2010 / دائن 1010).
- كشف حساب المورد يوحد الفواتير والسدادات والمرتجعات في مجرى واحد مع رصيد جاري.

الخطوة التالية في التوثيق: سداد الموردين وكشف الحساب التفصيلي في `07-suppliers-payments.md`.
