# 03 — شاشة نقاط البيع (POS)

> دليل تدريبي مبسّط لشاشة نقاط البيع في نظام KWPOS: عناصرها، أزرارها، دورة البيع الكاملة، الأخطاء المحتملة، الوضع السريع (Express)، والفواتير المعلَّقة (Parked). مكتوب لغير المختصين تقنياً، مع إشارة إلى الملفات المصدرية في الكود.

---

## 4.1 الغرض من شاشة نقاط البيع

شاشة نقاط البيع (Point of Sale) هي الواجهة اليومية للكاشير وبائع المتجر، وهي **أكثر الشاشات استعمالاً في النظام**. هدفها الأساسي: إنجاز عملية بيع كاملة من اختيار المنتج وحتى طباعة الإيصال في أقل عدد من النقرات.

تتولّى هذه الشاشة:

- عرض المنتجات المتاحة (مع صورها وأسعارها وكمياتها).
- البحث بالاسم أو مسح الباركود.
- بناء سلة شراء (Cart) مع تعديل الكميات والإزالة.
- اختيار العميل وربطه برقم الهاتف (بحث تلقائي).
- اختيار طريقة الدفع (نقدي / بطاقة / تحويل).
- تطبيق الخصم (لمن يملك الصلاحية).
- إضافة خدمة التوصيل (السائق + الرسوم + العنوان).
- اعتماد البيع بصفة نهائية بعد نافذة تأكيد.
- توليد رقم فاتورة تلقائياً وخصم المخزون لحظياً وإنشاء قيد محاسبي.
- طباعة الإيصال الحراري (80mm).

الملفات الرئيسية للشاشة:

| الملف | دوره |
|---|---|
| `src/components/sales/sales-view.tsx` | الواجهة القياسية (Standard) |
| `src/components/sales/express-pos-view.tsx` | الواجهة السريعة (Express) |
| `src/hooks/use-pos.ts` | إدارة حالة السلة والمنطق المشترك |
| `src/hooks/pos-helpers.ts | دوال مساعدة (لقطات السلة، حساب الإجماليات، أخطاء البيع) |
| `src/lib/sale/transaction.ts` | تنفيذ البيع فعلياً في قاعدة البيانات |
| `src/lib/sale/totals.ts` | حساب المجاميع (الخادوم يُعيد التحقق) |
| `src/lib/sale/decrement-planner.ts` | تخطيط خصم المخزون بين المخازن |
| `src/lib/sale/side-effects.ts` | نقاط الولاء بعد البيع |
| `src/app/api/sales/route.ts` | مسار POST/GET للمبيعات |

## 4.2 عناصر الشاشة

شاشة نقاط البيع تنقسم إلى عمودين رئيسيين:

| المنطقة | الموقع | المحتوى |
|---|---|---|
| **بحث المنتج** | أعلى يمين | حقل بحث بالاسم أو الباركود (Auto-focus) |
| **شجرة الفئات** | تحت البحث | تصفّح هرمي للفئات (`CategoryTree`) أو أزرار مسطحة |
| **شبكة المنتجات** | يسار الواجهة القياسية | بطاقات بصور وأسعار وكميات متاحة |
| **بطاقات الباقات** | فوق الشبكة | باقات نشطة (Bundles) عند عدم وجود بحث |
| **رأس السلة** | يمين | عنوان «السلة» + عدد الأصناف + زر المعلَّقات + زر التعليق + زر المسح |
| **بيانات العميل** | أعلى السلة | الاسم + الهاتف (بحث تلقائي) + الطبقة (RETAIL/WHOLESALE/CORPORATE) |
| **طريقة الدفع** | أعلى السلة | قائمة منسدلة: CASH / CARD / TRANSFER |
| **عناصر السلة** | وسط السلة | صف بصف: حذف، صورة، اسم، −/+(كمية)، إجمالي السطر |
| **ترقيم صفحات السلة** | تحت العناصر | عند تجاوز 10 أصناف يظهر «السابق/التالي» |
| **تبديل التوصيل** | أسفل السلة | مربع اختيار + حقول: العنوان، السائق، رسوم التوصيل |
| **حقل الخصم** | أسفل التوصيل | يظهر فقط لـ ADMIN/MANAGER/OWNER |
| **الملخّص** | أسفل | المجموع الفرعي، الخصم، الضريبة، التوصيل، الإجمالي |
| **زر الإتمام** | أسفل | «إتمام البيع بقيمة X» (يفتح نافذة التأكيد) |
| **شريط العلوي** | فوق العنوان | زر «Express» + Toggle «الطباعة التلقائية» |
| **نافذة التأكيد** | منبثقة | ملخّص البيع قبل الاعتماد + زر أخضر (اعتماد) + زر أحمر (إلغاء) |
| **نافذة الإيصال** | منبثقة | بعد نجاح البيع: رقم الفاتورة، الأصناف، الإجمالي، زر طباعة + زر «بيع جديد» |

> **ملاحظة:** لا يوجد حقل لإدخال نسبة الضريبة يدوياً في السلة، لأن الضريبة محسوبة على مستوى كل منتج (انظر `src/lib/sale/totals.ts`).

## 4.4 دورة البيع الكاملة (10 خطوات)

| الخطوة | الإجراء | المسؤول (الملف) |
|---|---|---|
| **1. اختيار المنتج** | النقر على بطاقة المنتج أو مسح الباركود | `sales-view.tsx` / `express-pos-view.tsx` |
| **2. تحديد الكمية** | ضغط «+» و«−» لكل سطر | `use-pos.ts` (دالة `changeQty`) |
| **3. ربط العميل** | إدخال رقم الهاتف → بحث تلقائي → اختيار الطبقة | `use-customer-lookup.ts` |
| **4. اختيار طريقة الدفع** | CASH أو CARD أو TRANSFER | `use-pos.ts` |
| **5. تطبيق الخصم (اختياري)** | إدخال مبلغ الخصم (للأدوار المصرّحة) | `sales-view.tsx` |
| **6. تأكيد البيع** | زر «إتمام البيع» ثم زر «اعتماد» الأخضر | `sale-confirm-dialog.tsx` |
| **7. خصم المخزون** | تنفيذ `UPDATE StockItem` دفعة واحدة عبر مخازن متعددة | `src/lib/sale/transaction.ts` و `decrement-planner.ts` |
| **8. إنشاء الفاتورة** | توليد `invoiceNo` من تسلسل PostgreSQL، ثم `Sale.create` مع عناصرها | `transaction.ts` |
| **9. إنشاء القيد المحاسبي** | مدين: حساب الدفع (1010/1020)، دائن: إيراد المبيعات (4010) + الضريبة المستحقة (2110) | `transaction.ts` داخل `createJournalEntry` |
| **10. الطباعة** | طباعة إيصال حراري 80mm تلقائياً أو يدوياً | `src/lib/print.ts` (دالة `printThermalReceipt`) |

بعد هذه الخطوات تأتي خطوات جانبية (Side Effects):

- مزامنة `Product.quantity` مع مجموع `StockItem` (استعلام واحد).
- تسجيل نقاط الولاء للعميل (`src/lib/sale/side-effects.ts`).
- كتابة سطر تدقيق `SALE_CREATED` في `AuditLog`.

> **ملاحظة هامة من الكود:** لا يستخدم النظام `Prisma.$transaction` (المعاملة التفاعلية) في تنفيذ البيع، لتجنّب خطأ «Transaction not found» على Supabase/PgBouncer. بدلاً من ذلك يستخدم نمط **Saga**: كل خصم مخزون atomic بحد ذاته، وفي حال فشل إنشاء الفاتورة بعد الخصم تتم **التعويض** (Compensation) بإعادة الكميات. راجع التعليقات في `src/lib/sale/transaction.ts`.

## 4.5 الأخطاء المحتملة

| الخطأ | الرسالة (مختصرة) | السبب | ما يحدث للمستخدم |
|---|---|---|---|
| **نفاد المخزون** | `stock-insufficient:...` | الكمية المطلوبة أكبر من المتاح | رسالة `toast` حمراء باسم المنتج، لا يُحفظ البيع |
| **لا يوجد مخزون افتراضي** | `no-warehouse-available` | المستخدم بدون مخزن مرتبط ولا يوجد مخزن افتراضي | رسالة خطأ 400، يجب على المدير ربط المخزن |
| **الخصم غير مصرّح به** | `discount-requires-manager` | SALES/CASHIER خصم أكثر من 10% | رسالة 403 توضّح الحد الأقصى المسموح |
| **الخصم السالب** | `discount-cannot-be-negative` | إدخال خصم بالسالب | رسالة 400 (يجب أن يكون ≥ 0) |
| **انتهاء الجلسة** | `session-expired` | المستخدم محذوف أو الـ JWT قديم بعد إعادة تهيئة | رسالة تنبيه ثم تسجيل خروج تلقائي |
| **الصنف مجمَّد للجرد** | `stock-frozen:...` | المنتج تحت جرد أعمى غير معتمد | رسالة توضّح أن المنتج لا يُباع حتى اعتماد الجرد |
| **فشل القيد المحاسبي** | يُسجَّل في الكونسول فقط | مشكلة في إنشاء القيد | البيع ينجح؛ تظهر فجوة محاسبية يجب تسويتها يدوياً |
| **فشل الطباعة** | «يرجى السماح بالنوافذ المنبثقة» | المتصفح يحجب النوافذ المنبثقة | تنبيه؛ يمكن تكرار الطباعة من شاشة الفواتير |
| **فشل اتصال قاعدة البيانات** | خطأ عام 500 | انقطاع شبكة أو توقف قاعدة البيانات | رسالة `checkout-failed` مع وصف الخطأ |
| **الهاتف مطلوب** | `posPhoneRequired` | محاولة إتمام البيع دون رقم هاتف | رسالة تحذيرية، لا تُفتح نافذة التأكيد |
| **العنوان مطلوب للتوصيل** | `posAddressRequired` | تفعيل التوصيل دون عنوان | رسالة تحذيرية، يبقى زر الإتمام معطّلاً |

> يتم التعامل مع الأخطاء عبر جدول مطابقة في `src/hooks/pos-helpers.ts` (دالة `handleSaleError`). كل نوع خطأ له معالج خاص يُظهر الرسالة المناسبة، والخطأ غير المعروف يُمرَّر إلى معالج عام.

## الوضع السريع (Express Mode)

### ما هو؟

الوضع السريع هو **واجهة مبسَّطة لكاشير المتاجر السريعة** (محلات العطور، البقالة، الكافيهات). يحذف كل ما لا يلزم لبيع سريع (شجرة الفئات، شريط التنقل، تعدد طبقات الأسعار) ويُبقي فقط:

- حقل باركود كبير (ارتفاع 14) مُركَّز دائماً.
- شبكة منتجات أكبر صوراً.
- زرَّين كبيرين: «نقدي» (أخضر) و«بطاقة» (أزرق).
- قابس لخيارات متقدمة (اسم العميل، خصم، توصيل) مطوي افتراضياً.
- فتح درج النقدية (إن وُجد جهاز متصل).

### كيف يُفعَّل؟

- زر «Express» في أعلى الشاشة القياسية.
- يُحفظ التفضيل في `user.posExpressMode` عبر `PATCH /api/users/me/preferences`.
- عند الدخول التالي يُفتح النظام مباشرة على الوضع المختار.

### من يستخدمه؟

- **الكاشير في أوقات الذروة** (طوابير سريعة).
- الموظفون المعتادون على الباركود فقط.
- أي مستخدم له حق الدخول لشاشة POS (OWNER/ADMIN/MANAGER/SALES/CASHIER).

### الفروقات الجوهرية بين الوضعين

| الجانب | الوضع القياسي (Standard) | الوضع السريع (Express) |
|---|---|---|
| شجرة الفئات | ظاهرة | غير ظاهرة |
| طبقات الأسعار (Tier) | قابلة للتبديل (RETAIL/WHOLESALE/CORPORATE) | مُجمَّدة على RETAIL |
| حقل البحث | عادي (ارتفاع 11) | باركود كبير (ارتفاع 14) مع تركيز تلقائي |
| شريط علوي | عنوان الصفحة + Toggle للطباعة | اسم المتجر + الخروج + فتح الدرج + التبديل للقياسي |
| طريقة الدفع | قائمة منسدلة | زرَّان كبيران ملونان |
| خيارات العميل/الخصم/التوصيل | ظاهرة دائماً | مطوية داخل «خيارات إضافية» |
| اختصارات لوحة المفاتيح | Enter للبحث فقط | Enter = مسح باركود، F2 = دفع، Escape = مسح |
| فتح درج النقدية | لم يظهر في الكود كزر في القياسي | زر ظاهر (`openCashDrawer`) |
| شاشة استلام باركود USB | تعمل عبر `useBarcodeScanner` | تعمل عبر `useBarcodeScanner` |

الملف المسؤول: `src/components/sales/express-pos-view.tsx`.

## الفواتير المعلَّقة (Parked / Held Sales)

### ما هي؟

«التعليق» (Park) هو **حفظ مؤقت للسلة الحالية** برقم مرجعي `HOLD-XXX` حتى يتمكن الكاشير من خدمة عميل آخر ثم العودة لإتمام البيع لاحقاً. لا تُخصم كميات المخزون عند التعليق (تُخصم فقط عند إتمام البيع فعلاً).

### كيف تُعلَّق الفاتورة؟

1. اضغط زر **«تعليق»** (Pause) في رأس السلة.
2. يُحفظ الكود لقطة JSON كاملة من السلة (`buildCartSnapshot` في `pos-helpers.ts`) تشمل:
   - الأصناف (المعرف، الاسم، الباركود، السعر، الكمية، الوحدة).
   - اسم العميل، الهاتف، الخصم، نسبة الضريبة، طريقة الدفع.
   - إعدادات التوصيل (السائق، الرسوم، حالة التفعيل).
3. يُعيَّن رقم `HOLD-001`, `HOLD-002`, ... تسلسلياً.
4. تُفرَّغ السلة الحالية وجاهزية لبيع جديد.

### كيف تُستعاد؟

1. اضغط زر **«المعلَّقات»** (History) في رأس السلة.
2. تظهر قائمة بكل الفواتير المعلَّقة للمستخدم الحالي.
3. لكل فاتورة: زر استئناف (Play) أو حذف (Trash).
4. عند الاستئناف:
   - إذا كانت السلة الحالية غير فارغة، يطلب النظام تأكيد الاستبدال.
   - يُحمَّل المنتج من جديد من الخادوم لضمان السعر والكمية الحاليَّين.
   - تُقارن الكمية المحفوظة بالمتاح، فإذا كان المتاح أقل تُؤخذ القيمة الأصغر.
5. بعد التحميل يتغيّر حالة المعلَّق من `PARKED` إلى `RESUMED`.

### الفرق بين Park و Parked Sale في القاعدة

| الجانب | Park (تعليق) | Sale (بيع مكتمل) |
|---|---|---|
| الخصم من المخزون | ❌ لا | ✅ نعم فوراً |
| رقم مرجعي | `HOLD-XXX` | `INV-XXXXXX` (تسلسل) |
| القيد المحاسبي | ❌ لا | ✅ قيد كامل |
| ربط بالعميل | ممكن في اللقطة | ممكن كرقم `customerId` |
| يمكن حذفه | ✅ من قائمة المعلَّقات | ❌ لا (يُلغى فقط عبر Cancel) |
| الحالة في قاعدة البيانات | `SuspendedSale` جدول منفصل | `Sale` |

الملفات المعنية: `src/app/api/suspended-sales/route.ts` و `src/app/api/suspended-sales/[id]/route.ts` و `src/hooks/use-pos.ts` (دالتا `parkCurrentCart` و `resumeParked`).

### ما الذي لم يظهر في الكود؟

- **حد أقصى لعدد الفواتير المعلَّقة لكل مستخدم:** لم يظهر حد صريح في الكود.
- **انتهاء صلاحية للفواتير المعلَّقة (auto-expire):** لم يظهر.
- **حفظ تعليق مع اختيار طبقة سعر محددة:** يُحفظ فقط الكميات والأسعار، أما الطبقة فتعود من بيانات العميل عند الاستئناف.

## خلاصة

شاشة نقاط البيع مصمَّمة لتكون **سريعة، آمنة، ومتعددة الأدوار**:

- سريعة عبر الوضع Express وحقل الباركود المُركَّز تلقائياً.
- آمنة عبر إعادة احتساب الأسعار على الخادوم (تجاهل سعر العميل) وخصم المخزون atomic.
- متعددة الأدوار عبر صلاحيات الخصم وطبقات الأسعار.
