# 04 — الفواتير والمرتجعات والإلغاء

> دليل تدريبي مبسّط لشاشة الفواتير في نظام KWPOS: تخطيطها، البحث فيها، لوحة التفاصيل، الطباعة (حراري + A4)، المرتجع الجزئي (Partial Refund)، الإلغاء الكامل (Cancel)، الفرق بينه وبين الحذف، حالات الفاتورة والمرتجع، عودة المخزون وعكس القيد المحاسبي، والصلاحيات. مكتوب لغير المختصين تقنياً مع إشارة للملفات المصدرية.

---

## 1. تخطيط شاشة الفواتير

تتبع شاشة الفواتير نمط **«Master–Detail»** (قائمة رئيسية + لوحة تفاصيل):

| المنطقة | الموقع | المحتوى |
|---|---|---|
| **العنوان** | أعلى الصفحة | «الفواتير» + زر «فاتورة جديدة» (يُعيدك لشاشة POS) |
| **حقل البحث** | تحت العنوان | بحث فوري في رقم الفاتورة واسم العميل |
| **القائمة الرئيسية** | يسار (5 أعمدة من 12) | بطاقات مختصرة لكل فاتورة |
| **لوحة التفاصيل** | يمين (7 أعمدة من 12) | تفاصيل الفاتورة المختارة |
| **أزرار الإجراء** | أسفل لوحة التفاصيل | طباعة حراري، طباعة A4، إلغاء فاتورة، مرتجع |
| **الترقيم** | أسفل القائمة الرئيسية | السابق/التالي + رقم الصفحة |

الملف المسؤول: `src/components/sales/invoices-view.tsx`.

---

## 2. البحث عن فاتورة

يتم البحث في خانة واحدة تشمل:

| حقل البحث | ما يُطابَق |
|---|---|
| **رقم الفاتورة** (`invoiceNo`) | تطابق جزئي غير حساس لحالة الأحرف |
| **اسم العميل** (`customerName`) | تطابق جزئي غير حساس لحالة الأحرف |

> **ملاحظة:** البحث بالهاتف **لم يظهر كحقل مستقل** في الكود الحالي. إن كان لديك رقم هاتف فقط، ابحث باسم العميل إذا كان محفوظاً، أو استخدم البحث الشامل ⌘K.

يُنفَّذ البحث عبر `GET /api/sales?q=...&page=...&pageSize=10` (ملف `src/app/api/sales/route.ts`)، ويُعاد:

```json
{
  "items": [ ... ],
  "pagination": { "page": 1, "pageSize": 10, "total": 145, "totalPages": 15 }
}
```

عند فراغ الصفحة الحالية يعود النظام تلقائياً للصفحة 1 (لتجنّب شاشة فارغة بعد حذف فاتورة).

## 3. لوحة تفاصيل الفاتورة (Invoice Detail)

تتكوّن من ثلاثة أقسام:

### 3.1 الترويسة (غير قابلة للتمرير)

| العنصر | الشرح |
|---|---|
| عنوان الفاتورة | «فاتورة مبيعات» + رقم الفاتورة بخط Mono |
| تاريخ الإنشاء | تاريخ ووقت البيع |
| شارة حالة المرتجع | شارة حمراء «مُرتجعة بالكامل» أو كهرمانية «مرتجع جزئي بقيمة X» |
| شارة الإلغاء | شارة حمراء «ملغاة» تظهر عند `status = CANCELLED` |
| سبب الإلغاء | نص مائل صغير أسفل شارة الإلغاء |

### 3.2 المحتوى (قابل للتمرير)

| القسم | المحتوى |
|---|---|
| بطاقة العميل | الاسم + الهاتف (أو «عميل نقدي») |
| بطاقة الدفع | طريقة الدفع (شارة ملوّنة) + اسم البائع |
| جدول الأصناف | الصنف، الكمية، سعر الوحدة، إجمالي السطر |
| المجموعات | المجموع الفرعي، الخصم، الضريبة (بنسبتها)، الإجمالي |

> عند كون الفاتورة مُرتجعة بالكامل يظهر الإجمالي **مشطوباً** باللون الرمادي.

### 3.3 الأزرار (ثابتة في الأسفل)

| الزر | الوظيفة | الصلاحية |
|---|---|---|
| طباعة حرارية 80mm | `printThermalReceipt(sale)` | الجميع |
| طباعة A4 | `printA4Invoice(sale)` | الجميع |
| إلغاء الفاتورة (Full Cancel) | يفتح `CancelDialog` | OWNER / ADMIN / MANAGER |
| مرتجع جزئي | يفتح `RefundDialog` | ADMIN فقط |

## 4. طباعة الفاتورة

يوفر النظام **قوالب طباعة متعددة** في `src/lib/print.ts`:

| القالب | الدالة | المقاس | الاستخدام |
|---|---|---|---|
| إيصال حراري | `printThermalReceipt(sale)` | 80mm × ارتفاع تلقائي | كاشير سريع، طابعة حرارية |
| فاتورة A4 | `printA4Invoice(sale)` | A4 عمودي | للأرشيف أو بريد العميل |
| إيصال تبديل | `printExchangeReceipt(exchange)` | 80mm | عند عملية تبديل |
| ملصقات باركود | `printBarcodeLabels(products, copies)` | حسب الطلب | لإلصاقها على المنتجات |

### آلية الطباعة

1. تفتح نافذة جديدة (`window.open`) بقياس مناسب.
2. تُكتب فيها صفحة HTML كاملة بأسلوب `@page` المناسب للمقاس.
3. يُنتظر تحميل الخطوط (Tajawal، Libre Barcode 39) ثم يُستدعى `window.print()`.
4. تُقرأ معلومات المتجر (الاسم، العنوان، الهاتف، الرقم الضريبي، الشعار) من `localStorage["erp-store-info"]`.

> إذا حظر المتصفح النوافذ المنبثقة، يظهر تنبيه: «يرجى السماح بالنوافذ المنبثقة للطباعة».

## 5. المرتجع الجزئي (Partial Refund)

### القاعدة الأساسية: قاعدة 14 يوماً

يُسمح بمرتجع جزئي **خلال 14 يوماً من تاريخ الفاتورة**. إذا تجاوزت الفاتورة 14 يوماً، يُسمح بالمرتجع فقط مع **تجاوز إداري صريح** (Manager Override) — أي بتفعيل مربع اختيار «موافقة المسؤول» داخل نافذة المرتجع.

الثابت: `MAX_REFUND_DAYS = 14` في `src/app/api/sales/[id]/refund/route.ts`.

### خطوات المرتجع الجزئي

| الخطوة | الإجراء |
|---|---|
| 1 | افتح الفاتورة من شاشة الفواتير. |
| 2 | اضغط زر **«مرتجع»** (متاح لـ ADMIN فقط). |
| 3 | تفتح نافذة المرتجع (`RefundDialog`) تعرض كل صنف بكميته الأصلية والمُرتجَعة سابقاً والمتبقية. |
| 4 | أدخل الكمية المُرتجَعة لكل صنف (لا يمكن تجاوز الكمية القابلة للإرجاع). |
| 5 | (اختياري) ابحث بالباركود داخل نافذة المرتجع لتحديد صنف بسرعة. |
| 6 | إن كان قد مضى على الفاتورة أكثر من 14 يوماً: فعِّل مربع «موافقة المسؤول». |
| 7 | اضغط **«اعتماد المرتجع»**. |
| 8 | تظهر شاشة نجاح فيها رقم **إشعار الدائن** `CN-INV-XXXXXX`. |

### ماذا يحدث على المستوى الخادومي؟

يتم داخل **معاملة واحدة** (`db.$transaction`):

1. **إعادة المخزون:** لكل صنف، يُزاد `StockItem.quantity` في المخزن المعني عبر `incrementStockItem`.
2. **تحديث `Product.quantity`:** يُعاد حسابه من مجموع `StockItem`.
3. **تحديث `SaleItem.returnedQty`:** يُزاد بمقدار الكمية المُرتجَعة.
4. **تحديث `Sale.refundTotal` و`refundStatus`:**
   - إن أُرجعت كل الكميات: `refundStatus = "FULL"`.
   - وإلا: `refundStatus = "PARTIAL"`.
5. **قيد محاسبي مالي:** مدين: مردودات المبيعات (4030) + ضريبة مستحقة (2010)؛ دائن: حساب طريقة الدفع (1010 للنقدي، 1020 لغيره).
6. **قيد مخزوني (اختياري):** مدين: المخزون (1010)؛ دائن: تكلفة البضاعة المباعة (5060) — بقيمة تكلفة الوحدات المُرتجَعة.
7. **سجل تدقيق:** `SALE_REFUNDED` في `AuditLog`.

> إن فشل أي من القيود المحاسبية، **تُلغى المعاملة كاملة** (لا مرتجع، لا إعادة مخزون) لضمان الترابط المحاسبي.

### رقم إشعار الدائن (Credit Note)

يُولَّد رقم إشعار دائن بصيغة `CN-{invoiceNo}` (مثل `CN-INV-000123`) ويُعاد في `refundSummary.creditNoteNo`.

## 6. الإلغاء الكامل (Full Cancel)

### متى يُستخدم؟

عند الحاجة لإلغاء **الفاتورة كاملة** لأسباب إدارية (خطأ في الإدخال، بيع مكرر، غش موظف). لا يخضع لقاعدة 14 يوماً لأنه **عملية إدارية وليست مرتجع عميل**.

### خطوات الإلغاء

| الخطوة | الإجراء |
|---|---|
| 1 | افتح الفاتورة من شاشة الفواتير. |
| 2 | اضغط زر **«إلغاء الفاتورة»** (شارة Ban حمراء) — متاح لـ OWNER/ADMIN/MANAGER. |
| 3 | تظهر نافذة `CancelDialog` تعرض ملخص الفاتورة. |
| 4 | اكتب **سبب الإلغاء** (3 أحرف على الأقل — حقل إلزامي). |
| 5 | اضغط زر **«تأكيد الإلغاء»** الأحمر. |
| 6 | رسالة نجاح بالاسم الفاتورة. |

### ماذا يحدث على المستوى الخادومي؟

الملف: `src/app/api/sales/[id]/cancel/route.ts`.

1. **التحقق من الحالة:** رفض إن كانت ملغاة أصلاً (`already-cancelled`) أو مُرتجعة بالكامل (`already-refunded`) أو مُرتجَعة جزئياً (`has-partial-refund`).
2. **إعادة كل المخزون:** لكل صنف، إعادة `quantity − returnedQty` إلى المخزن الحالي للمستخدم.
3. **وضع علامة ملغاة:** تحديث السجل:
   - `status = "CANCELLED"`
   - `cancelledAt = new Date()`
   - `cancelledById = user.id`
   - `cancellationReason = reason`
   - `refundStatus = "FULL"`
   - `refundTotal = total` (قيمة الفاتورة كاملة)
4. **قيد عكسي محاسبي** (fire-and-forget، غير فادح):
   - دائن: حساب طريقة الدفع (1010/1020) — برد المبلغ.
   - مدين: إيراد المبيعات (4010) — عكس الإيراد.
   - مدين: ضريبة مستحقة (2010) — عكس الضريبة.
5. **مزامنة `Product.quantity`** (استعلام SQL واحد).
6. **سجل تدقيق:** `CANCEL_TXN` في `AuditLog` مع سبب الإلغاء.

> الفاتورة **لا تُحذف من قاعدة البيانات**. تبقى ظاهرة في شاشة الفواتير بشارة «ملغاة»، وتُستثنى إيراداتها من التقارير المالية لأن القيد العكسي صفّر أثرها.

## 7. الفرق بين الإلغاء (Cancel) والحذف (Delete)

| الجانب | الإلغاء (Cancel) | الحذف (Delete) |
|---|---|---|
| **هل يُبقي السجل؟** | ✅ نعم، يبقى في `Sale` | ❌ لا، يُزال كلياً |
| **حالة السجل** | `status = "CANCELLED"` | لا يوجد سجل أصلاً |
| **الأثر المحاسبي** | قيد عكسي يصفّر الإيراد | لا أثر (السجل مفقود) |
| **الأثر المخزوني** | يُعاد كل المخزون | لا أثر (لم يكن قد خُصم أصلاً حسب المنطق) |
| **سجل التدقيق** | يُكتب `CANCEL_TXN` بالسبب | لا يُكتب شيء (إلا حذف السجل نفسه إن وُجد) |
| **يُستخدم لـ** | تصحيح أخطاء إدارية بعد البيع | (غير معتمد في النظام) |
| **الصلاحية** | OWNER/ADMIN/MANAGER | غير منفذ في كود المبيعات |

> **في نظام KWPOS لا يوجد «حذف فاتورة» كعملية معرَّضة.** الفكرة: كل فاتورة تُترك لقابلية التدقيق وعدم التلاعب. إن أردت إزالة أثر فاتورة، تستخدم الإلغاء.

## 8. إشعار الدائن (Credit Note)

عند كل عملية مرتجع أو إلغاء، يُولِّد الخادوم إشعار دائن برقم `CN-{invoiceNo}` يُعاد في الاستجابة ضمن `refundSummary`:

```json
{
  "refundSummary": {
    "refundSubtotal": 50.000,
    "refundTax": 7.500,
    "refundTotal": 57.500,
    "refundCost": 30.000,
    "creditNoteNo": "CN-INV-000123"
  }
}
```

> **ملاحظة:** لم يظهر في الكود جدول مستقل لتخزين إشعارات الدائن. القيمة تُحفظ ضمن `Sale.refundTotal` و`Sale.refundStatus`، ويعود الرقم `CN-INV-XXXXXX` من طبقة الـ API فقط كمرجع. للتدقيق المحاسبي يعتمد النظام على القيد المحاسبي + سجل التدقيق.

## 9. حالات الفاتورة (Sale.status)

من `prisma/schema.prisma` (نموذج `Sale`):

| الحالة | المعنى | متى؟ |
|---|---|---|
| `COMPLETED` | فاتورة مكتملة وسارية | الافتراضي عند إنشاء البيع |
| `CANCELLED` | فاتورة ملغاة | بعد استدعاء `POST /api/sales/[id]/cancel` |

> لا توجد حالة `PENDING` أو `DRAFT` لفاتورة البيع (هذه حالات لأوامر الشراء، لا للبيع). كل بيع فور اعتماده يُحفظ COMPLETED.

---

## 10. حالات المرتجع (Sale.refundStatus)

| الحالة | المعنى | شارة العرض |
|---|---|---|
| `NONE` | لا مرتجع | لا شارة |
| `PARTIAL` | مُرتجَع جزئي بقيمة `refundTotal > 0` | شارة كهرمانية «مرتجع جزئي بقيمة X» |
| `FULL` | مُرتجَع بالكامل (كل الكميات أُرجعت أو الفاتورة مُلغاة) | شارة حمراء «مُرتجعة بالكامل» + الإجمالي مشطوب |

التحديث يتم في كل من مسار المرتجع (`refund/route.ts`) ومسار الإلغاء (`cancel/route.ts`).

## 11. متى يمنع النظام الإلغاء أو المرتجع؟

| الحالة | الرد من الخادوم | الشاشة |
|---|---|---|
| محاولة إلغاء فاتورة مُلغاة | `already-cancelled` (409) | رسالة: «الفاتورة ملغاة أصلاً» |
| محاولة إلغاء فاتورة مُرتجعة بالكامل | `already-refunded` (409) | رسالة: «الفاتورة مُرتجعة بالكامل» |
| محاولة إلغاء فاتورة مُرتجَعة جزئياً | `has-partial-refund` (409) | رسالة: «لا يمكن إلغاء فاتورة بها مرتجع جزئي» |
| محاولة مرتجع بعد 14 يوماً بدون تجاوز إداري | `past-14-days` (403) | نافذة تحذير + مربع تجاوز إداري |
| محاولة مرتجع بكمية أكبر من المتاح | `exceeds-returnable:{name}:{qty}` (400) | رسالة محلية في النافذة |
| محاولة مرتجع بدون عناصر | `no-items` (400) | رسالة في النافذة |
| محاولة مرتجع/إلغاء بدون مخزن مرتبط | `no-warehouse-available` (400) | رسالة خطأ |
| عدم كتابة سبب الإلغاء (أقل من 3 أحرف) | `reason-required` (400) | رسالة «سبب الإلغاء مطلوب» |

في الواجهة، يظهر زر «المرتجع» **معطَّلاً** للفواتير المُرتجعة بالكامل، ويظهر زر «الإلغاء» **مخفياً** للفواتير المُلغاة أو المُرتجعة بالكامل.

## 12. كيف يعود المخزون؟

عند المرتجع الجزئي أو الإلغاء الكامل، يتم:

1. تحديد المخزن المستهدف عبر `resolveWarehouseId(user, null)` — مخزن المستخدم الحالي أو المخزن الافتراضي.
2. لكل صنف، استدعاء `incrementStockItem(tx, productId, warehouseId, qty)`:
   - إن وُجد سجل `StockItem` للمنتج في ذلك المخزن: يُزاد `quantity` بمقدار المُرتجَع.
   - إن لم يوجد: يُنشأ سجل جديد (upsert) بالكمية المُرتجَعة.
3. إعادة حساب `Product.quantity` من مجموع `StockItem` عبر `updateProductQuantityFromStockItems` — وهذا يضمن أن الكمية الكلية المعروضة في شاشة المخزون محدَّثة.

> **في الإلغاء:** تُعاد كمية `quantity − returnedQty` (أي الكمية الصافية التي لم تُرتجع أصلاً). هذا يمنع إعادة مخزون مضاعفة في حال مرتجع جزئي سابق.

## 14. الصلاحيات

| العملية | الأدوار المصرّحة | المصدر |
|---|---|---|
| **فتح شاشة الفواتير** | OWNER, ADMIN, MANAGER, ACCOUNTANT, SALES, CASHIER (حسب تفضيلات الدور) | `ROLE_PERMISSIONS` في `src/lib/session.ts` |
| **عرض تفاصيل فاتورة** | كل من يرى القائمة | نفس المصدر |
| **طباعة فاتورة** | كل من يرى الفاتورة | لا يوجد قيد إضافي في الكود |
| **مرتجع جزئي** | **ADMIN فقط** | `if (user.role !== "ADMIN") return 403` في `refund/route.ts` |
| **إلغاء كامل** | **OWNER + ADMIN + MANAGER** | شرط صريح في `cancel/route.ts` |

> **ملاحظة:** في الواجهة (`invoices-view.tsx`) يظهر الزرَّان للقيمة `isAdmin` المساوية لـ `ADMIN || OWNER || MANAGER`. لكن على مستوى الخادوم يكون مرتجع الـ ADMIN حصراً، فإذا ضغط MANAGER على زر المرتجع في الواجهة، يُرفض من الخادوم بـ 403.

## 16. التبديل (Exchange) — إشارة سريعة

التبديل موضوعه وثيقة منفصلة، لكن نُشير هنا إلى تكامله مع الفواتير:

- التبديل يتطلب **رقم الفاتورة الأصلية إلزامياً** (`original-invoice-required`).
- لا يمكن تبديل منتج لم يكن في الفاتورة الأصلية (`product-not-in-invoice`).
- يخضع لنفس قاعدة 14 يوماً (`invoice-too-old`).
- يُحدِّث `SaleItem.returnedQty` ضمن نفس المعاملة لمنع التلاعب.
- يُولِّد قيد محاسبي واحد بفرق الصافي (مدين/دائن حسب الإشارة).

الملف المسؤول: `src/app/api/exchanges/route.ts` و`src/components/sales/exchange-view.tsx`.

## خلاصة

- شاشة الفواتير تتبع نمط Master–Detail وتدعم البحث الفوري والترقيم.
- المرتجع الجزئي (Refund) مخصَّص لـ **ADMIN فقط** ويخضع لقاعدة 14 يوماً مع إمكانية التجاوز الإداري.
- الإلغاء الكامل (Cancel) متاح لـ **OWNER/ADMIN/MANAGER**، يتطلب سبباً، ولا يخضع لقاعدة 14 يوماً.
- لا يوجد «حذف فاتورة» — كل فاتورة تبقى لقابلية التدقيق، والإلغاء يضع `status = CANCELLED` فقط.
- إشعار الدائن برقم `CN-{invoiceNo}` يُولَّد في كلتا العمليتين.
- المخزون يعود عبر `StockItem` increment، والقيد المحاسبي يُعكس بقيد عكسي أو قيد مردودات حسب الحالة.

الخطوة التالية في التوثيق: شاشة التبديل والمرتجعات الشاملة.
