# 07 — الموردون وسداد الموردين

> دليل تدريبي مبسّط لشاشة الموردين وشاشة سداد الموردين في نظام KWPOS. يغطي تخطيط الشاشتين، أنواع الموردين (محلي/أجنبي)، نموذج السداد (المبلغ، طريقة الدفع، المرجع، الملاحظات)، منع السداد الزائد عن الرصيد المستحق، التجاوز الإداري، القيد المحاسبي للسداد، حذف السداد وأثره، وكشف حساب المورد. مكتوب لغير المختصين تقنياً مع الإشارة للملفات المصدرية.

---

## 1. شاشة الموردين — التخطيط

الملف: `src/components/purchases/suppliers-view.tsx`.

تُعرض الموردين على شكل **بطاقات شبكية** (responsive: عمود واحد على الجوال، عمودان على اللوحي، 3 أعمدة على الحاسوب). كل بطاقة تحتوي:

| المنطقة | المحتوى |
|---|---|
| **الترويسة** | أيقونة شاحنة + اسم المورد + شارة النوع (محلي/أجنبي) + جهة الاتصال |
| **قائمة MoreVertical** (للمصرّحين) | سداد، كشف حساب، تعديل، حذف |
| **بيانات الاتصال** | الهاتف (LTR)، البريد (LTR)، العنوان |
| **شارات سفلية** | عدد المنتجات + عدد الطلبات + **الرصيد المستحق** (موجب = أحمر، صفر/سالب = أخضر) |
| **زر «سداد مورد»** | يفتح `SupplierPaymentDialog` (للمصرّحين) |

### رأس الشاشة

| العنصر | الشرح |
|---|---|
| العنوان | «الموردون» + أيقونة شاحنة |
| الوصف | شرح مختصر للوحدة |
| زر «إضافة مورد» | متاح فقط لـ ADMIN/WAREHOUSE |

### مصدر بيانات الرصيد

تستدعي الشاشة `GET /api/suppliers/balances` (ملف `src/app/api/suppliers/balances/route.ts`) الذي يحسب لكل مورد:

```
balance = SUM(POSTED PurchaseInvoice.total)
        − SUM(SupplierPayment.amount)
        − SUM(PurchaseReturn.total)
```

- الرصيد **الموجب** = المتجر مدين للمورد (طبيعي للمشتريات الآجلة) — شارة حمراء.
- الرصيد **الصفر/السالب** = لا دين أو دين لصالح المتجر (مقدم/سداد زائد) — شارة خضراء.

## 3. شاشة سداد الموردين

الملف: `src/components/purchases/supplier-payments-view.tsx`.

تظهر هذه الشاشة كـ **تبويب** داخل وحدة المشتريات (التبويب `payments`). تتكون من:

| المنطقة | المحتوى |
|---|---|
| **الترويسة** | أيقونة محفظة + «سداد الموردين» + وصف + زر «سداد جديد» |

| **حالة فارغة** | رسالة «لا توجد سدادات» + زر «سداد جديد» |

### أعمدة جدول السدادات

| العمود | الشرح |
|---|---|
| **رقم السداد** | `paymentNo` (مثل `SPAY-0001`) بخط Mono LTR |

| **المبلغ** | `amount` باللون الأخضر (خروج نقدي) |
| **طريقة الدفع** | شارة ملوّنة: نقدية (أخضر) / بنك (أزرق) / شيك (كهرماني) |
| **التاريخ** | `paymentDate` |
| **المرجع** | `referenceNo` (رقم الشيك/تحويل البنك) — LTR |
| **الملاحظة** | `note` (مخفية على الجوال) |
| **إجراءات** | قائمة MoreVertical لـ ADMIN فقط → «حذف» |

### الصلاحيات

| العملية | الأدوار المصرّحة |
|---|---|
| عرض السدادات | كل من يصل لتبويب المشتريات |
| إنشاء سداد | ADMIN / WAREHOUSE |
| حذف سداد | **ADMIN فقط** |

> **لم يظهر في الكود:** تعديل سداد بعد إنشائه. لتصحيح خطأ: احذف السداد وأنشئ واحداً جديداً.

## 5. منع السداد الزائد (Balance Check)

### القاعدة

يحسب الخادوم **الرصيد المستحق** قبل قبول السداد:

```
outstandingBalance = SUM(POSTED PurchaseInvoice.total)
                   − SUM(SupplierPayment.amount)
                   − SUM(PurchaseReturn.total)
```

إن كان `amount > outstandingBalance + 0.001` **بدون تجاوز إداري**، يُرفض السداد بخطأ:

```json
{
  "error": "exceeds-balance",
  "message": "المبلغ يتجاوز الرصيد المستحق / Amount exceeds outstanding balance",
  "amount": 200,
  "outstandingBalance": 150
}
```

(الحالة 400).

> يسمح التفاوت 0.001 لتفادي مشاكل التقريب.

---

## 6. التجاوز الإداري (Manager Override)

### آلية العمل

يدعم الـ API حقل `override: true` مع `body` يُرسل مع السداد:

```js
const isAdmin = hasRole(user.role, ["OWNER", "ADMIN"])
const override = body?.override === true && isAdmin

if (amt > outstandingBalance + 0.001 && !override) {
  return NextResponse.json({ error: "exceeds-balance", ... }, { status: 400 })
}
```

- **متاح فقط لـ OWNER/ADMIN** (ليس لـ WAREHOUSE).
- يُسمح بالسداد الزائد عن الرصيد (مثلاً: دفعة مقدمة لمورد قبل استلام فاتورة).
- يُسجَّل في سجل التدقيق برسالة تنبيهية:
  ```
  سداد مورد SPAY-0007 — المورد X — 500.000 د.ك ⚠️ تجاوز إداري (المبلغ > الرصيد 150.000)
  ```

> **لم يظهر في الواجهة (`supplier-payment-dialog.tsx`):** مربع اختيار «تجاوز إداري» أو أي حقل يُرسل `override: true`. النموذج الحالي **لا يكشف هذه الميزة**، فالمستخدم العادي سيرى فقط خطأ `exceeds-balance` دون طريقة لتجاوزه. لاستخدام التجاوز: يجب تطوير الواجهة لتضيف مربع اختيار (ويفضّل أن يكون مرئياً لـ ADMIN فقط عند تجاوز المبلغ الرصيد)، أو إرسال الطلب يدوياً عبر API مع `override: true`.

ملف الاختبار الذي يغطّي هذه السلوكيات: `tests/supplier-payment-limit.test.ts`.

## 7. القيد المحاسبي للسداد

عند نجاح السداد، يُنشئ الخادوم قيداً محاسبياً متوازناً عبر `createJournalEntry`:

### شكل القيد

| الحساب | مدين | دائن |
|---|---|---|
| **2010 — ذمم دائنة (Accounts Payable)** | المبلغ | — |
| **1010 — الصندوق (نقدية)** *أو* **1020 — البنك (تحويل/شيك)** | — | المبلغ |

### تحديد الحساب الدائن حسب الطريقة

| الطريقة | الحساب الدائن | الوصف |
|---|---|---|
| `CASH` | **1010** | نقدي من الصندوق |
| `BANK` | **1020** | تحويل بنكي |
| `CHECK` | **1020** | شيك (يُعامل كبنك لأن الشيك يصرف عبر البنك) |

### الوصف في القيد

- **الوصف العام:** `سداد للمورد {اسم} — {paymentNo} ({الطريقة بالعربية})`
- **سطر المدين:** `سداد {paymentNo} — {اسم المورد}`
- **سطر الدائن:** `{الطريقة بالعربية} — {paymentNo}`

### الترابط

- القيد يُكتب **داخل نفس معاملة** إنشاء السداد (`db.$transaction`).
- فشل القيد **يرجع السداد بالكامل** (`supplier-payment-failed`).
- بعد نجاح القيد، يُحفظ `journalEntryId` في سجل `SupplierPayment` لربطه بالقيد.

## 8. حذف السداد وأثره

الملف: `src/app/api/supplier-payments/[id]/route.ts`.

### الصلاحية

**ADMIN فقط** (`hasRole(user.role, ["OWNER", "ADMIN"])`). WAREHOUSE لا يمكنه الحذف.

### ماذا يحدث عند الحذف؟

1. جلب السداد + `journalEntryId` المرتبط.
2. لكل سطر في القيد:
   - حساب `delta = debit − credit`.
   - `Account.balance -= delta` (عكس التأثير الأصلي).
3. حذف سطور القيد (`JournalLine`).
4. حذف رأس القيد (`JournalEntry`) — متسامح إن لم يعد موجوداً.
5. حذف سجل `SupplierPayment`.

### الأثر الناتج

| الجانب | الأثر |
|---|---|
| **رصيد المورد المستحق** | يرتفع (يستعيد الدين قيمته قبل السداد) |
| **رصيد الصندوق/البنك** | يرتفع (كأن السداد لم يحدث) |
| **رصيد ذمم الدائنة** | يرتفع (كأن السداد لم يحدث) |
| **القيد المحاسبي** | يُحذف بالكامل |
| **سجل التدقيق** | **لم يظهر في الكود** سجل تدقيق صريح لعملية الحذف |

> الحذف **متسامح مع الأخطاء** (idempotent): إن فُقد القيد المحاسبي لأي سبب، يُكمل الحذف بنجاح.

## 9. كشف حساب المورد (SupplierStatementDialog)

الملف: `src/components/purchases/supplier-statement-dialog.tsx` و`src/app/api/suppliers/[id]/statement/route.ts`.

يُفتح من القائمة المنسدلة في بطاقة المورد (خيار «كشف حساب»). يتكون من:

### 9.1 الترويسة

- العنوان: «كشف حساب مورد»
- اسم المورد + الوصف

### 9.2 فلترة التاريخ

| الحقل | الشرح |
|---|---|
| **من تاريخ** | `from` — تحديد بداية الفترة |
| **إلى تاريخ** | `to` — تحديد نهاية الفترة |

### 9.3 بطاقات الملخص (4 بطاقات)

| البطاقة | الشرح | اللون |
|---|---|---|
| **الرصيد الافتتاحي** | مجموع الفواتير − السدادات − المرتجعات **قبل** `from` | رمادي |
| **إجمالي الفواتير** | مجموع فواتير الفترة (POSTED فقط) | أزرق |
| **إجمالي السدادات** | مجموع سدادات الفترة | أخضر |
| **الرصيد الختامي** | الرصيد الافتتاحي + (الفواتير − السدادات − المرتجعات) | أحمر (إن > 0) أو أخضر |

### 9.4 جدول المعاملات

| العمود | الشرح |
|---|---|
| **التاريخ** | تاريخ المعاملة |
| **النوع** | شارة ملوّنة: فاتورة (أزرق) / سداد (أخضر) / مرتجع (كهرماني) |
| **المرجع** | رقم الفاتورة أو السداد أو المرتجع (LTR Mono) |
| **الوصف** | ملاحظة المعاملة أو وصف افتراضي |
| **مدين** | قيمة الفاتورة (تزيد رصيد المورد) |
| **دائن** | قيمة السداد أو المرتجع (تقلل رصيد المورد) |
| **الرصيد** | الرصيد الجاري بعد كل معاملة (أحمر إن > 0، أخضر إن ≤ 0) |

### 9.5 الأزرار

| الزر | الوظيفة |
|---|---|
| **إغلاق** | إغلاق النافذة |
| **طباعة** | طباعة كشف الحساب عبر `printReportOnly` |
| **تصدير** | شريط `ExportToolbar` لتصدير CSV/Excel/طباعة بأعمدة: التاريخ، النوع، المرجع، الوصف، مدين، دائن، الرصيد |

### 9.6 منطق الحساب في الخادوم

```js
// الرصيد الافتتاحي
openingBalance = SUM(invoices BEFORE from) − SUM(payments BEFORE from) − SUM(returns BEFORE from)

// المعاملات (مرتبة زمنياً)
txs = [invoices (POSTED)] + [payments] + [returns]

// الرصيد الجاري
running = openingBalance
for tx in txs:
  running += tx.debit − tx.credit
  tx.balance = running

closingBalance = running
```

## خلاصة

- شاشة الموردين تعرض بطاقات شبكية ببيانات الاتصال والرصيد المستحق لكل مورد.
- نوع المورد (محلي/أجنبي) يؤثر على حقول التكلفة الكاملة في فاتورة المشتريات.
- شاشة سداد الموردين قائمة جدولية، يُدار منها كل سداد مع رقم تلقائي `SPAY-NNNN`.
- نموذج السداد يتطلب: مورد + مبلغ + طريقة دفع (CASH/BANK/CHECK) + تاريخ، والرقم المرجعي والملاحظة اختياريان.
- يمنع الخادوم السداد الزائد عن الرصيد المستحق (`exceeds-balance`) ما لم يُفعّل **تجاوز إداري** (متاح لـ OWNER/ADMIN عبر `override: true` — لكن لم يُكشف في واجهة المستخدم).
- يُنشئ السداد قيداً محاسبياً متوازناً: **مدين 2010 (ذمم دائنة) / دائن 1010 (نقدية) أو 1020 (بنك/شيك)**.
- حذف السداد متاح لـ ADMIN فقط، ويُعكس القيد المحاسبي ويُحذف السداد نهائياً.
- كشف حساب المورد يوحّد الفواتير والسدادات والمرتجعات في مجرى واحد مع رصيد افتتاحي وختامي وجاري، مع إمكانية الطباعة والتصدير.

هذا يُنهي وثيقة الموردين والسدادات. الخطوة التالية في التوثيق: وحدة المحاسبة والقيود المجلة.
