# 05 — المخزون: الأصناف، الفئات، الوحدات، المخازن، الجرد، التحويلات

> دليل تدريبي مبسّط لوحدة المخزون في نظام KWPOS. يغطي شاشة المنتجات ونموذج الإضافة/التعديل، الفئات الأب-الابن، الوحدات، المخازن، حركات المخزون، الجرد الكامل والعشوائي، والتحويلات بين المخازن. مكتوب لغير المختصين تقنياً مع الإشارة للملفات المصدرية.

## 7.1 شاشة المنتجات

### التخطيط العام

تتكون شاشة المنتجات (تبويب `products`) من:

| المنطقة | المحتوى |
|---|---|
| **الترويسة** | العنوان + أزرار: استيراد Excel، تصدير Excel، طباعة باركود، «إضافة صنف» |
| **شريط الفلترة** | بحث نصي + فلتر فئة (شجرة أب-ابن) + زر «قارب على النفاد» |
| **الجدول** | الصنف، الباركود، الفئة، الكمية، حد إعادة الطلب، التكلفة (للأدوار المصرّحة فقط)، سعر البيع، إجراءات |
| **التذييل** | عدد المنتجات المعروضة |

الملف: `src/components/inventory/inventory-view.tsx`.

### أعمدة الجدول

| العمود | الحقل | ملاحظات |
|---|---|---|
| **الصنف** | `name` | يُعرض تحته اسم الفئة على الجوال فقط |
| **الباركود** | `barcode` | خط Mono، LTR |
| **الفئة** | `categoryName` | شارة (Badge) رمادية |
| **الكمية** | `quantity` | شارة ملوّنة: حمراء (حرجة ≤ نصف حد الطلب)، رمادية (منخفضة ≤ حد الطلب)، عادية |
| **حد إعادة الطلب** | `reorderLevel` | للشاشات الكبيرة فقط |
| **سعر التكلفة** | `costPrice` | يظهر فقط للأدوار التي ترى التكلفة (`canSeeCost`) |
| **سعر البيع** | `salePrice` | بخط عريض |
| **الإجراءات** | قائمة منسدلة | تعديل / حذف حسب الصلاحية |

### الفلاتر والبحث

| الفلتر | كيف يعمل |
|---|---|
| **البحث النصي** | `q` يُطابِق `name` أو `barcode` (تطابق جزئي غير حساس للحالة) — مُنفَّذ عبر `GET /api/products?q=` |
| **فلتر الفئة** | قائمة شجرية تُظهر الفئة الأب ثم الأبناء ببادئة ↳ |
| **زر «قارب على النفاد»** | يضيف معامل `lowStock=true` فيُعيد الخادوم المنتجات التي `quantity <= reorderLevel` |

### أزرار الترويسة

| الزر | الوظيفة | الصلاحية |
|---|---|---|
| **استيراد Excel** | رفع ملف Excel لإضافة منتجات دفعة | `canManageProducts` (OWNER/ADMIN/WAREHOUSE) |
| **تصدير Excel** | تنزيل قائمة المنتجات الحالية | `canManageProducts` |
| **طباعة باركود** | فتح نافذة طباعة ملصقات باركود لكل المنتجات المعروضة | الجميع |
| **إضافة صنف** | فتح `ProductFormDialog` فارغاً | `canManageProducts` |

### إجراءات الصف (قائمة MoreVertical)

| الإجراء | الوظيفة | الصلاحية |
|---|---|---|
| **تعديل** | فتح `ProductFormDialog` محمَّلاً ببيانات الصنف | `canManageProducts` |
| **حذف** | فتح نافذة تأكيد حذف الصنف | `canDelete` (OWNER/ADMIN/MANAGER) |

> حذف المنتج يتم عبر `DELETE /api/products/[id]` ويسجّل `PRODUCT_DELETED` في سجل التدقيق.

### حقول المنتج الكاملة (نموذج البيانات)

يحمل نموذج `Product` في الخادوم (`src/lib/serialize/inventory.ts`) الحقول التالية، حتى لو لم تظهر كلها في نموذج الإضافة المبسّط:

| الحقل | الشرح |
|---|---|
| `name` | اسم الصنف (إلزامي) |

| `reorderLevel` | حد إعادة الطلب |
| `optimalOrderQty` | الكمية المثلى لإعادة الطلب (تُستخدم في إنشاء أوامر الشراء التلقائية) |
| `costPrice` | سعر التكلفة |
| `salePrice` | سعر البيع للمستهلك |
| `wholesalePrice` | سعر الجملة |
| `corporatePrice` | سعر الشركات |
| `taxRate` | نسبة الضريبة المطبّقة على الصنف |
| `unit` / `unitId` | اسم الوحدة + معرف الوحدة (قطعة، علبة، كيلو…) |
| `imageUrl` | رابط صورة المنتج |
| `stockByWarehouse` | مصفوفة توزيع الكمية على المخازن |

## 7.2 نموذج إضافة/تعديل المنتج

الملف: `src/components/inventory/product-form-dialog.tsx`.

> **ملاحظة مهمة:** النموذج الحالي **مبسّط** — يعرض فقط: الصورة، الاسم، الباركود، الفئة، الوحدة. أما بقية الحقول المالية (التكلفة/البيع/الجملة/الشركات/الضريبة) وحدود الطلب والمورد، فيتم **حفظها من شاشة التسعير** (Pricing Engine) لـ ADMIN، ويحتفظ نموذج المنتج بقيمها الحالية عند التعديل ولا يلمسها.

### الحقول المعروضة في النموذج

| الحقل | النوع | الشرح |
|---|---|---|
| **صورة المنتج** | مكوّن `ImageUpload` | رفع صورة أو إدخال رابط URL |
| **اسم المنتج** | نص (إلزامي) | يُتحقق الخادوم أنه غير فارغ (`name-required`) |
| **الباركود** | نص LTR | حر، مع زر «توليد تلقائي» |
| **الفئة** | Select | يعرض الفئات الابنة فقط (انظر 7.3) |
| **الوحدة** | Select | يُفرَغ إن لم توجد وحدات، مع زر «إدارة الوحدات» ينقل لشاشة الإعدادات |

### زر «توليد الباركود تلقائياً»

يستدعي `GET /api/products/generate-barcode?categoryId=...` في الخادوم (`src/app/api/products/generate-barcode/route.ts`):

1. يطلب اختيار فئة أولاً (وإلا: تنبيه «اختر الفئة أولاً»).
2. يحدّد بادئة الفئة (1 خانة): إن وُجد `category.barcodePrefix` (1-9) يُستخدم، وإلا يُؤخذ ترتيب الفئة بين كل الفئات + 1.
3. يحدّد التسلسل (12 خانة) = عدد المنتجات الحالية في الفئة + 1.
4. يُولَّد باركود EAN-13 عبر `generateProductBarcode(prefix, sequence)`.

> مثال: الفئة الأولى (بلا بادئة صريحة) → `1000000000001`, `1000000000002`...

### الأزرار

| الزر | الوظيفة |
|---|---|
| **إلغاء** | إغلاق النافذة دون حفظ |
| **حفظ التغييرات / إضافة** | إرسال النموذج للخادوم |

عند الإرسال:

- **إنشاء:** `POST /api/products` بـ `quantity=0` وكل الأسعار صفر افتراضياً.
- **تعديل:** `PUT /api/products/[id]` مع **حفظ القيم الموجودة** للأسعار والمورد وحدود الطلب.

### توزيع المخزون على المخازن

يدعم الـ API حقلاً اسمه `warehouseStock: [{ warehouseId, quantity }]` يُمرَّر عند الإنشاء أو التعديل:

- عند الإنشاء: تُنشأ سجلات `StockItem` لكل مخزن، وتُجمَع الكميات في `Product.quantity`.
- عند التعديل: يُحدِّث الخادوم كل `StockItem` (upsert) داخل معاملة، ثم يعيد حساب `Product.quantity` من مجموع `StockItem` عبر `updateProductQuantityFromStockItems`.

> **لم يظهر في الكود:** واجهة إدخال مرئية لتوزيع الكميات على المخازن من نموذج المنتج نفسه. حالياً يتم توزيع المخزون عبر استلام فواتير الشراء أو التحويلات أو الجرد، لا بإدخال يدوي مباشر من النموذج.

## 7.3 الفئات (Categories)

الملف: `src/app/api/categories/route.ts` و`[id]/route.ts`.

### بنية الأب-الابن (Nested)

- كل فئة لها حقل اختياري `parentId`.
- الفئات الجذرية (`parentId = null`) هي الآباء.
- الفئة الواحدة يمكن أن تحتوي أبناء، وكل ابن يمكن أن يحتوي أبناء أيضاً (نظرية غير محدودة، لكن الواجهة تعرض مستويين فقط حالياً).

يُعيد الخادوم شكلين:
- `items`: قائمة مسطحة بكل الفئات + `parentName` + `childrenCount`.
- `tree`: شجرة تحتوي الجذور، وكل جذر يحمل `children`.

### لماذا المنتجات تستخدم فئات ابنة؟

في نموذج المنتج، يعرض Select الفئات الابنة فقط. الإجراء: إن كانت الفئة الجذرية ليس لها أبناء، تظهر مباشرة؛ وإن كان لها أبناء، يظهر كل ابن بالصيغة `الأب › الابن`. السبب: ضمان تخصيص المنتج لأدق فئة ممكنة، مما يُحسّن:

- التقارير المجمّعة حسب الفئة.
- توليد الباركود بمجموعات منطقية.
- فلترة سريعة في POS وشاشة المخزون.

> **لم يظهر في الكود:** منع صريح لتخصيص المنتج لفئة أب (الـ API يقبل `categoryId` لأي فئة، لكن الواجهة تدفع نحو الابنة).

### حقول الفئة

| الحقل | الشرح |
|---|---|
| `name` | الاسم (إلزامي) |
| `code` | رمز قصير (حتى 4 أحرف، فريد) |
| `barcodePrefix` | رقم 1-9 يُستخدم كبادئة لتوليد الباركود |
| `imageUrl` | صورة الفئة |
| `parentId` | الفئة الأب (اختياري) |

### الإجراءات

| الإجراء | المسار | الصلاحية | ملاحظة |
|---|---|---|---|
| **إضافة** | `POST /api/categories` | OWNER/ADMIN/WAREHOUSE | يتحقق من فريدة `code` |
| **تعديل** | `PUT /api/categories/[id]` | OWNER/ADMIN/WAREHOUSE | يمنع أن تكون الفئة أباً لنفسها |
| **حذف** | `DELETE /api/categories/[id]` | `canDelete` | **يُرفض** إن كان للفئة منتجات (`has-products`، 409) |

## 7.4 الوحدات (Units)

الملف: `src/app/api/units/route.ts` و`[id]/route.ts`.

الوحدات هي وحدات القياس (قطعة، علبة، كيلوغرام، لتر…). تُستخدم في:

- **المنتجات:** حقل `unitId` يربط المنتج بوحدة، ويُنسخ اسمها في حقل `unit` للتوافق مع الأنظمة القديمة.
- **التركيبات (Compositions):** في وصفات التصنيع (مكوّنات التركيبة).

### الإجراءات

| الإجراء | المسار | الصلاحية |
|---|---|---|
| **عرض الكل** | `GET /api/units` | كل المستخدمين المسجَّلين |
| **إضافة** | `POST /api/units` | **ADMIN فقط** |
| **حذف** | `DELETE /api/units/[id]` | **ADMIN فقط** |

> **لم يظهر في الكود:** مسار `PUT /api/units/[id]` لتعديل اسم الوحدة. التعديل غير مدعوم حالياً، فقط الإضافة والحذف.

عند محاولة إضافة وحدة باسم مكرر يُعاد خطأ `name-exists` (409).

## 7.5 المخازن (Warehouses)

الملف: `src/components/inventory/warehouse-manager.tsx` و`warehouse-form-dialog.tsx`، و`src/app/api/warehouses/route.ts` و`[id]/route.ts`.

### بطاقة المخزن تعرض

| العنصر | الشرح |
|---|---|
| الاسم | `name` |
| الرمز | `code` بخط Mono |
| الموقع | `location` |
| عدد الأصناف | `productsCount` (عدد سجلات `StockItem` بكمية > 0) |
| إجمالي الكميات | `totalStock` (مجموع كل الكميات) |
| شارة «غير نشط» | تظهر إن `isActive = false` |

### حقول المخزن

| الحقل | الشرح |
|---|---|
| `name` | إلزامي |
| `code` | اختياري (مثل: WH-01) |
| `location` | اختياري (وصف نصي) |
| `isActive` | افتراضياً `true`؛ يُمكن تعطيله لإخفائه من قوائم الاستلام |

### الإجراءات

| الإجراء | المسار | الصلاحية | ملاحظة |
|---|---|---|---|
| **إضافة** | `POST /api/warehouses` | ADMIN / WAREHOUSE | |
| **تعديل** | `PUT /api/warehouses/[id]` | ADMIN / WAREHOUSE | يتضمن تفعيل/تعطيل |
| **حذف** | `DELETE /api/warehouses/[id]` | **ADMIN فقط** | يُرفض إن كان فيه مخزون (`has-stock`، 409) |

### ربط المستخدم بالمخزن / المخزن الافتراضي

- لكل مستخدم مخزن محدد عبر `resolveWarehouseId(user, ...)` في الخادوم.
- إن لم يكن للمستخدم مخزن، يلجأ النظام إلى المخزن الافتراضي (الأول النشط) عبر `getDefaultWarehouseId()`.
- هذا التحديد يُستخدم في: استلام فواتير الشراء، إرجاع المرتجعات، الجرد، التحويلات، وإعادة مخزون المرتجعات.

> **لم يظهر في الكود بشكل صريح:** شاشة لربط مستخدم بمخزن محدد. يبدو أن الربط يتم في إعدادات المستخدم أو عبر المخزن الافتراضي للنظام.

### الكميات لكل مخزن

- جدول `StockItem` يحمل `(productId, warehouseId, quantity)` كمفتاح فريد.
- حقل `Product.quantity` هو **مشتق** (مجموع كل `StockItem` لنفس المنتج) ويُعاد حسابه عبر `updateProductQuantityFromStockItems` بعد كل عملية تؤثر على المخزون.

## 7.6 حركات المخزون (Stock Movements)

الملف: `src/components/inventory/stock-movement-report.tsx` و`src/app/api/inventory/movements/route.ts`.

### ما هي حركة المخزون؟

كل حدث في النظام يُغيّر كمية صنف ما في مخزن ما يُسجَّل ويظهر في تقرير الحركات. النظام **يوحِّد** كل الأحداث في مجرى واحد (Unified Stream) بالشكل:

```
{ date, type, productId, productName, quantityChange, referenceNo, userName }
```

### أنواع الحركات

| الرمز | المصدر | أثر الكمية |
|---|---|---|
| `SALE` | `SaleItem` | سالب (بيع) |
| `REFUND` | `SaleItem.returnedQty` | موجب (مرتجع عميل) |
| `EXCHANGE` | `ExchangeLine` | موجب/سالب حسب الإشارة |
| `PURCHASE_INVOICE` | `PurchaseInvoiceItem` (POSTED فقط) | موجب (استلام بضاعة) |
| `PURCHASE_RETURN` | `PurchaseReturnItem` | سالب (مرتجع لمورد) |
| `TRANSFER_OUT` | `StockTransferItem` من المخزن المصدر | سالب |
| `TRANSFER_IN` | `StockTransferItem` للمخزن الوجهة | موجب |
| `STOCK_TAKE` | `StockTakeItem.variance` (APPROVED فقط) | موجب/سالب حسب الفرق |
| `SPOT_CHECK` | `SpotCheck.variance` | موجب/سالب (للعرض فقط — لا يُعدّل المخزون) |

> **ملاحظة:** الجرد العشوائي (Spot Check) يُسجَّل فرقه في التقرير، لكنه **لا يُعدّل** المخزون فعلياً (يُستخدم للتدقيق فقط).

### الفلاتر

| الفلتر | الشرح |
|---|---|
| المنتج | تقتصر الحركات على صنف واحد |
| المخزن | تُستثنى المبيعات والتبديل والعشوائي (ليس لها مخزن) |
| من تاريخ / إلى تاريخ | على الحقل الزمني الطبيعي لكل حدث |
| النوع | نوع حركة واحد |
| زر «تحميل» | يستدعي `GET /api/inventory/movements` |

### تصدير CSV

زر «تصدير CSV» يُولّد ملف UTF-8 (مع BOM لتفادي تشويه العربية في Excel) بأعمدة: التاريخ، النوع، المنتج، التغيّر، المرجع، المستخدم.

## 7.7 الجرد الكامل (Stock Take)

الملفات: `src/components/inventory/stock-take-tab.tsx` و`src/components/stocktake/stocktake-view.tsx`، و`src/app/api/stock-takes/route.ts` و`[id]/approve/route.ts`.

### ما هو الجرد الكامل؟

عملية جرد شاملة للمخزون يقوم بها الموظفون دورياً (شهرياً، سنوياً) لمطابقة الكميات الدفترية مع الكميات الفعلية على الرفوف، وتسوية الفروقات محاسبياً.

### مراحل الجرد

| المرحلة | الإجراء |
|---|---|
| **1. طباعة كشف الجرد** | زر «طباعة كشف الجرد» → نافذة HTML بنموذج A4 يحتوي: #، الصنف، الباركود، الرصيد بالنظام، خانة فارغة للعد الفعلي، خانة الفرق |
| **2. عدّ فعلي** | يُكتب العدد الفعلي على الورق |
| **3. إدخال الأرقام** | يُختار المخزن + الملاحظة، يُضاف كل صنف بالباركود/الاسم، تُدخل الكمية الفعلية في حقل `actualQty` |
| **4. حفظ كمسودة (اختياري)** | زر «حفظ كمسودة» — لأغراض الجرد الكبير الذي يأخذ أياماً |
| **5. إنشاء + اعتماد** | زر «جرد جديد (اعتماد نهائي)» → يُنشئ الجرد بحالة DRAFT، ثم تُفتح نافذة تأكيد الاعتماد |

### الأعمدة في جدول الجرد

| العمود | الشرح |
|---|---|
| الصنف | اسم المنتج |
| دفتري | `systemQty` (الكمية في النظام وقت الإنشاء) |
| فعلي | `actualQty` (يدخله المستخدم) |
| الفرق | `variance = actualQty − systemQty` |
| قيمة الفرق | `varianceValue = variance × unitCost` |
| حذف | لإزالة الصنف من الجرد |

### ملخص جانبي (Live)

يعرض النظام أثناء الإدخال:
- **عجز** (Shortage): مجموع القيم السالبة (الكمية الفعلية أقل من الدفترية).
- **فائض** (Surplus): مجموع القيم الموجبة (الكمية الفعلية أكبر من الدفترية).

### الأثر على المخزون (عند الاعتماد)

ملف `src/app/api/stock-takes/[id]/approve/route.ts`:

1. يجب أن يكون للجرد `warehouseId` (إلا يُرفض: `warehouse-required`).
2. لكل صنف بفرق ≠ 0:
   - **فائض:** `incrementStockItem` (+variance) ثم `updateProductQuantityFromStockItems`.
   - **عجز:** `decrementStockItem` (−|variance|) — يُرفض إن لم يكن هناك مخزون كافٍ (`stock-insufficient:...`).
3. يُحدَّث الجرد إلى `status=APPROVED` + `approvedById` + `approvedAt`.
4. **قيد محاسبي داخل المعاملة** (فشل القيد → التراجع الكامل):
   - عجز: مدين 5070 (عجز المخزون) / دائن 1010 (المخزون).
   - فائض: مدين 1010 (المخزون) / دائن 4060 (إيرادات تسوية مخزنية).
5. سجل تدقيق `STOCK_TAKE_APPROVED`.

> **الجرد الأعمى (Blind):** يُمكن إجراء الجرد الأعمى عبر شاشة `stocktake-view.tsx` التي تعرض نفس الآلية لكنها تُظهر الكمية الدفترية (ليست عمياء تماماً في النسخة الحالية).

### قائمة الجرد السابقة

تعرض الجداول السابقة بـ: `takeNo` (مثل `ST-00001`، `ST-00002`)، الحالة (DRAFT/APPROVED)، العجز، الفائض، التاريخ، وزر «اعتماد» للمسودات.

## 7.8 الجرد العشوائي (Spot Check)

الملفات: `src/components/spotcheck/spotcheck-view.tsx` و`src/app/api/spot-checks/route.ts`.

### الفرق بينه وبين الجرد الكامل

| الجانب | الجرد الكامل (Stock Take) | الجرد العشوائي (Spot Check) |
|---|---|---|
| **النطاق** | عدة أصناف دفعة واحدة | صنف واحد في كل مرة |
| **الأسلوب** | مخطط ومجدول | مفاجئ وسريع |
| **رؤية الكمية الدفترية** | تُعرض عند الإدخال | **مخفية** (Blind) — لا تُكشف إلا بعد الإدخال |
| **الأثر على المخزون** | يُعدّل الكميات عند الاعتماد | **لا يُعدّل** الكميات |
| **القيد المحاسبي** | يُنشأ قيد للعجز/الفائض | **لا يُنشأ** قيد محاسبي |
| **الهدف** | تسوية دورية | كشف تلاعب أو خطأ سريع |
| **الرقم المرجعي** | `ST-XXXXX` | `SC-{آخر 6 خانات من المعرف}` |

### كيف يعمل الجرد العشوائي الأعمى؟

1. يبحث المشرف عن صنف بالاسم/الباركود.
2. **لا يُعرض الكمية الدفترية** (`bookQtyHidden`).
3. يُدخل الكمية الفعلية المعدودة على الرف.
4. عند الحفظ:
   - الخادوم يقرأ `bookQty` من `StockItem` للمخزن الافتراضي (أو مخزن مُمرَّر).
   - يحسب `variance = countedQty − bookQty`.
   - يحفظ السجل ويعيد النتيجة للمشرف.
5. تظهر النتيجة فوراً ببطاقة ملوّنة (أخضر=مطابق، أحمر=عجز، أزرق=فائض).

### سجل العمليات السابقة

جدول يعرض آخر 100 جرد عشوائي: الصنف، الباركود، الدفتري، الفعلي، الفرق، المستخدم، التاريخ.

> **لم يظهر في الكود:** خيار لربط العشوائي بمخزن محدد من الواجهة (الـ API يقبل `warehouseId` لكن النموذج لا يعرض القائمة).

## 7.9 التحويلات بين المخازن (Stock Transfers)

الملفات: `src/components/inventory/stock-transfer-tab.tsx` و`src/components/transfers/transfers-view.tsx`، و`src/app/api/stock-transfers/route.ts` و`[id]/receive/route.ts`.

### ما هو التحويل؟

نقل كميات من صنف أو أكثر من مخزن إلى مخزن آخر. مثال: نقل علب عطر من المخزن الرئيسي إلى فرع السالمية.

### خطوات التحويل

| الخطوة | الإجراء |
|---|---|
| 1 | اختيار «من مخزن» (المصدر) و«إلى مخزن» (الوجهة) — يجب أن يكونا مختلفين (`same-warehouse`). |
| 2 | إضافة الأصناف بالباركود أو البحث بالاسم. |
| 3 | تحديد الكمية المراد نقلها لكل صنف (يجب ألا تتجاوز المتاح في المصدر: `exceeds-available:...`). |
| 4 | (اختياري) ملاحظة. |
| 5 | الضغط على «إنشاء تحويل (إرسال)» → `POST /api/stock-transfers` |

### ماذا يحدث عند الإنشاء (OUT)؟

في معاملة واحدة:

1. التحقق من وجود المخزنين واختلافهما.
2. لكل صنف: قراءة `StockItem.quantity` في المخزن المصدر، التحقق أن الكمية المطلوبة ≤ المتاح.
3. `decrement` على `StockItem` المصدر (upsert إن لم يوجد سجل).
4. إنشاء سجل `StockTransfer` بحالة `OUT` ورقم `TR-XXXXX`.
5. سجل تدقيق `STOCK_TRANSFER_CREATED`.
6. **بعد المعاملة** (post-commit): إعادة حساب `Product.quantity` لكل الأصناف المتأثرة.

> في هذه المرحلة البضاعة «بالطريق» — خُصمت من المصدر ولم تُضف للوجهة بعد.

### الاستلام (RECEIVED)

الملف: `src/app/api/stock-transfers/[id]/receive/route.ts`.

زر «استلام» يظهر للتحويلات بحالة `OUT`. عند الضغط:

1. التحقق أن الحالة `OUT` (ليست `RECEIVED` بالفعل → `already-received`).
2. لكل صنف: `increment` على `StockItem` في المخزن الوجهة (upsert).
3. تحديث التحويل: `status=RECEIVED` + `receivedById` + `receivedAt`.
4. سجل تدقيق `STOCK_TRANSFER_RECEIVED`.
5. إعادة حساب `Product.quantity`.

### حالة الإلغاء

> **لم يظهر في الكود:** مسار صريح لإلغاء تحويل قائم بحالة `OUT` (لا يوجد `POST /api/stock-transfers/[id]/cancel`). الشارة `CANCELLED` معرَّفة في الواجهة لكن لا يوجد زر لإلغاء التحويل. الحل العملي: استلام التحويل ثم عمل تحويل عكسي.

### جدول سجل التحويلات

يعرض: رقم التحويل، من ← إلى، عدد الأصناف، القيمة الإجمالية (بالتكلفة)، الحالة (شارة ملوّنة)، زر «استلام» (لـ OUT فقط).

## خلاصة

- شاشة المخزون تضم 5 تبويبات: المنتجات، المخازن، الجرد، التحويلات، حركات المخزون.
- نموذج المنتج **مبسّط** (الاسم/الباركود/الفئة/الوحدة/الصورة)، أما الأسعار وحدود الطلب فتُحرَّر من شاشة التسعير.
- الفئات شجرية أب-ابن، والباركود يُولَّد تلقائياً ببادئة الفئة.
- الوحدات يديرها ADMIN فقط (إضافة/حذف، دون تعديل).
- المخازن تدعم التفعيل/التعطيل، ولا تُحذف إن كان فيها مخزون.
- حركات المخزون تُوحِّد 9 أنواع أحداث في تقرير واحد قابل للتصدير CSV.
- الجرد الكامل يُعدّل المخزون ويُنشأ قيود محاسبية (5070/1010/4060).
- الجرد العشوائي **أعمى** (لا يُظهر الدفتري)، لا يُعدّل المخزون، ويُستخدم للتدقيق السريع.
- التحويلات تمر بمرحلتين: `OUT` (خصم من المصدر) ثم `RECEIVED` (إضافة للوجهة).

الخطوة التالية في التوثيق: وحدة المشتريات (الأمر، الفاتورة، المرتجع) في `06-purchases.md`.
