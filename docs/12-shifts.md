# 12 — ورديات الكاشير (Shifts)

> دليل تدريبي مبسّط لوحدة الورديات في نظام KWPOS. يغطي فتح الوردية، إغلاقها مع المطابقة، المتوقع مقابل المعدود، أنواع الفروقات (Cash / K-Net / Visa)، الإغلاق الأعمى (Blind Close) واعتماد المدير، تأثير الوردية على المبيعات، والقيد المحاسبي الناتج عن العجز/الفائض. مكتوب لغير المختصين تقنياً مع الإشارة للملفات المصدرية.

---

## 1. نظرة عامة

الوحدة تُدار من ملف واحد رئيسي: `src/components/shift/shift-view.tsx`. الـ APIs:

| المسار | الطريقة | الوصف |
|---|---|---|
| `/api/shifts` | GET | قائمة الورديات (آخر 50) |
| `/api/shifts` | POST | فتح وردية جديدة |
| `/api/shifts` | PATCH | إغلاق وردية (المسار الرئيسي) |
| `/api/shifts/current` | GET | وردية المستخدم الحالي المفتوحة |
| `/api/shifts/[id]/close` | POST | إغلاق أعمى (Blind Close) |
| `/api/shifts/[id]/approve` | POST | اعتماد مدير لوردية مُغلَقة عمياً |

> **ملاحظة مهمة:** يوجد في النظام **مسارا إغلاق**:
> 1. **الإغلاق العادي** عبر `PATCH /api/shifts` (المُستخدَم في `shift-view.tsx`) — يطلب countedCash/Knet/Visa ويرى الكاشير المتوقع.
> 2. **الإغلاق الأعمى** عبر `POST /api/shifts/[id]/close` ثم `POST /api/shifts/[id]/approve` (موجود في الـ API لكن **لم يظهر في الكود كشاشة مستخدمة**). الإغلاق الأعمى مضاد للغش: يُدخِل الكاشير المعدود دون رؤية المتوقع.

## 3. فتح وردية (Open Shift)

الزر: زر **«فتح وردية جديدة»** أعلى شاشة الورديات.

| الخطوة | التفصيل |
|---|---|
| 1 | الضغط على زر «فتح وردية» في `shift-view.tsx`. |
| 2 | الـ API `POST /api/shifts` يتحقق إن كان **المستخدم الحالي** له وردية مفتوحة (`status = OPEN` لنفس `userId`). إذا نعم: `409 user-shift-already-open`. |
| 3 | إن لم تكن له وردية مفتوحة، يُنشَأ سجل Shift جديد برقم `SHF-XXXX` (XXXX = عدد الورديات + 1، مع padding لـ4 خانات). |
| 4 | `status = OPEN`, `openedAt = now()`, `openingBalance = body.openingBalance` (أو 0 إن لم يُرسَل). |
| 5 | يُكتَب حدث تدقيق: `SHIFT_OPENED` بوصف: «فتح وردية SHF-XXXX — رصيد افتتاحي: XXX». |

> **يدعم النظام ورديات متعددة:** كل كاشير له وردية مفتوحة مستقلة. الورديات لا تتقاطع. عدة كاشيرات قد يعملون في وقت واحد.

### رصيد افتتاحي (Opening Balance / Float)

يُدخِل الكاشير الرصيد الافتتاحي (النقدية في الدرج عند بدء الوردية). هذا الرقم **يُضاف** إلى النقدية المتوقعة عند الإغلاق:

```
expectedCash = openingBalance + cashSales − cashRefunds
cashVariance = countedCash − expectedCash
```

لو لم يُدخِل الكاشير رصيداً، يُعتبَر 0، وتكون النقدية المتوقعة = مبيعات النقدية فقط.

## 4. إغلاق الوردية (Close Shift) — المسار الرئيسي

الشاشة: `CloseShiftForm` في `shift-view.tsx`. الـ API: `PATCH /api/shifts`.

### الحقول في شاشة الإغلاق

لكل وسيلة دفع (نقدية، K-Net، Visa) عمود منفصل يعرض:

| العنصر | الوصف |
|---|---|
| **المتوقع (دفترياً)** | من الخادم: مجموع مبيعات الكاشير بهذه الوسيلة خلال الوردية |
| **المعدود (من الجهاز)** | حقل إدخال يدوي — يُدخِل الكاشير الرقم الفعلي |
| **الفرق** | معدود − متوقع (يُحسَب فورياً في الواجهة) |
| ✓ علامة توازن خضراء | إذا الفرق = 0 والمعدود > 0 |

### حقول إضافية

- **ملاحظة** (اختيارية): نص حر يُحفَظ مع الوردية.
- زر **«إغلاق الوردية والمطابقة»** (أحمر) يُرسِل: `id`, `countedCash`, `countedKnet`, `countedVisa`, `note`.

### حساب الفروقات (في `PATCH /api/shifts`)

| الفرق | المعادلة |
|---|---|
| `expectedCash` | مجموع `total − refundTotal` لكل مبيعات الكاشير بـ `paymentMethod = CASH` خلال [openedAt, now] |
| `expectedKnet` | نفسه لـ `paymentMethod = CARD` |
| `expectedVisa` | نفسه لـ `paymentMethod = TRANSFER` |
| `cashVariance` | `countedCash − (openingBalance + expectedCash)` |
| `knetVariance` | `countedKnet − expectedKnet` |
| `visaVariance` | `countedVisa − expectedVisa` |

> **النقدية تشمل الرصيد الافتتاحي**، أما K-Net و Visa فلا (لأن الرصيد الافتتاحي نقد فقط).

### تحذير فرق الدفع الإلكتروني

إذا `knetVariance` أو `visaVariance` ≠ 0، تظهر لافتة كهرمانية تحذر من أن **فروقات الدفع الإلكتروني غالباً تشير لخطأ تسجيل** (نظراً لأن أرصدة K-Net/Visا تأتي من البنك تلقائياً). يُنصَح بالتحقق من:
- هل فاتورة بـ `TRANSFER` سُجِّلت بـ `CARD` بالخطأ؟
- هل المرتجعات نُسيت تسجيلها؟

### القيد المحاسبي الناتج عن فرق النقدية

في نفس معاملة `PATCH`، إذا `|cashVariance| > 0.001`:

| الحالة | القيد |
|---|---|
| **عجز** (`cashVariance < 0`) | مدين: 5070 (مصروف عجز) / دائن: 1010 (النقدية) |
| **فائض** (`cashVariance > 0`) | مدين: 1010 (النقدية) / دائن: 4060 (إيراد فائض) |

> **ملاحظة:** `5070` و `4060` غير موجودة في `seed/route.ts`. ستفشل محاولة إنشاء القيد إن لم تكن موجودة يدوياً. القيد يُنشَأ عبر `safeCreateJournalEntry` (يسجّل warning بدل رمي خطأ)، فلا تتعطل عملية الإغلاق.

### حدث التدقيق

يُكتَب `SHIFT_CLOSED` بوصف: «إغلاق وردية SHF-XXXX — عجز/فائض نقدية: ±X.XXX».

## 5. الإغلاق الأعمى (Blind Close) واعتماد المدير

> **لم يظهر في الكود كشاشة مستخدمة** في `shift-view.tsx`. الـ APIs موجودة لكن لم تُربَط بزر ظاهر.

### المسار 1: الإغلاق الأعمى

الـ API: `POST /api/shifts/[id]/close`. الجسم: `{ actualCash, actualCard }`.

| الخطوة | التفصيل |
|---|---|
| 1 | الكاشير يُدخِل المعدود (نقد + بطاقة) **دون رؤية المتوقع** (مضاد للغش). |
| 2 | الخادم يحسب `expectedCashSales`, `expectedCardSales`, `expectedTransferSales` من مبيعات الكاشير منذ `openedAt`. |
| 3 | يستثني المبيعات المُسترجَعة كلياً (`refundStatus = FULL`)، ويخصم المرتجعات الجزئية. |
| 4 | `expectedCash = openingFloat + expectedCashSales`. |
| 5 | `cashVariance = actualCash − expectedCash`. |
| 6 | يُحدَّث `status = BLIND_CLOSED`, `blindClosedAt = now()`. |

### المسار 2: اعتماد المدير

الـ API: `POST /api/shifts/[id]/approve`. **ADMIN فقط.**

| الخطوة | التفصيل |
|---|---|
| 1 | يتحقق أن `status = BLIND_CLOSED`. |
| 2 | يُحدِّث: `status = APPROVED`, `approvedAt = now()`, `approvedBy = user.id`. |
| 3 | يُنشِئ **3 قيود** (داخل try/catch، لا تفشل العملية لو فشل القيد): |
| | **(أ) قيد ترحيل مبيعات الوردية:** مدين 1010 (النقدية، باستثناء float) + 1020 (K-Net + تحويلات) / دائن 4010 (إيراد مبيعات POS). |
| | **(ب) قيد عجز** (لو `cashVariance < 0`): مدين 2020 (ذمم الموظفين) / دائن 1010. |
| | **(ج) قيد فائض** (لو `cashVariance > 0`): مدين 1010 / دائن 4050 (إيرادات أخرى). |

> الحسابات `2020` و `4050` تُنشَأ تلقائياً لو لم تكن موجودة (`ensureAccount` في نفس الملف).

## 6. كل زر في شاشة الورديات

| الزر | متى يظهر | وظيفته |
|---|---|---|
| **«فتح وردية جديدة»** (Play) | في رأس الصفحة، فقط إن لا توجد وردية مفتوحة للمستخدم | استدعاء `POST /api/shifts` |
| **«فتح وردية»** (Play) | في حالة عدم وجود وردية مفتوحة (EmptyState) | نفس ما سبق |
| **إدخال «المعدود من الجهاز»** | 3 حقول (نقد، K-Net، Visa) داخل `CloseShiftForm` | إدخال يدوي |
| **«إغلاق الوردية والمطابقة»** (Square، أحمر) | داخل `CloseShiftForm` | استدعاء `PATCH /api/shifts` |
| **«إعادة المحاولة»** | عند فشل تحميل القائمة | `refetch()` |
| **«↻» (تحديث)** | ضمن `refetchInterval` في لوحة المدير فقط | تحديث تلقائي كل 60 ثانية |

> لم تظهر أزرار لـ «الإغلاق الأعمى» أو «الاعتماد» في شاشة الورديات الحالية. هذه العمليات متاحة عبر API فقط.

## 7. علاقة الورديات بالمبيعات

| السؤال | الجواب |
|---|---|
| هل يجب فتح وردية قبل البيع؟ | **لا**. البيع ممكن دون وردية، لكن لا يمكن مطابقة النقدية لاحقاً. |
| كيف تُربَط مبيعات بوردية؟ | عبر `sale.userId` و`sale.createdAt`: مبيعات الكاشير بين `openedAt` و`closedAt` تُنسَب للوردية. |
| ماذا عن مرتجع بعد إغلاق الوردية؟ | يُحتسَب ضمن مبيعات الوردية الجديدة (لو كان الكاشير نفسه فتح وردية جديدة). لو الكاشير مختلف، يُحتسَب ضمن مبيعات الكاشير الجديد. |
| هل تؤثر الوردية على المخزون؟ | لا. المخزون يُخصَم مباشرة عند البيع، مستقلاً عن حالة الوردية. |
| هل يمكن فتح ورديتين متداخلتين؟ | **لا** لنفس المستخدم. لكن يمكن لكاشيرين مختلفين أن يعملا في وقت واحد بورديتين مختلفتين. |
| من يفتح الوردية؟ | أي مستخدم مسجَّل دخوله (لا يوجد قيد على الدور في `POST`). |
| من يُغلق الوردية؟ | في الإغلاق العادي: الكاشير نفسه (لا يوجد فحص في `PATCH`). في الإغلاق الأعمى: الكاشير نفسه أو ADMIN. في الاعتماد: ADMIN فقط. |

## 8. حالات الوردية (Status)

| الحالة | المعنى | الانتقال التالي |
|---|---|---|
| `OPEN` | وردية جارية، يمكن تسجيل مبيعات خلالها | إغلاق عادي → `CLOSED` / إغلاق أعمى → `BLIND_CLOSED` |
| `CLOSED` | أُغلقت بالمسار العادي مع مطابقة | لا انتقال (نهائية) |
| `BLIND_CLOSED` | أُغلقت عمياً، بانتظار الاعتماد | `POST /approve` → `APPROVED` |
| `APPROVED` | اعتمدها المدير، أُنشئت قيود الترحيل | لا انتقال (نهائية) |

> **في `shift-view.tsx` الحالية:** الورديات المفتوحة تظهر في `CloseShiftForm`، والمُغلَقة (CLOSED) تظهر في جدول «الورديات المغلقة». حالتا `BLIND_CLOSED` و `APPROVED` **لم تظهر لها معالجة بصرية خاصة** في الشاشة.

## 9. جدول الورديات المغلقة

يعرض آخر ورديات بـ `status = CLOSED`. كل صف:

| العمود | الوصف |
|---|---|
| رقم الوردية | `shiftNo` بخط Mono |
| الفترة | من `openedAt` إلى `closedAt` |
| النقدية | `countedCash` + شارة فرق (✓ إن 0، −X إن سالب، +X إن موجب) |
| K-Net | `countedKnet` + شارة فرق |
| Visa | `countedVisa` + شارة فرق |

## 10. لوحة المدير والورديات

في `src/components/dashboard/manager-dashboard-view.tsx` تعرض بطاقة «الورديات النشطة»:
- عدد الورديات المفتوحة (`status = OPEN`).
- أسماء أول كاشيرين اثنين.
- بطاقة تفصيلية: كل كاشير، حرف أول من الاسم في دائرة، وقت فتح الوردية، الرصيد الافتتاحي.
- الضغط على البطاقة يفتح شاشة الورديات.

تتجدد البيانات كل 60 ثانية تلقائياً (`refetchInterval`).

## 11. نصائح عملية للكاشير والمدير

1. **افتح الوردية بداية كل يوم** برصيد افتتاحي صحيح (يُفضَّل عده فعلياً).
2. **سجِّل كل عملية على الوسيلة الصحيحة** — K-Net (CARD) ≠ Visa (TRANSFER). خطأ التصنيف يسبب فروقات وهمية.
3. **اعمل مرتجعات في نفس الوردية** ما أمكن، ليُخصم من المتوقع تلقائياً.
4. **عند فرق نقد كبير:** تحقق من فاتورة لم تُسجَّل، أو نقدية مفقودة، قبل اعتماد الإغلاق.
5. **فروقات K-Net/Visa شبه مستحيلة** — إن ظهرت، فهي خطأ تصنيف وسيلة دفع، راجع الفواتير الأخيرة.
6. **في حال العجز المتكرر:** فكِّر في تفعيل الإغلاق الأعمى (Blind Close) لمنع الكاشير من رؤية المتوقع وتعديل المعدود.
7. **الحسابات 5070 و 4060** (لعجز/فائض الورديات) غير موجودة في `seed/route.ts` الافتراضي. يجب إنشاؤها يدوياً لتسجيل قيد الفرق، وإلا سيُسجَّل warning فقط.
