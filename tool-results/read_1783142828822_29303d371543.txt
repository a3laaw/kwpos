   244→  `tx.sale.create()`. The session JWT held the user id from BEFORE a re-seed,
   245→  but re-seeding deleted + recreated users with NEW cuid() ids — so the sale's
   246→  `userId` FK pointed at a non-existent user.

   250→   fixed ids (`user-admin-demo`, `user-sales-demo`, `user-warehouse-demo`) instead
   251→   of random cuids. After a re-seed, the existing session JWT still references a
   252→   valid user id → sales work without re-login.
   253→2. **Defensive session check in sales route** (`src/app/api/sales/route.ts`):
   254→   before the transaction, `db.user.findUnique({ where: { id: user.id } })`.
   255→   If the user no longer exists (truly stale session), return
   256→   `{ error: "session-expired" }` (401) instead of a cryptic FK error.
   257→3. **Frontend auto-logout on session-expired** (`src/components/sales/sales-view.tsx`):
   258→   the checkout catch block detects `session-expired`, shows a toast, and calls
