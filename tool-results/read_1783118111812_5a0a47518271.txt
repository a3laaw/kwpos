   105→
   106→Root cause:
   107→- Zustand `persist` middleware hydrated the saved `view` (e.g. "inventory") from
   108→  localStorage SYNCHRONOUSLY during store creation on the client.
   109→- Server render used the default `view` ("dashboard") since localStorage is
   110→  unavailable server-side.
   111→- => client first render rendered a different component tree than the server
   112→  HTML, which shifted every subsequent Radix `useId` identifier
   113→  (radix-_R_27atplb_ vs radix-_R_8uatplb_) → React hydration mismatch error
   114→  on the Topbar DropdownMenu trigger.
   115→
   116→Fix:
   117→1. `src/lib/store.ts`: added `skipHydration: true` to the persist config so the
   118→   store starts with the default `view` on both server and client (matching
   119→   first render). Removed `user`/`setUser` from the store entirely.
   120→2. `src/components/app-shell.tsx`: call `useAppStore.persist.rehydrate()` inside
   121→   a `useEffect` so the persisted `view` is applied AFTER hydration completes
   122→   (safe post-hydration update, no mismatch).
   123→3. `src/components/user-context.tsx` (new): `UserProvider` + `useUser()` React
   124→   context populated from the server-rendered `user` prop. This gives views
   125→   synchronous access to the authenticated user (no `useEffect`/flash) and
   126→   removes the need to mirror the user in Zustand.
   127→4. Updated `inventory-view`, `purchases-view`, `suppliers-view` to use
   128→   `useUser()` instead of `useAppStore((s) => s.user)`; `user.role` is now
