### 🛠️ المكوّنات التقنية

| الطبقة | التقنية | السبب |
|---|---|---|
| **Frontend** | Next.js 16 + TypeScript + shadcn/ui | أحدث React Server Components · أمان أنواع |
| **Backend** | API Routes + Prisma ORM | RESTful · استعلامات آمنة · migration |
| **Database** | Supabase Postgres (Sydney) | Postgres مُدار · نفس منطقة Vercel |
| **Hosting** | Vercel (Edge Network) | نشر تلقائي من GitHub · CDN عالمي |
| **Real-time** | Socket.io mini-service | طباعة فورية · إشعارات حية |
| **Auth** | NextAuth.js + JWT (7d) + bcrypt | جلسات آمنة · تشفير كلمات المرور |

> 📝 **ملاحظات للمحاضر:** اشرح أن المعمارية Serverless توفّر تكلفة التشغيل (لا خوادم لإدارتها)، وأن اختيار منطقة Sydney لكل من Vercel وSupabase يقلّل زمن الاستجابة (latency) لأقل من 50ms داخل الكويت. اذكر أن Prisma يحمي من SQL Injection، وأن PgBouncer مفعّل للتعامل مع تزامن الاتصالات.

# 🛒 نقاط البيع POS — قياسي وسريع

<hr/>

<p align="center">
  <em>وضعان متكاملان: القياسي للكاشير المحترف · السريع لذروة الزحام.</em>
</p>

### ⚖️ مقارنة الوضعين

| 🎯 الميزة | 🟦 القياسي (Standard) | 🟩 السريع (Express) |
|---|---|---|
| **الحجم** | لوحة كاملة + سلة + عميل | لوحة مصغّرة + سلة فقط |
| **السكان** | الكاشير المحترف | ذروة الزحام (موسم رمضان/الأعياد) |
| **الباركود** | ✅ + بحث نصّي + فئة | ✅ فقط |
| **ربط العميل** | بالكامل (هاتف/اسم/بطاقة) | اختياري (افتراضي: عميل نقدي) |
| **تعدد الدفع** | نقدي + K-Net + Visa + ولاء | نقدي بشكل أساسي |
| **التعليق** | ✅ تعليق واستئناف | ❌ |
| **الطباعة** | حرارية 58mm + A4 | حرارية فقط |
| **اختصارات لوحة المفاتيح** | ✅ كاملة | ✅ مختصرة |

### ⌨️ اختصارات لوحة المفاتيح

| المفتاح | الوظيفة |
|---|---|
| `F2` | تركيز حقل البحث/الباركود |
| `Ctrl+Enter` | إتمام البيع فورًا |
| `Esc` | إلغاء/إغلاق النافذة |
| `F9` | ربط العميل بالهاتف |
| `F12` | تعليق الفاتورة الحالية |

### 📷 قارئ الباركود الشامل
- ✅ متوافق مع كل قارئات USB/Bluetooth
- ✅ يحوّل سلسلة الأرقام + Enter تلقائيًّا إلى بحث منتج
- ✅ يدعم باركود المنتجات (EAN-13) + التركيبات (prefix=9) + المولّد تلقائيًّا

> 📝 **ملاحظات للمحاضر:** أبرز أن وضع Express صُمِّم بعد ملاحظة أن 80% من المبيعات اليومية بضاعة واحدة نقدًا — فلا داعي لإظهار كل القوائم. اختصارات لوحة المفاتيح ترفع إنتاجية الكاشير 3x. اذكر أن قارئ الباركود يعمل في الوضعين دون إعداد إضافي.

| 7 | `ADJUSTMENT` | تسوية جرد | ➕/➖ حسب الفرق |
| 8 | `PRODUCTION` | إنتاج تركيبة | ➖ مكونات / ➕ ناتج |
| 9 | `DAMAGE` | تالف/مفقود | ➖ نقصان |

### ♻️ الحذف الناعم + الاسترجاع
- 🗂️ كل حركة تُحذف ناعمًا (soft delete) — تختفي من القوائم لكنها تبقى في DB
- ↩️ استرجاع فوري بضغطة زر مع قيد عكسي تلقائي
- 🛡️ لا يمكن حذف حركة قبل 90 يومًا (سياسة استباقية)

> 📝 **ملاحظات للمحاضر:** الفائدة الكبرى: مالك الفروع يرى فجأة فرعًا به مخزون من صنف نافد في فرع آخر — فينشئ تحويلًا فورًا بدل أن يخسر مبيعات. اشرح أن الخصم متعدد المخازن يعني: لو اشترى العميل 5 قطع، والفرع الرئيسي فيه 3 فقط، النظام يخصم 3 من الرئيسي و2 من أقرب فرع تلقائيًّا.

# 📋 الجرد الكامل والأعمى (Stock Take)

<hr/>

<p align="center">
  <em>جرد دقيق بدون توقف العمليات — مع خيار «الأعمى» لمنع التلاعب.</em>
</p>

### 🔁 دورة الجرد

| المرحلة | الإجراء | الحالة |
|---|---|---|
| **1. الإنشاء** | بدء جرد جديد → تحميل كل الأصناف تلقائيًّا | `DRAFT` |
| **2. العدّ** | إدخال الكمية الفعلية لكل صنف | `IN_PROGRESS` |
| **3. الحفظ** | حفظ تقدم (PUT على نفس المسودة) | `IN_PROGRESS` |
| **4. الاستئناف** | فتح المسودة السابقة ومتابعة العد | `IN_PROGRESS` |
| **5. الاعتماد** | موافقة المدير → قيد محاسبي | `APPROVED` |

### 🙈 الجرد الأعمى (Blind Count)

| الميزة | الوصف |
|---|---|
| **إخفاء الرصيد الدفتري** | لا يظهر للعّاد — يرى فقط: اسم الصنف، الباركود، حقل فارغ |
| **الهدف** | منع «تسوية العدّ» للرصيد الدفتري بدل العدّ الفعلي |
| **من يفعّله؟** | المدير عند بدء الجرد |
| **تقرير أعمى قابل للطباعة** | قائمة بالأصناف والكميات المدخلة فقط |

### 🧮 قيد التسوية المحاسبي
- الفروقات تُرحَّل تلقائيًّا إلى حساب **«مخزون البضائع»** مقابل **«أرباح/خسائر جرد»**
- ➕ زيادة فعلية = قيد مدين للمخزون
- ➖ نقصان فعلي = قيد دائن للمخزون (خسارة جرد)

### 💾 حفظ التقدم
- ✅ كل حفظ هو `PUT` على نفس المسودة (لا تُنشئ نسخًا)
- ✅ يمكن لعدّة موظفين العدّ في فرق مختلفة (ميزة قادمة)
- ✅ لا حاجة لإغلاق الفرع — الجرد متوازٍ مع البيع

> 📝 **ملاحظات للمحاضر:** الجرد الأعمى أداة قوية لكشف سرقات الموظفين: لو كان العدّ «عاديًّا»، يميل الموظف لكتابة الرصيد الدفتري حتى لو كان الفعلي أقل. مع الأعمى، الفرق يظهر بصدق. اشرح أن الجرد لا يوقف البيع — البائع يبيع، والعارّ يعدّ، والنظام يدير التزامن.

# 🛃 ملحق التخليص الجمركي — التكلفة الحقيقية (Landed Cost)

<hr/>

<p align="center">
  <em>لا تعرف ربحك الحقيقي إلا بعد احتساب الجمارك والشحن والضرائب على الواردات.</em>
</p>

### ⏰ متى يُنشأ الملحق؟

| الشرط | الحالة |
|---|---|
| ✅ فاتورة الشراء مُرحَّلة (POSTED) | إلزامي |
| ✅ البضاعة وصلت فعليًّا للمخزن | إلزامي |
| ⛔ قبل أيٍّ منهما | غير مسموح |

### 📊 نسب التكلفة الإضافية

| # | 🏷️ المكوّن | 📐 القاعدة | 🧮 طريقة الحساب |
|---|---|---|---|
| 1 | 🛃 **الجمارك** | % من قيمة الفاتورة | `subtotal × customs%` |
| 2 | 💰 **الضريبة** | % بعد الجمارك | `(subtotal + customs) × tax%` |
| 3 | 🚚 **الشحن** | % من قيمة الفاتورة | `subtotal × shipping%` |
| 4 | 📝 **مصاريف أخرى** | مبلغ ثابت | مباشر |

### 🔁 ملحقات متعددة لفاتورة واحدة
- 📦 شحنة واحدة قد تأتي على دفعات (partial shipments)
- 📄 كل دفعة → ملحق منفصل
- 📈 التكلفة تُجمَّع كلها على نفس فاتورة الشراء

### 💎 رسملة التكاليف في قيمة المخزون
```
التكلفة الحقيقية للوحدة = (قيمة الشراء + جمارك + ضريبة + شحن) ÷ عدد الوحدات
```
- ✅ تُحتسب عند `POST` للملحق
- ✅ تُضاف لـ `product.cost` كمتوسط مرجّح
- ✅ تنعكس فورًا على هامش الربح في POS والتقارير

> 📝 **ملاحظات للمحاضر:** هذه الميزة نادرة في أنظمة POS الكويتية — أغلبها يأخذ تكلفة الفاتورة فقط ويتجاهل الجمارك. النتيجة: التاجر يظن ربحه 30% وهو في الحقيقة 15%. رسملة التكاليف تحوّل المخزون من «قيمة شراء» إلى «قيمة حقيقية»، وهذا يصحّح كل قرارات التسعير والمشتريات.

| **الشارة في POS** | 🏷️ تركيبة |

### 🏭 الإنتاج الدفعي (Batch Production)
- 🎯 إنتاج كمية مباشرة (مثلاً: 50 زجاجة دفعة واحدة)

> 📝 **ملاحظات للمحاضر:** هذه الميزة تُحوّل محلات العطور من «بائع جاهز» إلى «مصنع صغير». الربح الثابت يعني أن التاجر يقرر ربحه بدل أن يحسبه — وهذا مناسب لطبيعة العطور (الربح ليس نسبيًّا للتكلفة بل للسوق). البادئة `9` تساعد الكاشير على تمييز التركيبة بصريًّا عند البيع.

# 🎁 الباقات (Bundles) — عروض موسمية ذكية

<hr/>

<p align="center">
  <em>بِع أكثر من منتج بسعر مخفّض — مع حساب التكلفة والربح تلقائيًّا.</em>
</p>

### 📦 ما هي الباقة؟

| الميزة | الوصف |
|---|---|
| **التعريف** | تجميع عدة منتجات في وحدة بيع واحدة بسعر مخفّض |
| **السعر** | يُحدّد يدويًّا (أقل من مجموع الفردي) |
| **التكلفة** | تُحسب تلقائيًّا (مجموع تكاليف المكوّنات) |
| **الربح** | تُحسب تلقائيًّا = السعر − التكلفة |
| **الموسمية** | تاريخ بداية + نهاية (تُلغى تلقائيًّا بعد الانتهاء) |
| **الحالة** | `ACTIVE` / `EXPIRED` / `DRAFT` |
| **البيع** | تظهر في POS كصنف واحد قابل للنقر |

### 🎯 مثال عملي: باقة «إهداء العيد»

| # | 🧴 المكوّن | 📊 الكمية | 💵 التكلفة | 💰 السعر الفردي |
|---|---|---|---|---|
| 1 | عطر «مسك خاص» 50ml | 1 | 8.000 KWD | 18.000 KWD |
| 2 | بخور عود فاخر 50g | 1 | 5.000 KWD | 12.000 KWD |
| 3 | مبخرة نحاسية صغيرة | 1 | 3.000 KWD | 7.000 KWD |
| **─** | **المجموع الفردي** | — | **16.000 KWD** | **37.000 KWD** |
| 🎁 | **سعر الباقة** | — | **16.000 KWD** | **29.900 KWD** |
| 📈 | **الربح** | — | — | **13.900 KWD** (86.8%) |

### 🗓️ الموسمية الذكية
- 📅 تاريخ البداية: 1 رمضان → تُفعَّل تلقائيًّا في POS
- 📅 تاريخ النهاية: 15 شوال → تُخفى تلقائيًّا
- 🎯 لا حاجة لتذكير الموظفين — النظام يديرها

> 📝 **ملاحظات للمحاضر:** الباقات ترفع متوسط قيمة الفاتورة (Average Ticket Size) بنسبة 20–40%. المثال الموضّح يبيّن كيف يدفع العميل 29.900 بدل 37 — فيشتري «صفقة» — بينما التاجر يحقق ربحًا ممتازًا (86.8%). الموسمية تمنع نسيان إلغاء الباقة بعد العيد، فلا تستمر في خصم غير مخطّط.

# 🌙 الورديات (Shifts)

<hr/>

<p align="center">

### 🔁 دورة الوردية

| المرحلة | الإجراء | المسؤول |
|---|---|---|

| **4. الفروقات** | حساب الفرق بين المتوقع والفعلي | تلقائي |
| **5. الاعتماد** | مراجعة المدير + قيد محاسبي | المدير |

### 💵 المبالغ المتوقعة مقابل الفعلية

| 💼 طريقة الدفع | 📊 المتوقع (حسب الفواتير) | 💰 الفعلي (عدّ المشرف) | ⚖️ الفرق |
|---|---|---|---|
| 💵 نقدي | 1,250.000 | 1,248.500 | −1.500 |
| 💳 K-Net | 3,400.000 | 3,400.000 | 0.000 |
| 💎 Visa | 890.000 | 890.000 | 0.000 |
| 🏆 **المجموع** | **5,540.000** | **5,538.500** | **−1.500** |

### 🙈 الإغلاق الأعمى (Blind Close)
- ✅ الكاشير لا يرى المتوقع عند الإغلاق
- ✅ يدخل الفعلي فقط — يكشف الفروقات بصدق
- ✅ يمنع «تسوية» العدّ مع المتوقع

### 📋 سياسات الفروقات

| مقدار الفرق | الإجراء |
|---|---|
| 0 — 1 KWD | ✅ مقبول (تقريب) |
| 1 — 10 KWD | ⚠️ يُسجَّل + سبب مطلوب |
| 10 — 50 KWD | 🔍 مراجعة المدير الإلزامية |
| > 50 KWD | 🚨 تحقيق + تقرير للمالك |

### 🧮 القيد المحاسبي عند الإغلاق
```
د: النقدية (الصندوق)     5,538.500
   ج: صندوق الورديات              5,538.500
(إثبات إغلاق وردية #1234)
```

> 📝 **ملاحظات للمحاضر:** الورديات هي العمود الفقلي لمكافحة سرقة الكاشير. الفرق البسيط يوميًّا (−1.5 KWD) قد يبدو تافهًا، لكنه 547 KWD سنويًّا — يكفي لتمويل حملة تسويق. الإغلاق الأعمى يكشف من يسرق بلا مواربة. لو تكرر فرق كبير مع كاشير معيّن → تنبيه تلقائي للمالك.

# 🔐 التدقيق والأمان

<hr/>

<p align="center">
  <em>كل إجراء حسّاس يُسجَّل · كل محاولة احتيال تُكشف · بياناتك محميّة.</em>
</p>

### 📜 سجل التدقيق (Audit Log)

| المبدأ | التطبيق |
|---|---|
| **Fail-closed** | لو فشل التدقيق → يُمنع الإجراء (لا يُسمح بالتنفيذ) |
| **Immutable** | السجلات لا تُحذف ولا تُعدّل — فقط تُقرأ |
| **24+ نوع إجراء** | كل إجراء حسّاس له رمز فريد |
| **بصمة المستخدم** | user · role · IP · timestamp · before/after |

### 🎬 أنواع الإجراءات المُسجَّلة (24+)

| 🏷️ الفئة | 📋 الإجراءات |
|---|---|
| 🛒 **المبيعات** | `SALE_CREATED` · `SALE_CANCELLED` · `REFUND_ISSUED` · `VOID_ITEM` |
| 📦 **المخزون** | `STOCK_ADJUSTMENT` · `TRANSFER_CREATED` · `STOCKTAKE_APPROVED` |
| 💰 **المالية** | `PAYMENT_PROCESSED` · `JOURNAL_ENTRY_POSTED` · `EXPENSE_RECORDED` |
| 🏭 **المشتريات** | `PURCHASE_INVOICE_POSTED` · `SUPPLIER_PAYMENT_ISSUED` |
| 🏷️ **التسعير** | `PRICE_CHANGED` · `BUNDLE_CREATED` · `COMPOSITION_PRODUCED` |
| 🌙 **الورديات** | `SHIFT_OPENED` · `SHIFT_CLOSED` · `DRAWER_OPENED` |
| 👥 **المستخدمون** | `USER_CREATED` · `ROLE_CHANGED` · `PASSWORD_CHANGED` |
| 🔐 **الدخول** | `LOGIN_SUCCESS` · `LOGIN_FAILED` · `LOGOUT` |

### 🚨 تحليل معدل الإلغاء (Void-Rate Analysis)

| 📊 المؤشر | 🚨 الحد | 🎯 الإجراء |
|---|---|---|
| void-rate لكل كاشير | > 5% | ⚠️ تنبيه للمدير |
| void-rate لكل صنف | > 8% | 🔍 مراجعة التسعير |
| مرتجعات كاشير معيّن | > المتوسط +2σ | 🚨 تحقيق |

### 🔒 طبقات الأمان

| الطبقة | التقنية | الحماية |
|---|---|---|
| **المصادقة** | NextAuth.js + JWT (7 أيام) | جلسات آمنة · انتهاء صلاحية |
| **كلمات المرور** | bcrypt (cost factor 10) | مقاومة rainbow table |
| **قفل الصفوف** | Prisma `SELECT FOR UPDATE` | حماية تزامن المخزون |
| **CORS** | مُقيّد على دومين الإنتاج فقط | حماية استدعاءات API |
| **Rate Limiting** | 100 طلب/دقيقة لكل IP | حماية هجمات brute-force |
| **HTTPS** | إلزامي على Vercel | تشفير النقل |

> 📝 **ملاحظات للمحاضر:** مبدأ «Fail-closed» جوهري: لو عجز النظام عن تسجيل التدقيق، يُوقف الإجراء بدل التنفيذ بلا سجل. هذا عكس الأنظمة الضعيفة التي تُسجّل لاحقًا أو تتغافل. تحليل معدل الإلغاء يكشف الكاشير الذي يلغي فواتير لإخفاء سرقة — أداة قوية لمكافحة الاحتيال الداخلي.
